# Determ CLI Reference

All `determ` subcommands. Run `determ --help` for the canonical built-in help.

**See also:** [`QUICKSTART.md`](QUICKSTART.md) for hands-on recipes, [`PROTOCOL.md`](PROTOCOL.md) for wire formats, [`WHITEPAPER-v1.x.md`](WHITEPAPER-v1.x.md) for the standalone technical paper.

## Node lifecycle

| Command | Purpose |
|---|---|
| `determ init [--data-dir D] [--profile P]` | Generate config + Ed25519 keypair in a fresh data dir |
| `determ start --config FILE` | Run the node daemon (foreground) |

Profiles: `cluster` (LAN), `web` (default), `regional`, `global`, `tactical` (mobile-unit swarm; sub-30 ms blocks), plus `*_test` variants (`single_test`, `cluster_test`, `web_test`, `regional_test`, `global_test`, `tactical_test`) with sub-30 ms timers for CI execution mirroring each prod sibling's (chain_role, sharding_mode, M, K). Differ in round timer durations (`tx_commit_ms`, `block_sig_ms`, `abort_claim_ms`) + committee size (M, K) + chain_role + sharding_mode. See `include/determ/chain/params.hpp` for the full profile-constant set.

## Inspection (block-explorer)

All inspection commands hit the running node's RPC. Default RPC port is in the node's config.

| Command | Returns |
|---|---|
| `determ status [--field <name>] [--rpc-port N]` | Chain head, head_hash, role, shard_id, epoch_index, peer_count, mempool, `committee_region` (this node's R1 region tag — empty = global pool), `pending_param_changes` count (A5 Phase 2 governance staging — total entries across all heights, queryable in detail via `determ pending-params`), mode counters + `protections` block (current state of every operator-tunable security/log flag — `rpc_localhost_only`, `rpc_hmac_auth`, `rpc_rate_limit`, `gossip_rate_limit`, `log_quiet`, `bft_enabled`, `sharding_mode`). Monitoring systems can alert on flag drift in production. **`--field <name>` flag** prints just the value of one top-level field (one-shot extraction for shell scripts; e.g., `determ status --field height` prints just the integer, no JSON envelope). Absent/null fields print empty line; non-scalar values (like `protections`) print as compact JSON dumps. |
| `determ summary [--json] [--rpc-port P]` | Operator dashboard. Gathers the most-frequently-checked status fields into a single compact view: height + head_hash, peer + committee + mempool sizes, this node's `committee_region`, sync state, A1 invariant status (computed from `chain_summary` A1 counters). Designed for the SSH-into-server quick-check workflow — one terminal-page summarizes "is the node healthy + caught up + connected?". Uses `status` RPC for most fields + `chain_summary` (last_n=1 for minimal payload) for A1 counter aggregation + `committee` for current K-of-K committee size. **No new server-side RPC method.** Default human output: 8 right-aligned lines (height, head_hash truncated to 24 chars, genesis truncated, peers, mempool, committee + region, sync_state, A1 invariant); `--json` emits structured envelope `{height, head_hash, genesis, peers, mempool, committee, committee_region, sync_state, a1_invariant_ok, live_total_supply, expected_total}`. **Exit code 2 on A1 invariant violation** (in addition to printing VIOLATED); exit 0 otherwise; exit 1 on RPC error. |
| `determ peers [--json] [--count] [--rpc-port N]` | Connected peer addresses. Default: one host:port per line. `--json` emits the raw RPC array (`["host:port", ...]`). **`--count` flag** prints a bare integer (count of connected peers; 0 if the RPC returns a non-array — defends monitoring dashboards against malformed responses). `--count` and `--json` compose: `--count --json` emits `{"count": N}` single-line for machine consumers. Useful one-shot for monitoring scripts (e.g., `if [ $(determ peers --count) -lt 3 ]; then alert; fi`). |
| `determ show-block <i> [--rpc-port N]` | Full block JSON at index `i` |
| `determ chain-summary [--last N] [--json] [--rpc-port N]` | Compact view of last `N` blocks (default 10). `--json` emits the raw RPC response (`{blocks: [...], height, total_supply, genesis_total, accumulated_subsidy, accumulated_inbound, accumulated_slashed, accumulated_outbound}` — the A1 unitary-supply counters). |
| `determ validators [--filter-region R] [--json] [--count] [--rpc-port N]` | Registered validator pool (domain, ed_pub, stake, active_from, region). Default: human-readable table including the region column (empty region renders as `(global)`). `--filter-region R` post-filters to validators whose region equals R (case-sensitive; useful in regional deployments to inspect the locally-eligible pool per rev.9 R1 region-pinned committee selection). `--json` emits the (filtered) RPC array verbatim — feeds directly into `verify-block-sigs --committee` for light-client K-of-K verification. **`--count` flag** prints a bare integer (count of validators after any `--filter-region` filter; 0 if RPC returns a non-array, defending dashboards against malformed responses). Composes with `--filter-region` (count just the regional subset — e.g., `determ validators --count --filter-region us-east` for "how many validators are eligible for the us-east committee?") and with `--json` (emits `{"count": N}` single-line for machine consumers). |
| `determ stakes [--top N] [--json] [--rpc-port P]` | List validators sorted by stake **descending**. Complements `validators` (which sorts alphabetically by domain). Operator use cases: (a) "Who are the top stakers?" — slashing impact analysis, committee-influence assessment, governance signal; (b) "Why does region X have low committee participation?" — sorted-by-stake view immediately shows whether the region's validators are minimally staked or fully staked. `--top N` truncates to top N entries; `--json` emits the (filtered + sorted) entries as a JSON array including an explicit 1-indexed `rank` field per entry. Tie-breaks by domain ascending. Zero-stake entries kept (may be pre-genesis registrants in DOMAIN_INCLUSION mode). Pulls from `validators` RPC + sorts client-side — no new server method. |
| `determ committee [--json] [--count] [--rpc-port N]` | Current epoch's K-of-K committee (deterministic from chain state). `--json` flag same as for `validators` — emits the raw RPC array shape `verify-block-sigs --committee` consumes. **`--count` flag** prints a bare integer (count of committee members; 0 if the RPC returns a non-array). Useful in regional deployments to monitor temporary under-quorum — a region may have `committee_size < K` until enough creators come back online. Composes with `--json` to emit `{"count": N}` for machine consumption. |
| `determ pending-params [--at-height N] [--json] [--rpc-port P]` | A5 Phase 2 governance visibility: list of staged PARAM_CHANGE entries (effective_height, name, value_hex, value_bytes) that have been validated and accepted but not yet activated. Sorted ascending by `effective_height`; within a bucket, in insertion order. Empty for `governance_mode=0` deployments. `--at-height N` post-filters to entries with `effective_height <= N` — useful for operators verifying that a planned upgrade window covers the expected param changes. `status` RPC also exposes the total count as `pending_param_changes` for monitoring drift. Default: human-readable table; `--json` pass-throughs the (filtered) RPC array. |
| `determ abort-records [--top N] [--json] [--rpc-port P]` | S-032 abort_records cache visibility: domains that have aborted in Phase-1, sorted by `count` descending (most-aborted first). Each entry is `{domain, count, last_block}`. Useful for diagnosing committee instability, BFT-escalation patterns, and which validators may need remediation. `--top N` truncates to the first N entries (client-side, since the RPC always returns the full sorted list). Empty for chains with no Phase-1 aborts. Phase-2 (timing-skew) aborts are NOT tracked here — they don't slash or accrue records. |
| `determ supply [--json] [--field <name>] [--rpc-port P]` | Compact A1 unitary-balance summary: `genesis_total` + 4 running counters (subsidy minted / inbound credited / slashed / outbound sent) + computed `expected_total` vs `live_total_supply` + invariant status (OK or VIOLATED). Useful for monitoring scripts that just want supply numbers without the per-block `chain-summary` detail. Pulls from the `chain_summary` RPC (last_n=1 for minimum payload) — no new server-side RPC method. **Exit code 2 on A1 invariant violation** (in addition to printing VIOLATED); exit 0 otherwise. **`--field <name>` flag** prints just the bare value of one field (one-shot scalar extraction for shell scripts; e.g., `determ supply --field live_total_supply` prints just the integer). Allowed: `live_total_supply`, `expected_total`, `genesis_total`, `accumulated_subsidy`, `accumulated_inbound`, `accumulated_slashed`, `accumulated_outbound`, `a1_invariant_ok` (`true`/`false`). Unknown field name → exit 1 with allowed-list diagnostic. In `--field` mode, exit 0 even on A1 violation (the `a1_invariant_ok` field already exposes that bit). |
| `determ tx-summary [--last N] [--by-type] [--json] [--rpc-port P]` | Per-type transaction count over the last `N` blocks (default 10). Operator capacity-monitoring tool: "how many TRANSFERs vs STAKEs vs DAPP_CALLs did we process in the last N blocks?". Useful for dashboards tracking workload mix. Pulls from `chain_summary` RPC (which already returns full block JSON including `transactions[].type`) — no new server-side method. Default human-readable output prints window (block range `[I..J]`), total tx count, and per-type breakdown (TRANSFER / STAKE / UNSTAKE / REGISTER / DEREGISTER / DAPP_REGISTER / DAPP_CALL / PARAM_CHANGE / COMPOSABLE_BATCH / MERGE_EVENT). `--by-type` is the default mode (kept as an explicit flag for forward-compat). `--json` emits a structured envelope `{total: N, by_type: {TRANSFER: N1, STAKE: N2, ...}, block_count: N, first_index: I, last_index: J}` (includes zero-count types omitted from human output). Empty chain prints "no blocks in window". Exit 1 only on RPC error or malformed response. |
| `determ chain-id [--rpc-port P]` | Print just the chain's genesis-block hash (canonical chain identity, 64-char hex). Useful for monitoring scripts cross-checking chain identity across nodes — `if [ "$(determ chain-id --rpc-port 7778)" != "$EXPECTED" ]; then alert; fi`. Pulls from `status` RPC. Exit 1 if no genesis loaded. |
| `determ block-hash <index> [--rpc-port P]` | Print just `compute_hash` of the block at `<index>` (64-char hex, no JSON envelope). Useful for cross-node fork-detection scripts — `[ "$(determ block-hash 100 --rpc-port 7778)" == "$(determ block-hash 100 --rpc-port 7779)" ]` → agree or FORKED. Pulls full block JSON from `block` RPC and extracts the `hash` field. Exit 1 if index out of range or hash field missing. |
| `determ block-info <index> [--field NAME] [--json] [--rpc-port P]` | Print one or more fields of a single block. Extension of `block-hash` (which only emits `compute_hash`) to any block field: `index`, `prev_hash`, `timestamp`, `tx_root`, `state_root`, `block_hash`, `consensus_mode`. Without `--field`, defaults to printing the full block JSON. With `--field NAME`, prints just the field value as bare scalar (no JSON envelope) for shell-script consumption. Operator examples: `determ block-info 100 --field state_root` (audit trail — "what was state_root at block 100?"); `determ block-info 100 --field timestamp` (chain progress velocity — "what was timestamp delta between blocks N and M?"). `block_hash` alias: maps to the server-side `hash` field. Unknown `--field` name → exit 1 with allowed-list diagnostic. Out-of-range index → exit 1. |
| `determ block-range <from> <to> [--field NAME] [--json] [--rpc-port P]` | Bulk-fetch a range of block fields via the `headers` RPC (inclusive on both ends). Builds on `block-hash` (single-height); operators doing fork detection across many heights pay one RPC + pagination round-trip per up-to-256 headers instead of one per height. Default output: one line per height `<index> <block_hash>`. `--field NAME` selects which field to emit and prints bare values (one per line) ready for `diff(1)` against another node's range: e.g. `diff <(determ block-range 100 200 --field state_root --rpc-port 7778) <(determ block-range 100 200 --field state_root --rpc-port 7779)`. Allowed `--field` values: `index`, `prev_hash`, `state_root`, `block_hash`, `timestamp`. `--json` emits a structured envelope `{headers: [...], from, to, received}`. Heights past the server's current height are silently truncated (honest reflection of chain state). Exit 1 on RPC error or invalid `--field` name. |
| `determ check-fork --node-a host:port --node-b host:port --from N --to M [--field NAME] [--json]` | Automated cross-node divergence detection. Fetches the requested range of block fields from BOTH nodes via headers RPC (paged 256/page; same path `block-range` uses internally) and reports the FIRST height where they disagree on `--field`. Real operator use: when two nodes report mismatched `state_root` at the same `height`, this command localizes the divergence to a specific block — operators can then drill into that block's `transactions` / `inbound_receipts` / `equivocation_events` to find which event caused the fork. Default field `state_root` (canonical chain-state commitment); `block_hash` also valuable for "did headers diverge?" checks. Allowed `--field` values match `block-range`. **Exit codes**: 0 ranges identical, 2 DIVERGENCE detected (operator alert gate — distinguishes from "no errors but found a problem"), 1 error (bad args, RPC error, unknown `--field`). `--json` emits `{identical: bool, first_divergence: N\|null, node_a_received: N, node_b_received: N, field, from, to}`. Example: `determ check-fork --node-a 127.0.0.1:7778 --node-b 127.0.0.1:7779 --from 100 --to 200`. |
| `determ head [--field height\|hash] [--json] [--rpc-port P]` | Print just the chain's current head — `<height> <head_hash>` space-separated by default. `--field height` prints just the height (no hash); `--field hash` prints just the hash (no height) — for `expr`-style scripts that need a single value. `--json` emits `{height, head_hash}` for structured scripts. For monitoring scripts tracking chain advancement (`while true; do read h hash <<< "$(determ head)"; ... done`). Pulls from `status` RPC. Exit 1 if chain not loaded or unknown `--field` value. |
| `determ show-account <addr> [--rpc-port N] [--json]` | Balance + nonce + registry record + stake for any address. Default human-readable; `--json` pass-throughs the raw RPC envelope for script consumption. Empty result → `{}` (JSON) or `(no on-chain state for X)` (human). |
| `determ show-tx <hash> [--rpc-port N] [--json]` | Locate a tx in a finalized block (block_index + payload). Default human-readable; `--json` pass-throughs the raw RPC envelope. Empty result → `{}` (JSON) or `(tx ... not found ...)` (human). |
| `determ where-is <addr\|domain> --shard-count N [--salt-hex HEX] [--json]` | Local-only shard-routing diagnostic. Given an address (anon-form `0x...64hex`) or a registered domain name, compute which shard it routes to via `crypto::shard_id_for_address(addr, shard_count, salt)`. Operator use cases: "Which shard does this anon address live on?" (incident analysis — "why is this balance not showing up on shard 2?"); "What shards do my faucet addresses route to?" (capacity planning — "are pre-mine recipients balanced across shards?"). `--shard-count N` required (from genesis `initial_shard_count`); `--salt-hex HEX` optional 64-hex `shard_address_salt` from genesis (defaults to zero salt for testnets / dev). Default output: `<address> -> shard <N> (of <shard-count>)`. `--json` emits `{"address": "<>", "shard": N, "shard_count": N}`. **Pure local computation — no RPC required.** |
| `determ balance <domain> [--rpc-port N]` | Balance only |
| `determ nonce <domain> [--rpc-port N]` | Next expected nonce |
| `determ stake_info <domain> [--rpc-port N]` | Locked stake + unlock_height |

## Snapshots (B6.basic — fast-bootstrap)

| Command | Purpose |
|---|---|
| `determ snapshot create [--out FILE] [--headers N] [--rpc-port N]` | Dump current chain state (accounts/stakes/registrants/dedup + tail headers) |
| `determ snapshot inspect --in FILE [--state-root <hex64>] [--json]` | Validate + summarize a snapshot file (round-trip via `restore_from_snapshot` — S-033 + S-038 gates fire on tampered state). Prints `block_index`, `head_hash`, `state_root`, account/stake/registrant counts, and chain parameters. Optional `--state-root` pins an externally-trusted root for trustless-fast-sync verification: prints `✓ matches --state-root` on agreement; exits non-zero with `FAIL` diagnostic if the snapshot's state_root doesn't match the supplied root (defeats a tampered snapshot pointed at a chain the operator doesn't trust). Optional `--json` emits a single-line JSON object with the same data for script consumption (`{"status":"ok","block_index":N,"head_hash":"...","state_root":"...","accounts":N,...}`); error cases also emit JSON (`{"error":"cannot_open"|"invalid_state_root_length"|"state_root_mismatch"|"exception","..."}`). Useful for bootstrap-orchestrator pipelines that consume snapshot verification results. |
| `determ snapshot fetch --peer host:port --out FILE [--headers N]` | Pull a snapshot from a running node over the gossip wire |
| `determ snapshot diff <file-a> <file-b> [--json]` | Compare two snapshot files and print divergence lines. Operator incident-analysis tool: if two nodes report mismatched state_root at the same height, snapshot each and run this command to localize the divergence to one of `block_index` / `head_hash` / `state_root`, account + stake + registry aggregate counts, balance/stake sums, or any of the 5 A1 unitary-supply counters. Both snapshots validated through `restore_from_snapshot` (malformed input fails fast); pure read-only comparison after. Default output: human-readable `FIELD: A != B` lines, or `(identical)` when equal. `--json` emits a structured envelope `{"a":"<path>","b":"<path>","identical":<bool>,"differences":[{"field":"<name>","a":<val>,"b":<val>},...]}` for script consumption. Exit 0 on success (differences are informational, not a test gate); exit 1 only on load failure. |
| `determ snapshot stats <file> [--json]` | Lightweight metadata-only snapshot inspection. Reads top-level JSON fields without running the full `Chain::restore_from_snapshot` pipeline. Useful for fast triage on large snapshot files before committing to `snapshot inspect`'s heavier restore + state_root verify path. Reports: file size on disk, version, block_index + head_hash, account/stake/registrant counts, and the 5 A1 unitary-supply counters. **Differences from `snapshot inspect`**: stats is O(parse + top-level field read) — fast even on 100 MB+ snapshots; inspect is O(parse + full restore + state_root compute) — slower but verifies S-033/S-038 internal consistency. Default human output is human-readable column; `--json` emits structured envelope `{file, size_bytes, version, block_index, head_hash, accounts, stakes, registrants, genesis_total, accumulated_*}`. Exit 0 on success; exit 1 on file-open / JSON-parse / not-object errors (JSON error responses also emitted to stdout when `--json` is set). |

To bootstrap from a snapshot, set `snapshot_path` in the node's `config.json` and `determ start`. The node will skip per-block replay and install state directly.

## State commitment + light-client RPC (v2.1 + v2.2)

| Command | Purpose |
|---|---|
| `determ state-root [--rpc-port N]` | Print the chain's Merkle state root + height + head_hash. Returns the live `compute_state_root()` value, which post-S-038 also matches the value stored in the head block's `state_root` field (the producer wires this on every finalized block). Operators can call this against multiple nodes to detect silent state divergence — pre-S-038 a real S-030 D1/D2 attack would manifest at the apply layer only; post-S-038, divergent nodes loud-fail at apply time with a `state_root mismatch` diagnostic before ever producing a divergent block. |
| `determ headers [--from N] [--count M] [--rpc-port P]` | Fetch a slice of block headers (Block JSON minus transactions / cross_shard_receipts / inbound_receipts / initial_state). Light-client header-sync primitive: returned headers carry committee + signatures + tx_root + delay_seed / delay_output + cumulative_rand + state_root for verify-state-proof anchoring, plus an explicit `block_hash` per header so the prev_hash chain is verifiable without re-deriving compute_hash (which needs the stripped heavy fields). Defaults: `--from 0`, `--count 16`. Server caps count at 256. Out-of-range `--from` returns an empty array (not an error). Returns `{headers, from, count, height}`. |
| `determ verify-headers [--in <file>] [--genesis-hash <hex64>] [--prev-hash <hex64>]` | Verify the prev_hash chain in a `determ headers` response. Reads JSON from `--in` or stdin. Walks consecutive header pairs and asserts `header[i].prev_hash == header[i-1].block_hash`. Optional anchors: `--genesis-hash` checks against the genesis block_hash when the slice starts at index 0; `--prev-hash` anchors a non-genesis slice (= block_hash of the block immediately before the slice). Without an anchor, the first header's prev_hash is unanchored — internal chain links still verified. Pure chain-of-hashes integrity check (does NOT verify committee signatures; use `verify-block-sigs` for that). |
| `determ verify-block-sigs --header <file> --committee <file> [--bft]` | Verify K-of-K committee Ed25519 signatures on a single block header. Reads the header from `--header` (accepts either a single-block JSON or a `determ headers` envelope — extracts the first header from envelope shape) and a committee pubkey map from `--committee` (a JSON array of `{domain, ed_pub}` entries, or `{members: [...]}` envelope shape — `determ validators --json` and `determ committee --json` both emit this shape directly). Computes `compute_block_digest(b)` over the header fields and verifies each `creators[i]`'s `creator_block_sigs[i]` against the looked-up pubkey. MD mode requires every signature; `--bft` allows up to `K - ceil(2K/3)` sentinel-zero slots. Together with `verify-headers` (chain integrity) + `verify-state-proof` (per-field state), this completes the v2.2 trustless light-client verification chain: fetch headers from any source, verify chain links + committee signatures locally, anchor state_root from a verified header, verify state-proofs against that root. |
| `determ state-proof --ns {a\|s\|r\|d\|b\|k\|c} --key <name> [--rpc-port N]` | Fetch a Merkle inclusion proof for any state entry. RPC-exposed namespaces: `a` = accounts, `s` = stakes, `r` = registrants, `d` = DApp registry (v2.18, key = DApp's owning domain), `b` = abort_records (S-032), `k` = genesis-pinned constants, `c` = A1 counters (via the composite `k:c:<name>` lookup). The full ten-namespace state tree (PROTOCOL.md §4.1.1) also has `i/m/p` but those use composite keys and aren't surfaced by this RPC in v2.2. Light-client primitive: the proof verifies against the current `state_root`, which is committed into the head block's `block_hash` (compute_hash via signing_bytes; the chain's `prev_hash` chain transitively authenticates it forward). |
| `determ verify-state-proof [--in <file>] [--state-root <hex64>]` | Local light-client demonstrator: verifies a state-proof response via `crypto::merkle_verify` without trusting the responding node. Reads JSON from `--in` or stdin. Optional `--state-root <hex64>` pins an externally-trusted root (real light-client mode); without it, the proof's self-claimed root is used (self-consistency check). Prints `OK` + structured summary on success; `FAIL` + reason on tampering / mismatch. Pair with `state-proof`: `determ state-proof --ns a --key alice \| determ verify-state-proof --state-root <trusted-hex>`. |
| `determ verify-genesis --in <file> [--expected-hash <hex64>] [--json]` | Standalone genesis.json validator. Loads, applies the same parsing + sane-bounds checks as `determ start` (LOTTERY-multiplier ≥ 2 if subsidy_mode == LOTTERY, block_subsidy / subsidy_pool_initial / zeroth_pool_initial ≤ 1e18, lottery_multiplier × block_subsidy ≤ 1e18, genesis_message ≤ 256 bytes), computes the chain-identity hash via `compute_genesis_hash`, prints a structured summary. Default output is human-readable; `--json` emits a single-line JSON object for script consumption (`{"status":"ok","genesis_hash":"...","chain_id":"...","m_creators":N,...}`). Optional `--expected-hash <hex64>` pins against an externally-trusted value — exits non-zero with a clear diagnostic when computed hash doesn't match (defeats config-rewrite attacks where a deployment template smuggles in a different chain identity). The output explicitly distinguishes operational params NOT bound to identity hash (m_creators / k_block_sigs / block_subsidy / min_stake / initial_shard_count / bft_enabled — see S-039 in `docs/SECURITY.md`) from identity-bound fields (genesis_message / committee_region), so operators can spot mismatches in unbinded fields by direct inspection. Workflows: pre-deployment validation, cross-team coordination on chain identity, cluster onboarding. |

### Complete v2.2 trustless light-client verification flow

The seven CLIs above (`state-root`, `headers`, `verify-headers`, `verify-block-sigs`, `state-proof`, `verify-state-proof`, plus `validators --json` / `committee --json` for committee pubkey export) compose into an end-to-end trustless verification pipeline. Each step verifies a different property locally without trusting the responding node, gated only by an externally-pinned `--state-root` / committee pubkey set:

```bash
# 1. Fetch recent headers (untrusted full node OK; we'll verify them).
$DETERM headers --rpc-port 8771 --from 0 --count 100 > headers.json

# 2. Verify the prev_hash chain (catches re-ordered / spliced headers).
$DETERM verify-headers --in headers.json   # OK on valid chain

# 3. Get the committee pubkeys for K-of-K signature verification.
#    For genesis-pinned light clients, the operator hard-codes this set;
#    for chains using key rotation, derive from a previously-verified
#    state-proof of the registrants_ map (namespace `r`).
$DETERM validators --rpc-port 8771 --json > committee.json

# 4. Verify K-of-K committee signatures on each header.
$DETERM verify-block-sigs --header headers.json --committee committee.json

# 5. Extract a verified state_root from one of the headers (any will do
#    since we've verified the chain links + signatures).
STATE_ROOT=$(jq -r '.headers[-1].state_root' headers.json)

# 6. Verify any state field against the pinned state_root.
$DETERM state-proof --rpc-port 8771 --ns a --key alice > proof.json
$DETERM verify-state-proof --in proof.json --state-root "$STATE_ROOT"

# 7. (Alternative for whole-state) Verify a downloaded snapshot.
$DETERM snapshot inspect --in snapshot.json --state-root "$STATE_ROOT"
```

What this proves: a malicious full node serving headers / state-proofs / snapshots is caught at one of these checks. The trust chain bottoms out at the externally-pinned committee + the chain-of-hashes link from the verified header forward; the state_root anchor defeats fabricated-root attacks; the committee-sig check defeats forged-header attacks; the prev_hash chain check defeats spliced-header attacks.

## DApp substrate RPC (v2.18 + v2.19)

The DApp's identity is its owning domain (`tx.from` at registration). There is no separate "dapp_id" — the registered Determ domain IS the dapp identifier. CLI verbs that name a DApp use `--from <domain>` for tx-authoring and `--domain <D>` for queries.

| Command | Purpose |
|---|---|
| `determ submit-dapp-register --priv <hex> --from <domain> --service-pubkey <64hex> --endpoint-url <url> [--topics t1,t2,t3] [--retention 0\|1] [--metadata-hex <hex>] [--fee <N>] [--rpc-port N]` | Register / update a DApp on the chain. Idempotent — re-registering with the same `--from` updates the entry. `--service-pubkey` is the libsodium box pubkey used for end-to-end DAPP_CALL encryption. `--retention 0` = full retention, `1` = pruneable-after-K. |
| `determ submit-dapp-register --priv <hex> --from <domain> --deactivate [--fee <N>] [--rpc-port N]` | Deactivate the DApp owned by `--from`. Sets `inactive_from = current_height + DAPP_GRACE_BLOCKS`; in-flight calls finish within the grace window. |
| `determ submit-dapp-call --priv <hex> --from <sender> --to <dapp-domain> [--topic <T>] [--payload-hex <hex>] [--amount <N>] [--fee <N>] [--rpc-port N]` | Submit a DAPP_CALL routed to `--to`. `--topic` must match one of the DApp's registered topics (or `""`). `--payload-hex` is the application's opaque ciphertext (typically AEAD(`service_pubkey`, plaintext, nonce); ≤ 16 KB). `--amount` is an optional payment credited to the DApp's account. |
| `determ dapp-list [--prefix P] [--topic T] [--rpc-port N]` | List registered DApps. Optional `--prefix` filters by domain prefix; `--topic` keeps only DApps whose registered topic list contains a match. |
| `determ dapp-info --domain <D> [--rpc-port N]` | Per-DApp record: `domain`, `service_pubkey`, `endpoint_url`, `topics`, `retention`, `metadata`, `registered_at`, `active_from`, `inactive_from`. |
| `determ dapp-messages --domain <D> [--from <H>] [--to <H>] [--topic <T>] [--rpc-port N]` | Retrospective DAPP_CALL poll. Returns up to 256 events (`DAPP_MESSAGES_PAGE_LIMIT`) in `[from, to]` addressed to `--domain`, optionally topic-filtered. Use the response's `last_scanned + 1` as the next `--from` to paginate. |

## determ-wallet binary (A2)

Separate executable from the `determ` daemon. Secret material never enters the chain daemon's address space — by design. The wallet handles the user's Ed25519 seed and threshold recovery; the daemon never sees them.

### Primitive layers

| Command | Purpose |
|---|---|
| `determ-wallet shamir split <hex> -t T -n N` | Split a secret into N Shamir shares with threshold T. Shares print one per line as `<x_hex>:<y_hex>`. |
| `determ-wallet shamir combine <share> [<share> ...]` | Reconstruct from ≥ T shares. Returns the original secret hex. |
| `determ-wallet envelope encrypt --plaintext <hex> --password <str> [--aad <hex>] [--iters N]` | AEAD-wrap arbitrary data (AES-256-GCM, PBKDF2-HMAC-SHA-256 keying, default 600k iters). |
| `determ-wallet envelope decrypt --envelope <blob> --password <str> [--aad <hex>]` | Unwrap. Returns plaintext hex; exit code 2 on AEAD tag failure. |

### Recovery flows

| Command | Purpose |
|---|---|
| `determ-wallet create-recovery --seed <hex> --password <str> -t T -n N --out <file> [--scheme {passphrase\|opaque}]` | Persist a T-of-N recovery setup as a single JSON document. Scheme `passphrase` (default) uses PBKDF2 directly off the password; scheme `opaque` routes through the OPAQUE adapter. |
| `determ-wallet recover --in <file> --password <str> [--guardians <i,j,k,...>]` | Reconstruct the seed from ≥ T envelopes. Scheme is auto-detected from the setup. Optional pubkey checksum gate prevents silent corruption. |

### OPAQUE primitives (diagnostic)

| Command | Purpose |
|---|---|
| `determ-wallet oprf-smoke` | Smoke-test libsodium primitives (ristretto255 scalar/point ops + Argon2id). Outputs sample scalars + a blinded point + a stretched key. |
| `determ-wallet opaque-handshake --mode {register\|authenticate} --password <str> --guardian-id <0..255> [--record <hex>]` | Exercise the OPAQUE adapter directly. Currently the **development stub** adapter; the real `libopaque` integration is tracked as **v2.14** (real OPAQUE wallet recovery) in `docs/V2-DESIGN.md`. See `wallet/PHASE6_PORTING_NOTES.md` for the MSVC porting status that gates the v2.14 ship. |
| `determ-wallet version` | Print version banner including current adapter suite name. |

### Adapter status check

The wallet's `is_stub()` flag exposes whether the linked OPAQUE adapter is the development stub or the real `libopaque` implementation (v2.14). The stub is offline-grindable against a compromised guardian — see `docs/proofs/WalletRecovery.md` (FA12) for the concrete-security bounds. The v2.14 ship replaces the stub with the real construction; once that lands, this `is_stub()` returns `false` for production builds. The wallet binary's `version` command surfaces the active suite tag.

## Exit codes

- `0` — success
- `1` — parsing error, missing argument, RPC failure, or assertion failure
- `2` — cryptographic failure (AEAD tag mismatch, OPAQUE authentication rejected, recovery reconstruction failed)

## Operator analytical scripts (tools/operator_*.sh)

## Wallet diagnostic CLIs

Operator-facing diagnostic commands shipped on the `determ-wallet` binary (built from the same CMake target; see `wallet/` source tree). All are pure local computation — no daemon RPC required.

| Command | Purpose |
|---|---|
| `determ-wallet inspect-envelope --in <file> [--json]` | Diagnostic CLI that reads an AEAD envelope file and dumps its metadata (format/version, PBKDF2 iteration count, salt + nonce hex, AAD presence + length, ciphertext + tag lengths) **without decrypting**. No password required. Use cases: operators verifying a file is a valid envelope before attempting a costly decrypt; forensic inspection of stale recovery artifacts; debugging which envelope corresponds to which share. Reads the canonical dot-separated hex blob produced by `envelope encrypt`. Calls `envelope::deserialize()` for header parsing only; never derives a key, never invokes AES-GCM. **Exit codes**: 0 valid envelope, 1 missing/unreadable file, 2 malformed envelope (truncated / wrong magic / parse failure). |
| `determ-wallet shamir-split --secret <hex> --threshold T --shares N [--json]` | Splits a hex-decoded secret into N Shamir shares with threshold T (any T-of-N reconstruct; T-1 reveals nothing — information-theoretic). Validates 1 ≤ T ≤ N ≤ 255, even-length hex, non-empty secret. Distinct from the `recovery` flow's high-level SSS+AEAD+OPAQUE composition — this is the raw primitive exposed for operator share-distribution workflows independent of the recovery envelope (printing physical shares, splitting a non-wallet secret). Default human output: `Share i: x=X y=YYYY…` per line; `--json` emits `{"shares":[{"x":<int>,"y_hex":"<hex>"},…]}`. **Exit codes**: 0 success, 1 validation failure. |
| `determ-wallet shamir-combine --shares <file> [--json]` | Reconstructs a secret from a JSON shares file (output of `shamir-split`) via Lagrange interpolation. Rejects duplicate x values, length-mismatched y values, missing fields, x outside [1, 255], empty array, missing file, odd-length y_hex. Threshold is implicit in the share count. Default output: hex secret on stdout; `--json` emits `{"secret_hex":"…"}`. **Exit codes**: 0 success, 1 validation failure or reconstruction error. **Note**: SSS is information-theoretic — providing T-1 shares may "succeed" but produce a value indistinguishable from random; callers must validate the reconstructed secret out-of-band. |
| `determ-wallet shamir-verify --shares <file> [--threshold T] [--json]` | Structural verification of a Shamir share-set **without reconstructing the secret**. Distinct from `shamir-combine` (which performs Lagrange interpolation): `shamir-verify` reports whether a share file is structurally well-formed under the SSS invariants. Checks: JSON shape (`{"shares":[{"x":<int>,"y_hex":"<hex>"},…]}`), every `x ∈ [1, 255]` (`x = 0` reveals the secret directly — must be rejected), all `x` values distinct (duplicate x's = singular Lagrange basis), all `y_hex` values share a common byte length (GF(2^8) interpolation operates byte-wise — mismatched lengths indicate a corrupt or merged-from-different-secrets share-set), even-length hex + valid hex character set, non-empty shares array. Optional `--threshold T` cross-check (errors if N < T — under-threshold share set cannot reconstruct). Default human output is a per-check pass/fail line plus a summary `shamir_valid=YES (N=… common_y_len=… distinct_x=…)` or `shamir_valid=NO (reason=…)`; `--json` emits `{valid, share_count, threshold_ok, common_y_byte_len, distinct_x_count, errors: [...]}`. Use cases: operator pre-flight verification of recovery-share files before attempting a costly OPAQUE-gated reconstruction; periodic audit of cold-storage share artifacts; CI gate for share-export pipelines. Pure local computation — no daemon RPC, no secret material exposed. **Exit codes**: 0 valid, 1 args/parse error (missing file, bad JSON, bad flags), 2 structurally invalid (operator alert gate — share file would either fail reconstruction or, worse, succeed at producing a meaningless value). Exercised end-to-end by `tools/test_wallet_shamir_verify.sh`. |
| `determ-wallet account-create-batch --count N [--out <file>] [--json] [--force]` | Bulk anonymous-account keypair generator. Produces `N` fresh Ed25519 keypairs (Ed25519 → anon-address derivation per the canonical wallet address scheme; each call uses the OS CSPRNG, never the deterministic-test seed path) in a single invocation, for operator workflows that need to pre-mint a large account set (load-testing fixtures, recovery-rehearsal cohorts, deterministic-genesis seed accounts, integration-test scenarios). Validates `1 ≤ N ≤ 10000` (hard cap prevents accidental memory blow-up — operators with larger needs should script multiple invocations). Output modes are mutually exclusive: `--out <file>` writes a JSON envelope `{count: N, accounts: [{ed_priv_hex, ed_pub_hex, anon_address}, …]}` to disk; `--json` (without `--out`) emits the same JSON shape to stdout; supplying neither emits a human-readable per-keypair listing (`account[i]: anon=… ed_pub=… ed_priv=…`). Refuses to overwrite an existing `--out` target unless `--force` is supplied (protects against destructive re-runs). Includes RNG-freshness lock-in: repeated invocations with the same `--count` produce disjoint keypair sets (the implementation does not seed from any fixed input, and the test wrapper asserts this property across 30-40 assertions covering small/medium batch sizes, JSON-output validity, anon-address-format validity, RNG-freshness across calls, and validation rejections — empty count, count > 10000, malformed flags). Exercised end-to-end by `tools/test_wallet_account_create_batch.sh`. **Exit codes**: 0 batch generated, 1 args/parse error (missing `--count`, out-of-range `N`, `--out` file exists without `--force`, mutually-exclusive flag conflict), 2 RNG failure / file I/O failure. |
| `determ-wallet backup-verify --shares <file> --envelopes <file> [--threshold T] [--json]` | Full wallet-backup structural verification. Validates a complete recovery artifact (a Shamir share-set file + a per-share AEAD-envelope file) **without decrypting**. Checks the shape of each input independently, then cross-checks them: every share in `--shares` must have a corresponding envelope in `--envelopes` (1:1 mapping, same count, share `x`-values match envelope-array indices in canonical order), each envelope passes `envelope::deserialize()` for header parsing only (never derives a key, never invokes AES-GCM), and the optional `--threshold T` cross-check rejects backups where `N < T` (the threshold cannot be reconstructed from the saved share count). Distinct from `shamir-verify` (which only inspects the share file in isolation): `backup-verify` is the operator-facing single command that verifies a complete recovery bundle is internally consistent before attempting a costly OPAQUE-gated reconstruction. Default human output is a per-check pass/fail line plus a summary `backup_valid=YES (shares=N envelopes=N threshold=T)` or `backup_valid=NO (reason=…)`; `--json` emits `{valid, share_count, envelope_count, threshold_ok, errors: [...]}`. Pure local computation — no daemon RPC required, no secret material exposed. Use cases: pre-flight verification of a freshly-exported recovery bundle, periodic audit of cold-storage backups, CI gate for share-export pipelines. Exercised end-to-end by `tools/test_wallet_backup_verify.sh` (~30-40 assertions). **Exit codes**: 0 valid backup, 1 args/parse error (missing files, bad JSON, bad flags), 2 structurally invalid (operator alert gate — backup would fail reconstruction or fail envelope-deserialization on recovery). |
| `determ-wallet backup-create --secret <hex> --threshold T --keyholders <file> --shares-out <file> --envelopes-out <file> [--force] [--json]` | Wallet-backup generator — the **inverse** of `backup-verify`. Given a hex-decoded master secret, an SSS threshold `T`, and a keyholders file (JSON `{keyholders: [{x: <int>, passphrase: "..."}, …]}` mapping each share's `x`-coordinate to its keyholder's passphrase), splits the secret via Shamir over GF(2^8) into N shares (N = keyholders.length), encrypts each share independently under an AEAD envelope (AES-256-GCM + PBKDF2 key derivation per the canonical wallet envelope scheme; matches the `envelope encrypt` primitive) using that keyholder's passphrase, and writes the resulting `--shares-out` (JSON `{shares: [{x, y_hex}, …]}` — identical shape to `shamir-split` output) and `--envelopes-out` (JSON `{envelopes: [<canonical hex blob>, …]}` — one per keyholder, indexed by share-`x` canonical order) files atomically. | `determ-wallet keyfile-create --priv <hex> --passphrase-from <source> --out <file> [--force] [--json]` | Passphrase-encrypted `node_key.json` generator — the S-004 closure use-case (passphrase-encrypted node-key files at rest). | `determ-wallet message-sign --priv <hex> --domain <tag> --message <bytes-or-@file> [--json]` | Off-chain Ed25519 message signer — distinct from transaction signing (which targets the chain's `signing_bytes` schema).
