# BlockDigestCrossBinaryParity — byte-identity of the light client's block-digest recompute vs the producer's, on every block (incl. F2/cross-shard) (FB62)

This document formalizes the **cross-binary block-digest parity invariant**: that `determ-light`'s independent recompute of the K-of-K committee's Phase-2 signature target (`light_compute_block_digest`) is **byte-for-byte identical** to the daemon's authoritative definition (`compute_block_digest`) on **every** block, including F2 / cross-shard blocks with non-empty `inbound_receipts` or non-zero `creator_view_eq_roots` / `creator_view_abort_roots`, as well as merged blocks (non-zero `partner_subset_hash`) and timestamp-reconciled blocks. The two functions bind the **same field set in the same order** under the **same data-driven gates** — including the three F2 view roots (inbound / eq / abort) — so for any `Block` value the two digests coincide exactly, and the light client's Ed25519 verification of each committee signature is sound: a signature the committee produced over the daemon's digest verifies under the light client's recomputed digest, and a tampered header fails.

The distinction between block classes is now purely **operational**, not a soundness boundary. When the light client is fed an `rpc_headers`-STRIPPED header (the heavy F2 collections removed, view roots zero), every F2 gate is false and the digest collapses to the byte-identical v1 core — so non-F2 header sync runs on the fast, header-only path. When a stripped header's committee-sig check fails (the case of an F2 / cross-shard block whose digest the stripped header cannot reproduce), the walk re-fetches the **full block** and re-verifies against the now-F2-aware digest, which binds the three roots the committee signed. Both paths are byte-exact; neither fail-closes. This closed finding **F-7** (see §4.2): previously `light_compute_block_digest` was a deliberate *partial* mirror that omitted the three F2 roots and fail-closed on every F2 block, breaking trustless reads on any shard chain that had applied a cross-shard receipt.

It is the digest-layer analog of `CanonicalSigningBytesParity.md` (FB61), which discharges the same kind of obligation for the Transaction `signing_bytes` pre-image. Where FB61 keeps "the transaction" a single binary-independent object, this proof keeps "the digest the committee signed" a single binary-independent object across the producer and light binaries — on **all** block classes, not just a non-F2 subset.

**Companion documents.** `Preliminaries.md` (F0) §2.0 canonical assumption labels — **A1** = Ed25519 EUF-CMA (§2.2), **A2** = SHA-256 collision/preimage resistance (§2.1). `S030-D2-Analysis.md` (the digest field model: which block fields are bound into `compute_block_digest`, the F2 view-reconciliation that gates the three pool-fed roots, and the `partner_subset_hash` / `timestamp` deterministic-and-median tail). `CanonicalSigningBytesParity.md` (FB61 — the sibling byte-identity proof for the tx pre-image; same proof shape, same `tools/`-guard discipline). `CrossBinaryCanonicalFormat.md` (CBF — CBF-2 names `light/verify.cpp::light_compute_block_digest` as a re-implementation surface that "must stay byte-identical to the daemon"; this proof is the byte-identity discharge of that obligation on **every** block class). `Safety.md` (FA1 — the K-of-K committee signature is the FA1 finalization target this digest IS). `LightClientThreatModel.md` / `LightClientCompositionMap.md` (the T-L block-sig-verification leg whose soundness reduces to this parity; the F-7 full-block fallback shares the `committee_bound_state_root` full-block-recompute trust step those docs cover).

## 1. The invariant

### 1.1 The digest field model

A `Block b` is signed by its K-of-K committee in Phase 2 over `compute_block_digest(b)` — a single SHA-256 over an ordered token stream (`src/node/producer.cpp:619-704`). The token stream is a fixed **core** of 10 field-groups followed by up to five **conditional appendages**, each gated on an intrinsic, JSON-stable signal in the block. The light client (`light/verify.cpp:124-191`) binds the **same** groups under the **same** gates:

| Group | Fields | Gate | producer.cpp | light/verify.cpp |
|---|---|---|---|---|
| core (10) | `index`, `prev_hash`, `tx_root`, `delay_seed`, `consensus_mode`, `bft_proposer`, `creators`, `creator_tx_lists`, `creator_ed_sigs`, `creator_dh_inputs` | always | `:621-631` | `:126-136` |
| F2 inbound | root over `hash_cross_shard_receipt(inbound_receipts)` | `!inbound_receipts.empty()` | `:640-646` | `:146-151` |
| F2 eq | root over `hash_equivocation_event(equivocation_events)` | `any_nonzero(creator_view_eq_roots)` | `:661-666` | `:157-162` |
| F2 abort | root over `hash_abort_event(abort_events)` | `any_nonzero(creator_view_abort_roots)` | `:667-672` | `:163-168` |
| partner | `partner_subset_hash` | `!is_zero_hash_(partner_subset_hash)` | `:685-687` | `:175-177` |
| timestamp | `b.timestamp` | `!creator_proposer_times.empty()` | `:700-702` | `:187-189` |

The three **F2** appendages (inbound / eq / abort) are the v2.7 F2 / S-030-D2 view-reconciliation bindings — they bind a *reconciled* set computed from the K signed Phase-1 commits (`S030-D2-Analysis.md` §3.2, §4 items 7+8). The **partner** and **timestamp** appendages are NOT pool-fed views: `partner_subset_hash` is deterministic (every committee member at a merged height computes the identical value from the merge state — `S030-D2-Analysis.md` §3.2, item 9) and `timestamp` is the deterministic lower-median of the K committed `proposer_time`s (`S030-D2-Analysis.md` §5, item 10). Field order is fixed and identical in both functions: **core, inbound, eq, abort, partner_subset_hash, timestamp** (`producer.cpp:656/684/699`; `light/verify.cpp:145/173/186` order comments).

The four F2 per-record helpers (`hash_cross_shard_receipt`, `hash_equivocation_event`, `hash_abort_event`, `compute_view_root`) that the F2 roots are built over are defined once in the daemon (`producer.cpp:354-368` / `:322-337` / `:339-352` / `:377-382`) and **re-implemented byte-for-byte as file-local statics** in an anonymous namespace at the top of the light client (`light/verify.cpp:34-95`: rcpt `:47-61`, eq `:63-75`, abort `:77-86`, view-root `:88-93`). The light client deliberately does NOT link `producer.cpp` (a minimal-footprint reuse list in `CMakeLists.txt`), so these four are duplicated and pinned BYTE-FOR-BYTE against the originals by the source guard (§5).

### 1.2 Stripped-header vs full-block operation (not a soundness boundary)

The F2 gates are **data-driven**: they read intrinsic `Block` fields (`inbound_receipts`, `creator_view_eq_roots`, `creator_view_abort_roots`), not a "block class" flag. Two operational regimes follow from how the walk feeds those fields:

- **Stripped-header fast path.** `rpc_headers` strips the heavy F2 collections and zeroes the view roots, so a stripped header has all three F2 gates FALSE in *both* functions. The producer's digest on such a header (re-derived from the same stripped fields) and the light client's digest both emit **core + (partner if non-zero) + (timestamp if reconciled)** — the byte-identical v1 core. Every ordinary single-shard block, every **merged** block (non-zero `partner_subset_hash`), and every **timestamp-reconciled** block is verified header-only on this path.
- **Full-block fallback.** When a stripped header's committee-sig check fails — the symptom of an F2 / cross-shard block whose committee signed the *full* digest (F2 roots bound) that the stripped header can no longer reproduce — the walk re-fetches the FULL block (heavy fields intact) and re-verifies against the now-F2-aware light digest, which binds the same inbound / eq / abort roots the committee signed (§4.2). The fallback fires ONLY on F2 blocks; non-F2 chains incur zero extra fetches.

Both regimes are byte-exact. There is **no fail-close boundary** — the prior partial mirror (which omitted the F2 roots and refused F2 blocks) was finding **F-7**, now closed.

### 1.3 Statement

> **BDP (block-digest parity).** For every block `b` (any class),
> `light_compute_block_digest(b) == compute_block_digest(b)` (byte-for-byte),
>
> hence for each committee member `i`, `crypto::verify(pk_i, light_digest, sig_i)` returns the same boolean as `crypto::verify(pk_i, producer_digest, sig_i)`. Since the committee produced `sig_i` over `producer_digest`, the light client's K-of-K verification (`light/verify.cpp::verify_block_sigs`, digest at `:382`, Ed25519 verify at `:398`) — fed either the stripped header (non-F2) or the re-fetched full block (F2, via the walk's fallback) — accepts exactly the genuinely-signed headers and rejects any post-signing tamper (A1).

BDP is an **unconditional byte equality** on the full block domain — both functions are pure straight-line SHA-256 builders over the same `Block` fields in the same order, gated on the same intrinsic signals; A1 enters only to lift digest equality to signature-verification equivalence, A2 only for §6's divergence-detection direction.

## 2. Preliminaries

We rely only on:

- **(P-det) Determinism of each builder.** `compute_block_digest` and `light_compute_block_digest` are pure functions of the `Block` argument — a fresh `SHA256Builder`, a straight-line sequence of `h.append(...)` calls, no clock, no RNG, no global mutable state, no iteration over an unordered container whose order is not already fixed in the block (`creators` / `creator_tx_lists` / `creator_dh_inputs` are committee-selection-ordered vectors, identical in both). A pure straight-line builder returns the same 32 bytes on the same `Block` on every host. (This is the digest specialization of `CrossBinaryCanonicalFormat.md` §4.3's determinism premise; the in-process `determ` leg is pinned by `tools/test_block_digest.sh`.)
- **(P-app) `SHA256Builder::append` is type-stable across the two binaries.** Both functions call the *same* `determ::crypto::SHA256Builder` (`#include <determ/crypto/sha256.hpp>`; `light/verify.cpp:19`), and both feed it the same C++ types from the same `determ::chain::Block` struct (`light/verify.cpp:22` includes `<determ/chain/block.hpp>` — the light client links the canonical `Block`, it does not re-declare it). The four F2 per-record helpers are file-local statics that take the canonical `determ::chain::CrossShardReceipt` / `EquivocationEvent` / `AbortEvent` structs (`light/verify.cpp:47/63/77`), so the F2 root pre-images are over the same fields in the same order as the daemon. So `append(b.index)` etc. — and `hash_*` of each F2 record — serialize identically by construction; there is no second `Block` layout to drift against. (This is strictly stronger than FB61's tx-pre-image case, where the wallet re-declares the byte layout inline — here only the *call sequence* and the four small helper bodies can drift, not the per-field encoding.)
- **A1** (Ed25519 EUF-CMA) lifts digest byte-equality to signature-verification equivalence in §3.2/§4 (a verifier recomputing the identical message bytes decides each signature the same way the signer intended).
- **A2** (SHA-256 collision/preimage resistance) enters only in §5/§6 for the divergence-*detection* direction (a drifted token sequence overwhelmingly yields a different digest, so a guard reliably flags it); the parity equality itself is exact.

The two functions are deliberately decoupled: `light/verify.cpp:97-123` (the function-header comment block) marks `light_compute_block_digest` "COPY OF producer.cpp::compute_block_digest (src/node/producer.cpp:619-704) — keep in sync," states "FULL PARITY (F-7): this is now a byte-for-byte mirror of the node's compute_block_digest — it binds EVERY field the node does, including the three conditional F2 view roots ... and the merged-block tail," and closes "If the upstream byte-order or field set ever changes, mirror it here." The copy exists because the light client is a separate binary that the daemon's `compute_block_digest` does not link (and that deliberately does NOT link `producer.cpp` — `light/verify.cpp:36-44`); the parity is therefore a maintained mirror, not a shared call — exactly the re-implementation regime CBF-2 flags and §5's source guard polices.

## 3. The producer's digest is the spec

### 3.1 Theorem T-1 (producer defines the spec)

**Statement.** `compute_block_digest(b)` (`producer.cpp:619-704`) is the digest the K-of-K committee actually signs in Phase 2; it *defines* the reference token sequence against which the light recompute is measured.

**Proof.** The committee's Phase-2 `BlockSigMsg` carries an Ed25519 signature over `compute_block_digest(b)` (the Phase-2 sign target; `producer.cpp` digest at `:619`, consumed at the round-finalize site). Validators recompute it (`src/node/validator.cpp` `check_block_sigs`) and the light client recomputes it (`verify.cpp:382`). There is exactly one such function in `src/`; `determ` links it directly. Whatever bytes it emits ARE the message the signatures attest to — there is no obligation beyond exhibiting the token stream, which §1.1's table maps field-by-field to `producer.cpp:621-702`. T-1 is the definitional anchor; T-2/T-3 are proved relative to it. ∎

The **three F2 roots** (F2-a/F2-b/F2-c) now appear in BOTH functions, in the same position and under the same data-driven gate — the light client re-implements the four F2 per-record helpers as file-local statics (`verify.cpp:34-95`) and the `any_nonzero` predicate inline (`verify.cpp:152-156`), so on any `Block` value its F2 appendages reproduce the producer's exactly. On an `rpc_headers`-STRIPPED header the F2 collections are empty / the view roots are zero, so all three gates are false in both functions and the digest collapses to the v1 core; on a full block (re-fetched by the walk's F2 fallback, §4.2) they are populated and both functions bind the same roots. Crucially, every gate is *intrinsic to the block value*: a merged block (non-zero `partner_subset_hash`) emits position 11 in BOTH functions, an F2 block emits the relevant F2 root in BOTH, and the byte streams stay identical. This is why BDP holds on every block class — plain single-shard, merged, timestamp-reconciled, and F2 / cross-shard alike.

## 4. Theorems

### 4.1 Theorem T-2 (light == spec on every block)

**Statement.** For every block `b` (any class), `light_compute_block_digest(b) == compute_block_digest(b)` byte-for-byte; hence each committee signature decides identically under both digests, and `verify_block_sigs` is sound (accepts the genuinely-signed header, rejects tamper) on whatever block representation it is fed — the stripped header (non-F2 fast path) or the re-fetched full block (F2 fallback, §4.2).

**Proof obligation.** Two parts: (i) the producer's emitted token sequence equals the light client's emitted sequence on every `Block` value (the source-guard equality, now over the full append set including the three F2 roots), and (ii) every conditional appendage's trigger coincides between the two functions (the conditional-trigger equality).

**Proof.**

*(i) Source token-sequence equality (= the source guard).* Positions 1-10 (the core) are emitted unconditionally in both functions and map token-for-token: `producer:621-631 ≡ light:126-136` (same `h.append` over the same `Block` field, in the same order, via the same `SHA256Builder` under (P-app)). The three F2 appendages are present in both functions with identical bodies: each builds its keys vector over the same canonical record helper (`hash_cross_shard_receipt` / `hash_equivocation_event` / `hash_abort_event`, byte-identical across the binaries — `producer.cpp:354-368/322-337/339-352 ≡ verify.cpp:47-61/63-75/77-86`) and appends `compute_view_root` of it (`producer.cpp:645/665/671 ≡ verify.cpp:150/161/167`, the helper itself byte-identical — `producer.cpp:377-382 ≡ verify.cpp:88-93`). So the two functions emit the identical ordered token stream over positions 1-10 and the F2 slots, independent of the tail gates.

*(ii) Conditional-trigger equality.* Each conditional appendage must fire on the same blocks in both functions:
- **F2 inbound** (F2-a): both gate on `!b.inbound_receipts.empty()` (`producer:640 ≡ verify:146`) — syntactically identical over the identical `Block` field.
- **F2 eq** (F2-b): both gate on `any_nonzero(b.creator_view_eq_roots)` (`producer:661 ≡ verify:157`). The `any_nonzero` predicate is defined identically: producer's lambda (`producer:657-660`) returns true iff some root is non-zero; light's lambda (`verify:152-156`) does the same with an explicit zero `Hash`. Equal results on every input.
- **F2 abort** (F2-c): both gate on `any_nonzero(b.creator_view_abort_roots)` (`producer:667 ≡ verify:163`), via the same `any_nonzero` predicate.
- **partner** (position 11): producer gate `!is_zero_hash_(b.partner_subset_hash)` (`:685`) vs light gate `b.partner_subset_hash != zero` (`:175`). Both test "is `partner_subset_hash` non-zero," over the identical `Block` field; `is_zero_hash_` is the all-zero-bytes predicate and `!= zero` is its De-Morgan complement, so they fire on exactly the same blocks, and when they fire both append the identical 32-byte `b.partner_subset_hash` (`:686 ≡ :176`).
- **timestamp** (position 12): producer gate `!b.creator_proposer_times.empty()` (`:700`) vs light gate `!b.creator_proposer_times.empty()` (`:187`) — *syntactically identical* predicate over the identical `Block` field; when fired both append `b.timestamp` (`:701 ≡ :188`). Because `creator_proposer_times` survives the `rpc_headers` strip and `from_json` repopulates it (`src/chain/block.cpp:586-588`), the light client's `b.creator_proposer_times` is the same vector the producer digested, so the gate evaluates identically.

Combining (i) and (ii): on any `Block` the two functions emit the identical ordered token stream — core, then (iff present/reconciled) inbound, eq, abort, then (iff non-zero) partner, then (iff reconciled) timestamp — and feed it to the same `SHA256Builder`. By (P-det)+(P-app) the finalized 32-byte digests are byte-equal:

```
∀ b: light_compute_block_digest(b) == compute_block_digest(b).
```

*Soundness of `verify_block_sigs`.* `verify.cpp:382` computes `digest = light_compute_block_digest(b)` and `:398` runs `crypto::verify(pk_i, digest, sig_i)` for each creator. The committee produced `sig_i` over `compute_block_digest(b)` of the *full* block. For a non-F2 block, the stripped header `b` reproduces that digest (all F2 gates false in both); for an F2 block, the walk's fallback feeds the re-fetched full block (§4.2), whose populated collections make `light_compute_block_digest` bind the same F2 roots. Either way the light digest equals the signed digest, so under A1 each `verify` returns TRUE for a genuine signature and FALSE for any post-signing tamper (a flipped `tx_root`, a swapped `creators` entry, a stripped/altered `partner_subset_hash`, a tampered reconciled `timestamp`, a removed/altered inbound receipt) — the verifier's message argument matches the signed bytes iff the body is intact. Therefore `verify_block_sigs` accepts exactly the genuinely-signed headers/blocks and rejects tamper. ∎

**Mechanical discharge.** `tools/test_block_digest_xbinary_parity.sh` (§5) reduces both source sites to ordered token lists and asserts the light list equals the producer list **exactly** (full F-7 parity, all 15 tokens incl. `INBOUND_ROOT`/`EQ_ROOT`/`ABORT_ROOT`; `:81-88`, `:113-116`) AND the partner/timestamp tail in the same order under the same gates — turning any drift RED at the source level pre-build. The runtime end-to-end leg is `tools/test_light_verify_block_sigs.sh` (a cluster mints a real block 1, the light client verifies its K-of-K sigs against its own recompute); the F2 fallback's runtime leg is `tools/test_light_verify_receipt_inclusion.sh` (a cross-shard receipt is applied, and the light client reports INCLUDED at the F2 block via the full-block re-fetch).

### 4.2 Theorem T-3 (light verifies F2 / cross-shard blocks via the full-block fallback)

**Statement.** On an **F2 / cross-shard** block (non-empty `inbound_receipts`, or non-zero `creator_view_eq_roots`, or non-zero `creator_view_abort_roots`), the chain walk re-fetches the FULL block and verifies its K-of-K signatures against the now-F2-aware light digest, which binds the same three roots the committee signed (T-2). The fallback is **sound**: a daemon can neither forge the committee signatures over a doctored body nor substitute a differently-positioned validly-signed block, because the full body's recomputed `compute_hash()` is pinned to the stripped header's `block_hash` that `verify_headers` already chained.

**Proof.** Let `b` be F2: at least one of the three F2 gates is TRUE on the *full* block, so `compute_block_digest(full)` contains the corresponding F2 root token (one of `producer:645` / `:665` / `:671`) — the digest the committee signed. The `rpc_headers`-STRIPPED header presented first has those collections removed and view roots zero, so its light digest collapses to the v1 core and the K signatures (which sign the full F2 digest) fail to verify; the walk's per-block check `verify_block_sigs(h, …)` returns FAIL in both MD and BFT modes (`trustless_read.cpp:191`, BFT retry `:196`). This triggers the **F-7 full-block fallback** (`trustless_read.cpp:198-248`):

1. The walk reads the stripped header's `block_hash` — the value `verify_headers` already chained into `prev_anchor` (`:218`).
2. It re-fetches the full block via `rpc.call("block", {index})` (`:221`), parses it (`:225-226`), and recomputes `fb.compute_hash()` (`Block::compute_hash`, `src/chain/block.cpp:356`).
3. **SOUNDNESS PIN:** it requires `to_hex(fb.compute_hash()) == chained_hash` (`:227`) — the full body must hash to exactly the stripped header's already-chained `block_hash`. A daemon therefore cannot substitute a differently-positioned but validly-signed block to satisfy the per-block check while continuity tracks another hash.
4. Only on that match does it re-run `verify_block_sigs(full, …)` (MD then BFT, `:228-232`) against the now-F2-aware light digest. By T-2, `light_compute_block_digest(full) == compute_block_digest(full)`, so under A1 the K genuine signatures verify and the block is accepted (`recovered = true`, `:233`).

Soundness has two legs, both reducing to A1+A2: (a) a daemon cannot forge the committee signatures, so a *doctored* full body changes `compute_block_digest` (A2: a different token stream collides with probability `≤ 2⁻¹²⁸`) and the K sigs fail at step 4 — the block is rejected (`recovered` stays false → throw, `:240-247`); (b) a daemon cannot pass off a *different* validly-signed block, because step 3 pins the body to the `block_hash` `verify_headers` chained into the running anchor (and A2 makes a second body with that hash infeasible). This is the **same** full-block-recompute trust step `committee_bound_state_root` already relies on (`trustless_read.cpp:465`; the comment at `:217` names it explicitly). The fallback fires ONLY on F2 blocks: a non-F2 chain's stripped headers verify on the fast path and never enter the `!vbs.ok` branch, so it incurs zero extra `block` fetches. ∎

**Note (why a forged F2 block cannot be accepted).** Acceptance requires BOTH the `block_hash` pin (step 3) AND the K committee signatures over the F2 digest (step 4). The light client now *does* compute the producer's exact F2 digest (T-2), so it cannot be tricked into accepting a body the committee did not sign: a doctored body either fails the hash pin (if it doesn't hash to the chained `block_hash`) or fails the sig check (its digest no longer matches the K signatures). The only residual cost is one extra `block` RPC per F2 header — an availability/bandwidth cost, never a soundness one. The §6 "Direction 1/2" tail-drift risks remain the *maintenance* hazard the §5 source guard polices.

### 4.3 Finding F-7 (closed) — completing the light digest

**Before.** `light_compute_block_digest` was a deliberate *partial* mirror of `compute_block_digest`: it bound the v1 core + `partner_subset_hash` + `timestamp` but OMITTED the three F2 view roots (inbound / eq / abort), because it could not reconstruct them from an `rpc_headers`-STRIPPED header. As a result, every committee signature over an F2 / cross-shard block failed the light client's `verify_block_sigs`, and the chain walk threw — breaking `verify-receipt-inclusion` and all trustless reads on any shard chain that had applied a cross-shard receipt.

**Direction-of-error safety (then as now).** The prior behavior was a **capability gap, never a soundness bug**: the light client's digest was *missing* tokens the producer included, never a superset, so it could not compute the producer's exact F2 digest for *any* F2 block — hence it could never assemble K passing signatures over a digest an attacker would also satisfy. The only reachable error was a refusal to attest a genuine F2 header (false NEGATIVE / UNVERIFIABLE), never a false INCLUDED.

**Fix (Path 3 — "complete the light digest", commit `dc7bb3e`).** Two coordinated changes closed it:
1. `light/verify.cpp` (`:124-191`) now binds the three F2 roots on the SAME data-driven gates as the node (`:146-168`), making `light_compute_block_digest` a FULL byte-for-byte mirror of `compute_block_digest`. The four F2 helpers are re-implemented as file-local statics in an anonymous namespace (`:34-95`). On a stripped header every F2 gate is false (collapse to v1 core); on a full block they bind the same roots the committee signed (T-2).
2. `light/trustless_read.cpp::verify_chain_walk` (`:198-248`) gained the F2 full-block fallback with the `block_hash` soundness pin (T-3): when a stripped header's committee-sig check fails, it re-fetches the full block, pins `compute_hash() == chained block_hash`, and re-verifies against the now-F2-aware digest.

**Guard.** The source guard `tools/test_block_digest_xbinary_parity.sh` was FLIPPED from "light == producer MINUS the 3 F2 roots" to "light == producer EXACTLY" (full byte-parity — a STRONGER invariant; `:81-88`, `:113-116`). The SELFTEST's old "light binds an F2 root → RED" case was replaced by "light MISSING an F2 root → RED" — the post-F-7 failure mode (`:449-490`). Validation on the fix: `test_light_verify_receipt_inclusion` (applied receipt → INCLUDED at the F2 block via the fallback), xbinary parity (live + SELFTEST), `test_light_verify_block_sigs`, `test_light_verify_chain`, `test_light_stake_trustless`. `producer.cpp::compute_block_digest` was UNTOUCHED.

## 5. Mechanized witnesses

Defense-in-depth, mirroring FB61's layering: a **source-level guard** catches a maintainer's drift in either digest function *before* a build (critical on hosts that cannot build all binaries), a **producer-side in-process field fence** pins the daemon's digest shape, and a **cluster end-to-end** test confirms a real light binary verifies a real minted block's K-of-K signatures against its own recompute.

| Layer | Script | What it pins | Live-check (negative control) |
|---|---|---|---|
| Source (pre-build) | `tools/test_block_digest_xbinary_parity.sh` | Parses BOTH C++ sites (`src/node/producer.cpp::compute_block_digest` and `light/verify.cpp::light_compute_block_digest`), reduces each to an ordered `append`-token list (15 tokens incl. `INBOUND_ROOT`/`EQ_ROOT`/`ABORT_ROOT`; `:113`), and asserts: (a) the producer list equals the canonical 15-token sequence; (b) the light list equals the producer list **EXACTLY** (full F-7 byte-parity — same field set, same order, all three F2 roots in both; `:81-88`, `:572-587`); (c) BOTH binaries contain all three F2 root tokens (`:594-597`); (d) the **partner** + **timestamp** tail is gated identically in both (`is_zero_hash_`/`!= zero` → `NONZERO_PSH`; the identical `!creator_proposer_times.empty()` → `NONEMPTY_PT`; `:605-625`). Pure `awk`/`grep` over `.cpp`; no binary; never SKIPs; offline; deterministic. A dropped/added/reordered field, a moved tail, or an F2 root missing from EITHER copy all break the check. | `SELFTEST=1 bash …` (`:224-544`) runs a liveness self-check feeding synthetic DRIFTED snippets through the same `extract_tokens()` + cross-site logic: (1) full-parity sanity (producer == light), (2) a swapped partner/timestamp tail in light, (3) a producer missing an F2 root, (4) **a light copy MISSING an F2 root — the post-F-7 failure mode** (`:449-490`, replacing the obsolete "light binds an F2 root → RED" case), (5) a dropped core field. | Cluster (end-to-end) | `tools/test_light_verify_block_sigs.sh` | Boots a cluster, mints a real **block 1** (the common non-merged, non-F2 case), serves its header via `rpc_headers`, and runs `determ-light verify-block-sigs` — asserting the light binary's `light_compute_block_digest` recompute (B) reproduces the daemon's `compute_block_digest` (A) closely enough that all K committee Ed25519 sigs verify (the `(B)==(A)` end-to-end equality for the non-merged common case). | A tamper variant mutates a header field after signing; the light client's recomputed digest must then DIFFER and the sig check must FAIL — proving the verification is a live check, not a green-by-default pass. |

**Coverage split (why all three).** The cluster test (`test_light_verify_block_sigs.sh`) pins the **common case at runtime** — but the block-1 it mints is plain: non-merged (`partner_subset_hash == 0`), non-reconciled, and non-F2 in the minimal harness, so its digest exercises only the **core 10** and *never triggers* the F2 roots or the partner/timestamp tail. The conditional **F2 roots + merged/reconciled tail** — positions F2-a/b/c and 11-12, the sub-domain where BDP is least obvious because the two functions each run a *conditional* append — is exactly what the source guard (`test_block_digest_xbinary_parity.sh`) pins: it asserts the full append set's order and matching gates *statically*, on every host, without needing a cluster to mint an F2 / merged block. The F2-fallback runtime leg is `test_light_verify_receipt_inclusion.sh` (an applied cross-shard receipt → INCLUDED at the F2 block via the full-block re-fetch). The producer fence (`test_block_digest.sh`) closes the third side: it confirms the daemon's `compute_block_digest` actually fires the conditional appendages on a merged/reconciled block (so the source guard is asserting parity against a *live* feature, not dead code). Source guard ⇒ the mirror is faithful pre-build (incl. the F2 roots + tail the basic cluster block never reaches); producer fence ⇒ the daemon side of the mirror is real; cluster ⇒ a built light binary verifies a built daemon's signatures end-to-end. The negative controls make each layer a *live* equality/verification check rather than a tautology.

## 6. Threat / why it matters

The block digest is the **FA1 committee-signature target**: the K-of-K Phase-2 signatures attest to `compute_block_digest(b)`, and that attestation is what a light client trusts in place of replaying the chain. A drift between the two digest functions is therefore a consensus-adjacent fault, not a cosmetic one, and it bites in *both* directions.

**Direction 1 — light rejects valid sigs (liveness / availability loss for light clients).** Suppose a maintainer edits `compute_block_digest` — say reorders `partner_subset_hash` and `timestamp`, changes the partner gate, or drops one of the three F2 roots — but forgets to mirror it in `light_compute_block_digest` (the `verify.cpp:97-123` "keep in sync" header comment exists precisely because this is a separate binary the daemon does not link). Then on every block of the affected class (a **merged** block for a partner/timestamp drift; an **F2 / cross-shard** block for an F2-root drift) the light client recomputes a digest that no longer matches the committee's signed digest, and `verify_block_sigs` rejects a perfectly valid block as UNVERIFIABLE (`:401-404`) — and, post-F-7, the walk's full-block fallback also fails (the re-fetched full body still recomputes the *drifted* light digest), so the chain walk throws. Light clients would silently lose the ability to verify the affected header class — a targeted availability outage that a single-shard, non-F2 test harness would not surface. The source guard turns this RED at review time, on any host, before a build.

**Direction 2 — producer-side bug lets two distinct blocks share a digest (FA1 equivocation-binding loss).** Symmetrically, a bug on the *producer* side — e.g. failing to bind `partner_subset_hash`, or an F2 root, when it should — would let two distinct blocks at a height carry the *same* digest while differing in the omitted commitment. Both would collect the same K-of-K signatures; FA1's "≤ 1 finalized digest per height" would hold *vacuously* while two distinct bodies circulated behind it — exactly the S-030-D2 removal gap that the `partner_subset_hash` digest binding (commit `8585a50`, `S030-D2-Analysis.md` item 9) and the three F2-root bindings (v2.7 F2 / S-016) were added to close. The producer fence (`test_block_digest.sh`) pins that the daemon binds the fields; this proof pins that the light client mirrors the bindings, so the *verifier* side of the FA1 guarantee is also drift-protected. (The full S-030-D2 analysis — which fields the digest must bind and why the F2 roots use reconciliation while partner/timestamp do not — lives in `S030-D2-Analysis.md`; this proof is the cross-binary-parity complement: given that the producer binds the right fields, the light client recomputes the same digest on every block class.)

**Why full parity (not fail-close) is the right design.** Before F-7 the light client handled F2 / cross-shard blocks by structural omission — it could not reconstruct the F2 roots from a stripped header, so it fail-closed (every sig failed, the block was UNVERIFIABLE, and the operator was routed to a full node). That was direction-of-error-safe (an availability loss, never a false acceptance) but it broke trustless reads on any chain that had applied a cross-shard receipt. F-7 closes the capability gap: the light digest now binds the F2 roots on the same data-driven gates as the node, and the walk re-fetches the full block (with the `block_hash` soundness pin, T-3) whenever the stripped-header digest cannot reproduce the signed digest. The result is byte-parity on **every** block class with no soundness give-up — the full-block fallback fetches one extra `block` per F2 header but accepts only a body whose recomputed hash matches the already-chained `block_hash` and whose K sigs verify against the F2-aware digest. BDP is the invariant that keeps "the digest the committee signed" a single binary-independent object across the producer and light binaries on all block classes, and the three mechanized witnesses keep BDP from rotting.
