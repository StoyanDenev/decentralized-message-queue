# EncryptedNoteDeliveryDesign — shielded-pool Option A (NC-8): on-chain encrypted-note delivery so a recipient can SCAN + trial-decrypt a confidential output's note secret without an out-of-band channel

This is the **design record** for **shielded Option A** — an on-chain, ephemeral-static ECIES channel that delivers a confidential output's note secret `(value, blinding[, memo])` to its recipient, closing [`ShieldedPoolSoundness.md`](ShieldedPoolSoundness.md) **NC-8** ("no on-chain output-secret delivery — recipient channel is off-chain"). It is the receiver amount-communication channel that [`ConfidentialTxIntegrationDesign.md`](ConfidentialTxIntegrationDesign.md) §1.3 flagged as *"a hard prerequisite"* to a usable confidential transfer, now built — over **P-256** (the shipped CT curve) rather than the X25519 handshake that older document anticipated.

**What shipped in the FIRST increment is the crypto primitive only** (`src/crypto/enote/enote.c` + its header), plus its dual-oracle test gate — **library-only, additive, and state-root-invariant**: no chain, consensus, validator, producer, wallet, or light-client code path calls it. The consensus wiring that carries a ciphertext on a `CONFIDENTIAL_TRANSFER` output — and the scan RPC / light-client scan that consume it — was laid out in §5 with genuine sub-decisions. **Those sub-decisions are now owner-decided profile-keyed (§5), and wiring increments 1 (the genesis `CryptoProfile` pin, §5.1), 2 (the per-output enote region, §5.2), 3 (the `scan_enotes` RPC, §5.3), 4 (recipient note-key derivation, §5.4), and 5a (the on-chain `note_pk` publication tx `REGISTER_NOTE_KEY` + `nk:` leaf, §5.5) have shipped.** Increments 5b (the trustless read side, §5.5) and 6 (the light-client enote-inclusion scan, §5.6) have since also shipped — **the full NC-8 pipeline is now consensus-complete and trustlessly readable end-to-end** (derive → publish → discover → seal → carry → scan → confirm); only a determ-light *builder* for driving live INCLUDED legs remains (client tooling, no consensus surface).

**The load-bearing property this increment preserves is no-double-spend-by-design.** NC-8 delivery is a **passive payload**: it adds amount-recipient *delivery* only. It introduces **no** nullifier set, leaves inputs **named** (NC-7 unchanged), and does **not** touch the commitment-as-its-own-nullifier model, so the pool's bounded-unspent-commitment-set invariant — and every SP-* claim of `ShieldedPoolSoundness.md` — is fully preserved. It also introduces **no new hardness assumption**: security reduces to P-256 ECDH + HKDF-SHA256 (RFC 5869) + ChaCha20-Poly1305 (RFC 8439), all shipped, dual-oracle-frozen c99 primitives.

## 1. The problem (NC-8)

A confidential output today is a **bare Pedersen commitment** `C = v·G + r·H` with **no owner field**. Two facts follow directly from `ShieldedPoolSoundness.md`:

- **The recipient cannot spend an output they were paid** unless they learn its opening `(v, r)`. In a `CONFIDENTIAL_TRANSFER` (`TxType = 14`) the **sender** chooses each output's blinding `r`, so only the sender knows `(v, r)`. To later spend that note (as a named input to its own transfer, or to UNSHIELD it) the recipient must learn `(v, r)` — and §3.22c delivers **nothing** on-chain for this. NC-8, verbatim: *"the sender and recipient coordinate the output secret off-chain (or the sender transfers to their own future notes)."* That off-chain coordination is exactly what makes the confidential transfer unusable as a **payment** to a third party: there is no on-chain signal a wallet can scan to discover "a note was paid to me."

- **The A2 audit layer publishes an opaque key and can decrypt nothing on-chain.** The shipped audit layer (`ROTATE_AUDIT_KEY`/`LOG_AUDIT_ACCESS`, `TxType = 15/16`; `AuditLayerSoundness.md`) lets an account publish a standing view/audit pubkey, and `viewkey.c` derives per-epoch view keys from a view-master secret — but there is **no ciphertext on-chain** for such a key to open. An auditor holding the view key can prove *authority* to read amounts, yet there is nothing published to read. The amount is committed (hiding) and never delivered in a form any key can recover.

So NC-8 is a **delivery gap**, not a hiding gap: the chain hides amounts correctly (CTB-* / the DCT1 range∧balance bundle), but provides no channel to hand the opening to the party entitled to it (recipient, or authorized auditor). The absence of that channel is what keeps `CONFIDENTIAL_TRANSFER` a same-wallet / out-of-band primitive rather than a payment rail.

**What NC-8 is NOT.** It is not graph privacy and not input-unlinkability — those are **NC-7** (named inputs), a strictly larger, separately owner-gated crypto increment (nullifier-from-secret + set-membership). NC-8 and NC-7 are orthogonal: delivering the output secret says nothing about hiding *which* input was spent, and closing NC-8 neither helps nor harms NC-7.

## 2. The decision context — Option A (encrypted-note delivery) over graph-privacy (B/C) and do-nothing (D)

The owner (2026-07-17) chose **Option A: on-chain encrypted-note delivery**. The alternatives considered:

- **Option A — encrypted-note delivery (CHOSEN).** Publish, alongside each confidential output, a ciphertext sealing that output's `(v, r[, memo])` to the recipient's key. The recipient (or an authorized view-key holder) scans the chain and trial-decrypts; a verifying AEAD tag *is* the "this note is mine" signal. Delivers amount-recipient information **only** — no graph privacy. Reuses shipped primitives; no new hardness assumption; passive payload that never touches the state transition of the commitment set.

- **Option B — hidden inputs / input-unlinkability (NC-7).** A nullifier-from-secret + set-membership argument so a spend hides *which* note it consumes, breaking the deposit↔withdrawal and note-graph correlation. This is the genuinely private-payments increment — but it is a **materially larger** crypto build (a new nullifier set = new consensus state + a membership proof system) and it **replaces** the named-input model that the no-double-spend-by-design property and the SP-11 inflation dedup both rest on. Owner-deferred as its own track; not this increment.

- **Option C — stealth addresses / one-time output keys.** Per-output one-time addresses so an observer cannot link two outputs to the same recipient. This adds *receiver* graph privacy but still needs an amount-delivery channel (Option A) to be spendable, and it too enlarges the addressing/consensus surface. A superset of A's problem, not a substitute; deferred with B.

- **Option D — do nothing (keep the off-chain channel).** Leave NC-8 open: senders and recipients coordinate `(v, r)` out-of-band. Zero code, zero risk — but it is precisely what keeps confidential transfers unusable as third-party payments and leaves the A2 view key with nothing to open. Rejected as a permanent posture; acceptable only if the owner judged the payment use-case out-of-model.

**Why A preserves no-double-spend-by-design.** The pool's spend-safety is structural: a note is an entry in the bounded unspent-commitment set (`shielded_pool_`), the commitment **is its own nullifier** (apply erases it — SP-8), and there is **no separate nullifier set**. Option A adds a ciphertext that is **inert with respect to that machinery**: it is never an input to an accept-rule, never mutates the commitment set, never appears in the double-spend or inflation-dedup checks (SP-8/SP-11). It rides *alongside* an output as delivery metadata. Whether or not the ciphertext is present, malformed, or ignored, the set-membership state transition is byte-identical — so every SP-* claim, and the bounded-set / no-nullifier property, is untouched. (This is why the consensus follow-on in §5 must keep the enote **out of** any accept-rule: its presence must remain consensus-inert, at most block-hash-bound like `pq_auth`, never a spend precondition.)

**Why A stays inside the WHITEPAPER §12.1 on-chain-privacy non-goal.** §12.1 declares graph/on-chain privacy a v1.x non-goal. Option A does **not** cross that line: it hides nothing additional from third parties and reveals no less — inputs stay named, senders/recipients stay public, the transfer's existence stays public. It only routes the *already-hidden* amount secret to the one party entitled to spend it. Amount-privacy-in-motion (hiding amounts from observers) was already delivered by `CONFIDENTIAL_TRANSFER`; NC-8 is the dual — delivering the secret *to the recipient* — and adds **zero** observer-facing privacy. It is therefore inside the non-goal, not an exception to it.

## 3. The as-built construction — ephemeral-static ECIES over NIST P-256

A textbook DHIES/ECIES instantiation over **NIST P-256** — the CT stack's curve in **both** profiles (MODERN + FIPS), so the channel is profile-agnostic like the rest of §3.22. Composition **only**, from shipped, dual-oracle-frozen c99 primitives: **P-256 ECDH** (`determ_p256_*`) + **HKDF-SHA256** (RFC 5869, `determ_hkdf_sha256`) + **ChaCha20-Poly1305** (RFC 8439, `determ_chacha20_poly1305_*`). **No new primitive, no new hardness assumption.**

### 3.1 Wire format (frozen)

```
seal(recipient_pub[33 compressed], pt, eph_sk[32]):          # eph_sk fresh per note
    E     = eph_sk · G                                        # ephemeral pubkey
    E33   = compressed SEC1(E)                                # 33 bytes, 0x02/0x03 prefix
    Z     = eph_sk · R   (ECDH) ; z = Z.x                     # 32 big-endian bytes (shared x)
    okm   = HKDF-SHA256(salt = "determ-enote-v1",            # 15-byte DST, no trailing NUL
                        ikm  = z,
                        info = E33 ‖ recipient_pub33,         # 66 bytes
                        L    = 44)
    key   = okm[0:32] ; nonce = okm[32:44]                    # 12-byte nonce, KDF-derived
    ct‖tag = ChaCha20-Poly1305(key).encrypt(nonce, pt, aad = E33)   # 16-byte tag appended
    wire  = E33 ‖ ct ‖ tag                                    # len = len(pt) + 49

open(recipient_sk[32], wire):
    E     = decompress(wire[0:33])                            # bad point ⇒ -1 (not ours)
    Z     = recipient_sk · E ; z = Z.x
    recipient_pub33 = compressed(recipient_sk · G)            # SAME R33 bound into info
    okm, key, nonce  = HKDF-SHA256(... info = E33 ‖ recipient_pub33 ...)
    pt    = ChaCha20-Poly1305(key).decrypt(nonce, ct, tag, aad = E33)   # tag fail ⇒ -1
```

Constants: `DETERM_ENOTE_EPH_LEN = 33`, `DETERM_ENOTE_TAG_LEN = 16`, `DETERM_ENOTE_OVERHEAD = 49`; ciphertext length is exactly `len(pt) + 49`. The intended plaintext for the CT use-case is `value(8, BE) ‖ blinding(32) = 40` bytes (a memo may extend it), sealing to a `49 + 40 = 89`-byte wire note.

### 3.2 Properties (EN-1 .. EN-5)

- **EN-1 (delivery — a verifying tag *is* the scan signal).** `open(sk, wire) == 0` **iff** `wire` was sealed to `sk`'s pubkey and is untampered, in which case it yields the exact plaintext. A non-matching / tampered / malformed note returns `-1` and writes **nothing** to the output buffer — the constant-time AEAD compare is the "not mine" gate a wallet scans with. No separate recipient-tag or address-match step is needed.

- **EN-2 (no new hardness assumption).** Confidentiality reduces to ECIES/DHIES over P-256: an eavesdropper without `recipient_sk` faces the P-256 CDH/DDH problem to recover `z`, then HKDF (modeled as a random oracle / PRF) and the ChaCha20-Poly1305 IND-CCA2 AEAD. Every step is an already-relied-on assumption of the CT stack (`ConfidentialTxBalanceSoundness.md` rests on the same P-256 DL). No primitive is introduced.

- **EN-3 (binding — a ciphertext cannot be re-pointed).** The HKDF `info` commits to **both** `(E, R)` and the AEAD `aad` commits to `E`. So `(key, nonce, tag)` are bound to the specific ephemeral key and the specific recipient: an attacker cannot lift a valid note onto a different recipient key or splice a different ephemeral prefix without failing the tag. (`open` recomputes `R33` from `sk` locally, so the recipient identity in `info` is not attacker-chosen.)

- **EN-4 (determinism + dual-oracle testability).** `seal` is a **pure function of `(R, pt, eph_sk)`** — the caller supplies the ephemeral secret, so the ciphertext is byte-exactly reproducible and can be pinned against an independent oracle (the c99 house discipline). This is what makes the primitive gateable by a fixed KAT corpus rather than only by roundtrip.

- **EN-5 (nonce-reuse safety is a caller obligation).** `eph_sk` **MUST** be fresh (CSPRNG) per note. Reuse repeats `(key, nonce)` and breaks AEAD confidentiality. The nonce is **KDF-derived** from `z` (which depends on `E = eph_sk·G`) rather than a caller counter precisely so that a fresh `eph_sk` mechanically yields a fresh `(key, nonce)` — the safety obligation collapses to the single, standard "fresh ephemeral per encryption" ECIES requirement. The header states this contract explicitly.

## 5. Owner-gated follow-on — the consensus wiring (NOT in this increment)

Closing NC-8 *end-to-end* means carrying an enote on-chain per confidential output and giving wallets a way to find it. That is consensus surgery on `CONFIDENTIAL_TRANSFER` and the tx/state wire, so it is **deferred to the owner** with the sub-decisions below. Sketch of the full wiring:

- **Attach the ciphertext to the output.** For each `CONFIDENTIAL_TRANSFER` (`TxType = 14`) output note, publish `seal((recipient key), value‖blinding[‖memo], fresh eph_sk)` in an **additional payload region**. (`SHIELD`'s single note is **self-owned** — the depositor chose `r` — so SHIELD delivery is moot except when depositing to a *different* recipient's key; the primary consumer is `CONFIDENTIAL_TRANSFER` outputs.) The enote must remain **consensus-inert** per §2: at most block-hash-bound (like `pq_auth`, serialized-only-when-present), **never** a spend precondition or accept-rule input, so the no-double-spend machinery stays byte-identical.
- **A scan RPC.** A node endpoint returning the enote payloads over a height range (or for the current unspent-output set) so a wallet can pull-and-trial-decrypt, using EN-1 (a verifying tag = mine) as the filter.
- **Light-client scan.** The same, but the light client verifies the enote's inclusion against the committed `state_root` before trial-decrypting — only meaningful if enotes are a state leaf (sub-decision 2).

**Sub-decisions for the owner:**

1. **Which recipient key.** (a) A **dedicated P-256 note key** per account (a new "note pubkey", registered or address-derived) — keeps scan/spend authority cleanly separate from the audit layer; or (b) **derive from the A2 / view-key layer** (`viewkey.c`, the view-master / per-epoch view pubkey) — one key infrastructure, and it simultaneously closes the "auditor has an opaque key that opens nothing" gap of §1, at the cost of coupling recipient-decryption to the audit-disclosure key. Both are P-256 points the primitive already accepts. **→ Owner-decided (2026-07-18), profile-keyed: FIPS ⇒ (b) view-master derivation (one auditable key infrastructure, matching the FIPS posture's audit-coupling appetite); every other profile (MODERN) ⇒ (a) dedicated note key (scan/spend authority separate from the audit layer). The derivation library for BOTH branches is SHIPPED — see §5.4 (`determ_notekey_*`, profile-split subfolders); what remains owner-gated is the on-chain PUBLICATION of `note_pk`.**
2. **Ciphertext-in-tx-payload vs a new state leaf.** (a) **In the tx payload** (append to the `CONFIDENTIAL_TRANSFER` payload, `pq_auth`-style: block-hash-bound, serialized only when present, **not** in the state root — preserves state-root-invariance, but a light client cannot trustlessly prove an enote's presence and must trust a node's scan); or (b) a dedicated **`en:` state leaf** (state-root-committed → light-client-provable scan, but adds a state namespace and makes the enote part of the state-root surface — a larger consensus change). **→ Owner-decided, profile-keyed: FIPS ⇒ (a) tx payload (holds state-root-invariance); MODERN ⇒ (b) `en:` state leaf (light-client-provable scan).**
3. **Per-output vs per-tx.** (a) **Per-output** — one enote per output note, each sealed to that output's recipient — the general case that supports genuine multi-recipient transfers (matches the `m` outputs of a DCT1 bundle); or (b) **per-tx** — a single enote — simpler wire, but limits a transfer to a single recipient / self-transfer. **→ Owner-decided: (a) per-output, both profiles.**

**RESOLUTION (owner, 2026-07-18).** The sub-decisions above are **decided profile-dependently**: `{FIPS ⇒ 1b + 2a, MODERN ⇒ 1a + 2b, 3a both}`. Because the wiring now *branches on the crypto profile*, the profile can no longer live as a build-time posture label (`params.hpp`): the value that selects a chain's enote wiring must be **consensus-visible and pinned at genesis** so two operators cannot silently disagree on which branch a block was applied under. **Increment 1 — the genesis profile pin — is SHIPPED (§5.1);** the remaining branch-bodies (MODERN `en:` leaf / FIPS payload region; MODERN note-key vs FIPS view-master derivation; the per-output enote vector; the `CONFIDENTIAL_TRANSFER`/`SHIELD` accept-rule + apply; the scan RPC; the light-client scan) follow as later owner-gated increments.

### 5.2 SHIPPED — increment 2: the enote region on `CONFIDENTIAL_TRANSFER` (placement 2a/2b + per-output 3a)

The per-output ciphertext is now carried on-chain. A `CONFIDENTIAL_TRANSFER` (`TxType = 14`) payload is its DCT1 bundle **optionally followed by a per-output enote region**; the genesis-pinned profile (§5.1) selects how it is committed, exactly per the RESOLUTION: **MODERN ⇒ 2b `en:` state leaf** (light-client-provable) / **FIPS ⇒ 2a payload-only** (block-hash-bound), **3a per-output** on both.

- **Wire (region, appended to the bundle).** `count(1)` then `count ×` `{ output_index(1), enote_len(2 BE), enote_bytes[enote_len] }`; `output_index` strictly increasing and `< m`, `enote_len ∈ [49, 512]` (`CTX_ENOTE_MIN = DETERM_ENOTE_OVERHEAD`, `CTX_ENOTE_MAX`), and the region must consume the payload **exactly**. Absent region (`payload == bundle`) is byte-identical to a pre-NC-8 tx.
- **The split, without touching the frozen crypto.** The frozen `determ_ctx_bundle_header`/`_verify` demand an *exact* length, so the consensus callers can't hand them a payload with a trailing region. The **one shared mirror** `ctx_split_enotes` (`include/determ/chain/ctx_enote.hpp:64`) reads `n_in/m/n` from the fixed DCT1 prefix, computes `bundle_len = determ_ctx_bundle_len(n_in,m,n)` (the exported, pure length fn), and hands the verifiers only the `[0, bundle_len)` prefix — then frames the remainder. **Both** the validator accept-rule (`src/node/validator.cpp:1262`) and chain apply (`src/chain/chain.cpp:1095`) consume this identical function (the eligibility-floor "one mirror" discipline), so they cannot drift on where the bundle ends or what a well-formed region is.
- **Consensus-inert (design §2).** `ctx_split_enotes` validates **structure only** — count/index/length bounds and exact consumption. It never calls the enote primitive and never inspects ciphertext content: a malformed *frame* rejects the tx, a valid-but-garbage ciphertext is **accepted** and never decrypted in consensus. The enote never enters the balance/excess proof, the fee arithmetic, or the SP-11 input/output dedup set — every SP-* claim of `ShieldedPoolSoundness.md` is untouched.
- **Profile-keyed apply.** After the pool mutation, MODERN records each enote as a conditional `en:` state leaf (`src/chain/chain.cpp:524`) keyed by `hex(output commitment)`, value `SHA256(commitment_bytes ‖ enote_wire_bytes)` (binds the ciphertext to the note it delivers *and*, via the leaf key, to that commitment); FIPS records nothing (payload-only). A spent note's `en:` entry is dropped at **every** spend site (`CONFIDENTIAL_TRANSFER` inputs `:1132`, `UNSHIELD` `:1059`), so `en: ⊆ cn:` stays bounded. The set round-trips through **both** snapshot paths — JSON `serialize_state`/`restore_from_snapshot` (`:2190`/`:2393`) and the in-memory reorg-rollback `StateSnapshot` (lazy `__ensure_enote_commitments` capture `:859`, restore `:763`) — serialized only when non-empty.
- **Producer.** No change: the `CONFIDENTIAL_TRANSFER` producer path is a provisional no-op that carries the tx (with its region) verbatim.

### 5.3 SHIPPED — increment 3: the enote SCAN (`scan_enotes` RPC, read-only)

A wallet can now retrieve the ciphertexts increment 2 carries. `chain::scan_enotes(chain, from, to[, cap])` (`include/determ/chain/enote_scan.hpp`) walks blocks in the half-open, chain-clamped range `[from, to)`, finds `CONFIDENTIAL_TRANSFER` (`TxType = 14`) transactions, re-runs the **same** `ctx_split_enotes` mirror the validator/apply consume, and emits one `EnoteHit = {height, tx_hash, output_index, hex(output commitment), ciphertext bytes}` per delivered enote. The recipient trial-decrypts each ciphertext with `determ_enote_open` — a verifying AEAD tag (EN-1) is the "this note is mine" signal — and the `commitment` keys the recovered `(value, blinding)` to the exact pool note it can later spend.

- **Pure reader.** No chain/consensus/state mutation, no wire/tx-format change; all golden `state_root`s + snapshots are byte-identical with it compiled in. It surfaces only well-framed regions on txs consensus already accepted (a malformed region — which could never have entered a block — is skipped defensively).
- **Profile-agnostic.** The ciphertext rides the tx payload on **both** MODERN and FIPS chains, so the scan returns hits on either; the profile only decides whether an `en:` **state** leaf *also* exists (MODERN 2b → light-client-provable inclusion; §5.6, the remaining increment).
- **The RPC.** `Node::rpc_scan_enotes` (`src/node/node.cpp`) exposes it under method `scan_enotes` with params `{from_height, to_height, limit}` under the same `state_mutex_` shared-read lock as `rpc_state_root`/`rpc_state_proof`. The range is clamped to the chain and the result **work-bounded**: `scan_enotes` is handed `limit + 1` as `cap` and stops the walk as soon as that many hits are collected, so `limit` bounds materialization **and** the lock hold (not merely the response), with a `truncated` flag; a caller pages by advancing `from_height`.

**Gate** `test-scan-enotes` (9 assertions; FAST-suite member via `scan_enotes`, wrapper `tools/test_scan_enotes.sh`): every per-output enote surfaced with the correct height / commitment / ciphertext / tx hash; the half-open `[from,to)` range excludes the boundary block; empty + past-end ranges are safe (no OOB); a region-free `CONFIDENTIAL_TRANSFER` yields nothing; and a FIPS chain returns the same hits as MODERN. A 4-lens adversarial review (read-only purity, memory-bounds, correctness/completeness, DoS/lock) returned **zero** HIGH/MEDIUM (its one actionable LOW — the cap was originally applied post-scan — is fixed by the `limit + 1` work-bounded early-out above).

**Remaining owner-gated increments:** the recipient-key *derivation* (§5.4) and the on-chain **publication** of `note_pk` (§5.5, `REGISTER_NOTE_KEY` + `nk:` leaf) are now SHIPPED; what remains is the read side of publication (inc.5b — the `nk` `rpc_state_proof` branch + light `verify-notekey`) and the **light-client scan** (§5.6 — verify an `en:` leaf's inclusion against a committed `state_root` before trial-decrypting; meaningful only on MODERN, which has the `en:` leaf).

### 5.4 SHIPPED — increment 4: recipient note-key derivation (both profiles, profile-split)

The scan (§5.3) trial-decrypts an enote **with a note secret**; increment 4 is the deterministic derivation of that per-address note keypair — the key a sender seals to and the recipient (or, on FIPS, the auditor) opens with. It is decided profile-dependently per §5 sub-decision 1 (`{MODERN ⇒ 1a dedicated note key, FIPS ⇒ 1b view-master derivation}`), and — because "the differences are so much" — the two profiles live in **separate subfolders that reuse one shared base**:

- **Shared base** (`src/crypto/notekey/notekey.c`, `include/determ/crypto/notekey/notekey.h`): `determ_notekey_from_ikm(ikm[32], dst, chain_id, addr, index)` builds `msg = ikm‖len64_be(chain_id)‖chain_id‖len64_be(addr)‖addr‖index_u64_be`, derives `note_sk = determ_p256_hash_to_scalar(msg, dst)` (RFC 9497 shape, `L=48`, canonical in `[1,n)`), and returns `note_pk = compress(note_sk·G)`. The `msg` scratch (holds the secret IKM) and `sk` are `determ_secure_zero`'d on every path; fail-closed on NULL / empty `chain_id`|`addr` / field > 256 B / zero scalar with the outputs untouched. A thin composition over the §3.9b P-256 primitives — **no new hardness assumption** (see `CRYPTO-C99-SPEC.md` §3.25).
- **1a MODERN** (`src/crypto/notekey/modern/`): `determ_notekey_modern_derive` — DST `determ-notekey-modern-v1`, IKM = an INDEPENDENT 32-B note seed. Scan/spend authority stays cleanly separate from the audit layer.
- **1b FIPS** (`src/crypto/notekey/fips/`): `determ_notekey_fips_derive` — DST `determ-notekey-fips-v1`, IKM = the A2 `view_master_sk` (§3.24). ONE secret backs both audit and note delivery: an auditor disclosed `view_master_sk` re-derives every `note_sk` and reads all deliveries — closing the "opaque audit key opens nothing" gap of §1. The distinct DST domain-separates the profiles, so a MODERN and a FIPS key from the same inputs are independent.

**Gate** `test-notekey-modern-c99` / `test-notekey-fips-c99` (~15 assertions each; FAST-suite members via `notekey_modern_c99|notekey_fips_c99`, wrappers `tools/test_notekey_{modern,fips}_c99.sh`): determinism (= the FIPS auditor re-derive), `note_pk == compress(note_sk·G)`, distinctness over index/addr/chain_id/IKM, cross-profile domain separation, fail-closed edges, an end-to-end **seal→open** (an enote sealed to `note_pk` opens with the derived `note_sk`; a *different* derived `note_sk` does NOT open it — the scan "not mine" filter), and the **dual-oracle** KAT corpus byte-for-byte. Python-first: `tools/verify_notekey.py` (dependency-free RFC 9380 XMD + a from-scratch P-256 ladder) froze `tools/vectors/notekey_{modern,fips}.json` (6 vectors each) before the C-side test existed. A 4-lens adversarial review (correctness/oracle-integrity, memory/fail-closed, crypto-soundness/domain-separation, CT/determinism) returned **zero** HIGH/MEDIUM.

- **Library-only, additive, state-root-invariant.** No chain/consensus/validator/producer path calls it; goldens byte-identical.
- **Non-claims / still owner-gated.** This derives the keypair; it does **not** decide how `note_pk` is PUBLISHED on-chain so a third-party sender can find it — a new tx vs a REGISTER-extension vs address-derived (plus the MODERN note-seed registration mechanism). That publication is a genuine consensus sub-decision, deferred to the owner as the next increment. Self-delivery (depositor seals to a key it already holds) needs no publication and works today.

### 5.5 SHIPPED — increment 5a: on-chain note-key publication (`REGISTER_NOTE_KEY` + `nk:` leaf)

The §5.4 derivation produces a recipient's `note_pk`, but a **third-party** sender cannot compute it (the derivation feeds a *secret* IKM through `hash_to_scalar`, so the public point is not address-derivable). The recipient must PUBLISH it. Owner-decided **Option (a)** — a dedicated tx + state leaf — over extending `REGISTER` (option b) or address-derivation (option c), because it is the only path that **reaches the recipient population NC-8 must serve: anonymous/bearer payees**. `REGISTER` is validator-only (rejected from anon accounts); address-derivation is cryptographically impossible without a new hardness assumption + key escrow.

`REGISTER_NOTE_KEY` (`TxType = 17`) is a near-verbatim clone of the shipped `ROTATE_AUDIT_KEY`/`ak:` template: fee-only (`amount == 0`, `to` empty), payload = the 33-byte SEC1-compressed `note_pk` to SET or empty to CLEAR/revoke; the account's own Ed25519 signature (`signing_bytes` includes the payload) is proof-of-possession — no separate PoP. **Anon-reach**: `REGISTER_NOTE_KEY` is added to the validator anon whitelist (alongside the audit txs), so a plain bearer payee publishes with its own key. Apply writes a conditional **`nk:` + addr** state leaf, value `= SHA256(note_pk)`, emitted **only while set** → a chain that never publishes a note key is **byte-identical** on both profiles (one-mirror conditional-emit discipline of `ak:`/`en:`). **Consensus-inert**: the point is never decompressed or validated — a garbage 33-byte value is accepted and simply never opens an enote (exactly like the ciphertext region); it is never in balance/dedup/fee math, so every SP-* claim and the A1 supply identity hold. Profile-agnostic (MODERN + FIPS emit identically — neither profile's `note_pk` is derivable from any other on-chain value, §5.4).

Sites (all clones of the `audit_keys_`/`ak:` precedent): `block.hpp` (`REGISTER_NOTE_KEY = 17`, `NOTE_KEY_PAYLOAD_SIZE = 33`); `validator.cpp` (anon whitelist + accept-rule shape gate); `producer.cpp` (grouped with the fee-only audit txs); `chain.hpp` (`note_keys_` map + `note_key()`/`note_key_count()` + `StateSnapshot::note_keys`); `chain.cpp` (apply set/clear, `nk:` leaf emit, `__ensure_note_keys` lazy-snapshot, restore-move, genesis-clear, JSON serialize/restore). Full depth-1-reorg rollback + snapshot round-trip via the four shipped `audit_keys_` sites.

**Gate** `test-register-note-key` (29 assertions; FAST via `register_note_key`): the set/rotate/clear lifecycle on the `nk:` leaf (a **real inc.4-derived** `note_pk`), fee-only accounting (A1 intact), apply-level fail-closed skips (wrong length / non-zero amount / non-empty `to`), state-proof observability, JSON-snapshot round-trip (cleared + live), and **the decisive property** — an anonymous bearer account's `REGISTER_NOTE_KEY` PASSES the validator while its `REGISTER` is REJECTED. Falsify-on-mutant: dropping `REGISTER_NOTE_KEY` from the anon whitelist turns the anon-pass assertion RED (the entry is load-bearing). 4-lens adversarial review (byte-neutrality, accept/apply parity, anon-reach/permanence, reorg/snapshot) **zero HIGH/MEDIUM**; two accepted NITs — a re-issue/clear that lands mid-seal makes an already-broadcast enote unopenable (a liveness footgun the design accepts; the note + value stay safe in the pool, no safety/double-spend impact), and the determ-light client-side *builder* has no `REGISTER_NOTE_KEY` branch yet (inc.5b scope).

- **PERMANENCE (no-migrations):** once any account publishes, three things freeze forever — `TxType = 17`, the `nk:` leaf preimage `SHA256(note_pk[33])`, and the 33-byte SEC1 format. Rotation/revocation within v1 is re-issue (SET overwrites, empty clears); a v2 would be a new TxType + `nk2:` + a `-v2` DST, never in-place. Byte-neutrality means the freeze binds only chains that actually use it.
- **Read side (SHIPPED, inc.5b):** a sender now obtains a published `note_pk` **trustlessly** — a new `nk` `rpc_state_proof` branch (leaf key `"nk:"+addr`), a cleartext `note_pk` field on the `account` RPC, and a light-client **`verify-notekey`** command (a simplified clone of `verify-registrant`) that binds the proof to the account on BOTH axes (`key_bytes == "nk:"+addr` **and** `value_hash == SHA256(note_pk)` recomputed from the untrusted cleartext) and Merkle-verifies against a **committee-attested** `state_root` (via `committee_bound_state_root`, never the bare header field). Gate `test-light-verify-notekey` (offline arg/exit-code contract [FAST] + a live NOT-INCLUDED + wrong-genesis leg). The INCLUDED-specific `SHA256(note_pk)` recompute is unit-gated by `test-register-note-key` (the `nk:` leaf is state-proof-provable there) and the committee-anchor + Merkle machinery is byte-identical to the live-gated `verify-registrant` INCLUDED.
- **Builder (SHIPPED):** the determ-light **`register-note-key`** command builds a canonical, submittable `REGISTER_NOTE_KEY` (`TxType 17`, fee-only, account-Ed25519-signed) — the client half that publishes what §5.4 derives; anon/bearer payees included. It reuses the shipped `sign_audit_tx` signing-bytes helper (byte-parity with `rotate-audit-key`), so it emits exactly what a validator accepts. The `determ verify-audit-tx` accept-check was extended to `REGISTER_NOTE_KEY` (the same anon-account shape gate the block validator runs), giving a **provable client→consensus loop**: gate `test-light-register-note-key` (7/7) builds SET + CLEAR forms and asserts `verify-audit-tx` VERIFIES them, a tampered fee → INVALID, a 32-byte note-pk → build error, and a TRANSFER → rejected by the type gate. This closes the inc.5a review NIT and makes the full build → `submit-tx` → `verify-notekey` INCLUDED loop composable. Self-delivery already needs no publication.

### 5.6 SHIPPED — increment 6: the light-client enote-inclusion scan (the final NC-8 increment)

A wallet pulls candidate `(commitment, ciphertext)` pairs from a full node's `scan_enotes` RPC (§5.3) — an **untrusted** source — and, before spending CPU trial-decrypting them (or trusting the node did not fabricate/omit deliveries), **proves** each pair is the genuine on-chain delivery. The MODERN `en:` state leaf commits `value = SHA256(commitment_bytes ‖ enote_wire_bytes)`, keyed by the output commitment (§5.2/2b). The light-client `verify-enote-inclusion` command binds the caller-provided pair on **both** axes — `key_bytes == "en:"+hex(commitment)` **and** `value_hash == SHA256(commitment ‖ enote)` recomputed locally — and Merkle-verifies against a **committee-attested** `state_root` (`committee_bound_state_root`, never the bare header field). It is READ-ONLY / additive; goldens byte-identical.

- **The property.** A lying node that returns a **fabricated/tampered** ciphertext is caught (`value_hash` mismatch → UNVERIFIABLE); one that invents a **commitment** with no delivery is caught (`not_found` → NOT-INCLUDED). A pass (INCLUDED) means the ciphertext is exactly the one committed on-chain — trial-decrypt safe. Unlike `verify-notekey` there is no daemon cleartext to fetch: the caller supplies the pair (from the scan), and the leaf's value-hash-bind is what makes the read trustless.
- **MODERN-only** (by design). On a FIPS chain the ciphertext is payload-only (2a) with no `en:` leaf, so `verify-enote-inclusion` returns NOT-INCLUDED and a light client there necessarily trusts the serving node's scan — the profile trade-off recorded since §5.2.
- **Sites.** `src/node/node.cpp` `rpc_state_proof` (`en` namespace branch, leaf key `"en:"+lowercase-hex(commitment)`); `light/main.cpp` `cmd_verify_enote_inclusion` (a clone of `verify-notekey`; decodes + length-checks the 33-byte commitment, canonicalizes to lowercase, recomputes the exact `build_state_leaves` `en:` preimage). Gate `test-light-verify-enote-inclusion` (offline arg/exit-code contract [FAST] — including a non-33-byte-commitment reject — + a live NOT-INCLUDED for an unseen commitment + wrong-genesis fail-closed; 12/0 with the live leg). The INCLUDED-specific `SHA256(commitment ‖ enote)` recompute is unit-gated by `test-ctx-enote` (the `en:` leaf value is that hash there); the committee-anchor + Merkle machinery is byte-identical to the live-gated `verify-registrant`/`verify-notekey` INCLUDED.

**This completes the NC-8 delivery pipeline end-to-end: derive (§5.4) → publish (§5.5/inc.5a) → discover the recipient key (§5.5/inc.5b) → seal (§3.23 enote) → carry on-chain (§5.2) → scan (§5.3) → trustlessly confirm inclusion (§5.6).** The only remaining NC-8-adjacent item is a determ-light `register-note-key`/`confidential-transfer` *builder* to drive live INCLUDED legs end-to-end (client tooling, not consensus).
