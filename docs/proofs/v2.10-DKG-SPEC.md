# v2.10 threshold randomness — DKG ceremony specification

**Status:** specification only. No code. Resolves the 4 primary + 3 cascade design decisions for distributed key generation (DKG) underlying v2.10's t-of-K threshold-signature randomness mechanism. Implementation should not begin until this document's design choices are reviewed and committed.

- `F2-SPEC.md` — analogous structure for v2.7 (per-field reconciliation)

## 1. Scope

This spec covers ONLY the DKG ceremony and threshold-signature share lifecycle for v2.10. It does NOT cover:
- The threshold-signature scheme itself (BLS12-381 + FROST or equivalent — described at the protocol level here, with library choice in §2.5)
- Block-format changes beyond the wire-format additions in §3.4
- Wallet / RPC changes (not required — DKG is committee-internal)
- VRF (v2.9, deferred)
- Composition with v2.25 T-OPAQUE OPRF (shares curve choice; see §2.5 cascade)

The DKG ceremony produces:
- A **threshold public key** `PK_epoch` committed on-chain at epoch boundary
- K **secret shares** `s_1, …, s_K`, one per committee member, no member knowing any other's share or the master secret
- A **verifiable transcript** that any honest party can replay to confirm `PK_epoch` was generated correctly

These artifacts are consumed by the v2.10 threshold-signature mechanism: each member produces a partial signature with their share; any t partials combine into the canonical `R = combine(partial_1, …, partial_t)` that replaces today's `delay_output`.

## 2. Design decisions (4 primary + 3 cascade)

### Q1: Genesis-time vs runtime DKG

**Decision: epoch-boundary runtime DKG, not genesis-time.**

A fresh DKG round runs at the start of each epoch (`epoch_blocks` configurable, default 100). New validators that REGISTER + STAKE during the prior epoch acquire shares in this round. Departing validators (DEREGISTER + UNSTAKE) are dropped from the share set.

**Rationale.** Permissionless deployments have routine validator churn — REGISTER/STAKE/UNSTAKE are normal operations, not exceptional events. A DKG model that doesn't accommodate routine churn is wrong for Determ's design center. Epoch boundaries are already a natural protocol sync point (committee rotation, A5 PARAM_CHANGE), so adding DKG at the boundary reuses an existing coordination primitive.

**Alternatives rejected:**
- *Genesis-time only (static):* No native validator-rotation support. Would require operator-driven re-DKG ceremonies (governance actions), which reintroduce operator privilege and break the mutual-distrust property v2.10 is meant to deliver.
- *Per-block DKG:* Latency-prohibitive (DKG is a multi-round protocol). Wastes throughput on overhead.

### Q2: Trusted vs trustless dealer

**Decision: fully trustless (no dealer).**

DKG runs as a K-party protocol where no single party knows the master secret. Pedersen-style verifiable secret sharing (VSS) underpins the share generation; each member contributes their own randomness; the master secret is the sum of K contributions, computable only by all K acting together.

**Rationale.** A trusted dealer reintroduces a single point of trust — exactly what Determ's K-of-K mutual-distrust posture is designed to eliminate. Trustless DKG matches the consensus model: K members must cooperate (or t of K, post-DKG, for signing); no member is special.

**Implication.** The K founding validators at chain genesis are themselves the K participants of the first DKG round, run during chain bootstrap. No external operator/dealer is involved.

### Q3: Validator-rotation support

**Decision: rotation handled natively via per-epoch DKG re-run.**

The DKG round at each epoch boundary includes:
- All validators eligible for the committee at epoch start (registered, staked above `MIN_STAKE`, not under suspension/slashing).
- The threshold `t` is recomputed as `ceil(2K/3)` where K is the new committee size.
- The threshold public key `PK_epoch` rotates per epoch (unless proactive refresh is used — see Q4).

**Rationale.** No special "join" or "leave" protocol — the next epoch's DKG round is the join mechanism. Departing validators simply don't participate in the next round. This keeps the protocol surface minimal.

**Edge case: committee size change.** If the committee size changes between epochs (one validator joined, one left), the new DKG produces a freshly-sized t-of-K share set. The previous epoch's shares are useless for signing in the new epoch (different K, different threshold, different public key).

### Q4: Share refresh — proactive vs full re-key

**Decision: proactive secret sharing (PSS) per epoch, stable public key across epochs unless committee membership changes.**

When the committee membership is unchanged between epochs, run proactive secret sharing (Herzberg-Jarecki-Krawczyk-Yung style) rather than a fresh DKG. PSS refreshes the shares (new randomness) while preserving the public key. Members who held shares in epoch N receive new shares for epoch N+1 that combine the same master secret; old shares are deleted.

When committee membership changes, run a fresh DKG (the public key rotates).

**Rationale.** Forward secrecy at zero cost in the common case (membership stable). A compromised share in epoch N becomes useless after the PSS refresh at epoch N+1. The public key only rotates when the committee actually changes, minimizing on-chain key-rotation events and downstream re-verification costs.

**Implication.** External consumers of `PK_epoch` (e.g., for verifying randomness commitments) need to track key rotations. Block headers commit the current `PK_epoch`, so verification is one-lookup-per-epoch, not per-block.

### Q5 (cascade): Threshold-signature scheme

**Decision: BLS12-381.**

**Rationale.**
- Most mature DKG protocols (Pedersen, GG18, FROST) target BLS12-381 or compatible curves.
- Production-grade reference implementations exist: `blst` (Supranational), `herumi/bls` (Cybozu), `arkworks/bls12_381` (Arkworks).
- Signature aggregation property: t partial signatures over the same message combine into a single canonical signature. Required for v2.10's "all t-subsets produce the same R" property.
- Pairing-friendly: enables signature verification with just the public key + message + signature, no per-signer state.
- Shared with v2.25 T-OPAQUE OPRF (the same curve is the natural choice for OPRF — see cascade implication below).

**Trade-off acknowledged.** Adds BLS12-381 to Determ's cryptographic stack (currently SHA-256 + Ed25519). The "two primitives only" minimalism is a design value being relaxed here for genuine necessity — threshold signatures structurally require pairing-friendly curves; there's no SHA-256/Ed25519 path that delivers the same property.

**Library selection: `blst`** (recommended over alternatives because it's the most-audited, fastest, and used in production by Ethereum 2.0 / Lighthouse / Prysm). Vendored as a git submodule with pinned version. Constant-time implementations only.

### Q6 (cascade): DKG protocol

**Decision: FROST-BLS (Flexible Round-Optimized Schnorr Threshold signatures, adapted for BLS).**

**Rationale.**
- Most actively maintained DKG protocol for production threshold signing (2020 onwards).
- Tolerates up to ⌊(K-1)/3⌋ malicious participants during the DKG ceremony itself (consistent with BFT bound).
- Two-round protocol: round 1 commits, round 2 distributes shares. Compatible with epoch-boundary timing constraint.
- Compatible with proactive refresh (PSS step is a single-round extension).
- Well-understood security proofs (the FROST family has been formally analyzed in multiple papers).
- Reference implementations available in `rust-frost` and ports.

**Alternatives considered:**
- *Pedersen DKG (1991):* Older; less robust to misbehaving participants during ceremony. Pedersen requires an additional complaint round for full security.
- *GG18 / GG20:* Designed for ECDSA, not BLS. Wrong fit.
- *DKG-from-scratch:* Rolling our own is not justified given FROST's maturity.

### Q7 (cascade): Epoch DKG timing

**Decision: First R blocks of each epoch, where `R = 5` blocks at tactical profile, scaling per profile.**

The DKG ceremony occupies the first R blocks of each epoch. During these blocks, the committee runs the DKG protocol; normal block production continues using the previous epoch's keys. At block R+1, the new `PK_epoch` and shares are active; the previous epoch's keys are retired.

**Per-profile R values:**

| Profile | Block time | R blocks | DKG wall-clock budget |
|---|---|---|---|
| Tactical | 40 ms | 5 | 200 ms |
| Web | 200 ms | 3 | 600 ms |
| Regional | 300 ms | 3 | 900 ms |
| Global | 600 ms | 3 | 1.8 s |
| Cluster | 50 ms | 5 | 250 ms |
| *_test | 5-10 ms | 3 | 15-30 ms |

**Rationale.** Decouples DKG round from block-production round (no consensus interference). R is sized to absorb realistic network jitter at each profile while staying well under the epoch duration (epoch_blocks = 100 → DKG consumes 3-5% of epoch). Tactical and Cluster profiles use R=5 because their per-block budget is tightest; slower profiles can afford R=3 because each block has more wall-clock headroom.

## 3. Protocol description

### 3.1 Epoch-boundary DKG round (3-phase, R blocks)

At block boundary `epoch_start = E * epoch_blocks`:

**Block E+0 (Commit phase).** Each committee member `i` ∈ {1..K}:
1. Generates a random polynomial `f_i(x)` of degree `t-1` (where `t = ceil(2K/3)`).
2. Computes commitments `C_i = [g^{a_{i,0}}, g^{a_{i,1}}, …, g^{a_{i,t-1}}]` (each coefficient × group generator).
3. Broadcasts `DKGCommitMsg { epoch, member_id, commitments: C_i }` via gossip.

**Block E+1 (Share-distribution phase).** Each member `i`:
1. Computes shares `s_{i,j} = f_i(j)` for every other member `j`.
2. Sends `DKGShareMsg { epoch, from: i, to: j, share: s_{i,j} }` to member `j`, encrypted with `j`'s long-term encryption key (Ed25519-X25519 wrap).
3. Each member `j` decrypts their received shares `s_{1,j}, …, s_{K,j}`, verifies each against the corresponding commitment `C_i` (verifiable secret sharing check: `g^{s_{i,j}} ?= prod_k C_{i,k}^{j^k}`).

**Block E+2 (Complaint phase, optional).** If any member `j` finds an inconsistent share (VSS check fails), they broadcast `DKGComplaintMsg { epoch, accuser: j, accused: i, share: s_{i,j} }` revealing the bad share for public verification. The accused either provides a valid share (counter-evidence) or is excluded from the DKG (their commitment contribution is dropped from the master secret).

**Block E+3 (Finalize phase).** Each member `j` computes their final share `s_j = sum_i s_{i,j}` (sum over all non-excluded contributors). The threshold public key is `PK_epoch = prod_i C_{i,0}` (product of non-excluded constant terms). Block E+3 commits `PK_epoch` into the chain via a new on-chain field `block.epoch_public_key`.

**Block E+4 (Activation, tactical/cluster only).** Slack block — DKG finalized at E+3, normal threshold signing resumes at E+5 onward. Other profiles activate immediately at E+4.

### 3.2 Proactive refresh (membership-unchanged case)

If `committee(E) == committee(E-1)`:

1. Each member `i` generates a fresh polynomial `h_i(x)` of degree `t-1` with `h_i(0) = 0` (refresh polynomial, zero at zero).
2. Computes refresh shares `r_{i,j} = h_i(j)` and distributes encrypted as in §3.1.
3. Each member `j` updates their share: `s_j_new = s_j_old + sum_i r_{i,j}`. Old `s_j_old` is securely deleted.
4. The master secret is unchanged (since each `h_i` adds zero at x=0); the public key `PK_epoch` is unchanged.
5. Compromised shares from epoch N are useless for signing in epoch N+1 onward.

PSS occupies only 1 block (not the full R) because no commitment phase is needed (commitments to zero are trivial).

### 3.3 Failure handling

**DKG round timeout.** If R blocks elapse and any phase fails to complete (e.g., a member crashed and didn't broadcast their commit), fall back to the previous epoch's keys. Mark the epoch with `dkg_status: PREVIOUS_EPOCH_KEYS_REUSED` on-chain. Operator alerted via metric `dkg_round_failures`. Next epoch's DKG re-attempts.

**Member exclusion.** Members excluded during complaint phase don't receive a share for the new epoch — they can't sign in that epoch. They re-participate in the next epoch's DKG if they remain in the committee.

**Below-threshold committee.** If exclusions reduce the share-holding committee below `t`, the DKG fails (insufficient signers for the threshold). Same fallback as timeout.

### 3.4 Wire-format extensions

Three new message types (all carried over existing gossip layer):

```
DKGCommitMsg {
    epoch:        u64
    member_id:    u8                  // index in committee
    commitments:  [G1; t]             // t group elements (BLS12-381 G1)
    sig:          Signature           // Ed25519 sig by member's long-term key
}

DKGShareMsg {
    epoch:        u64
    from_member:  u8
    to_member:    u8
    share_ct:     bytes (AEAD ciphertext over share_scalar, AAD = "DKG" || epoch || from || to)
    sig:          Signature
}

DKGComplaintMsg {
    epoch:        u64
    accuser:      u8
    accused:      u8
    revealed_share: scalar            // share decrypted; let everyone verify the VSS check fails
    sig:          Signature
}
```

New on-chain block fields (extending `Block`):

```
epoch_public_key:    G1                // BLS12-381 G1 element; committed every epoch_blocks blocks
dkg_status:          u8                // 0 = active, 1 = failed (previous keys reused), 2 = PSS refresh
dkg_excluded:        Vec<u8>           // member indices excluded during complaint phase (if any)
```

Existing `ContribMsg.creator_dh_secrets` becomes `ContribMsg.creator_partial_sigs: Vec<Signature>` per the v2.10 design. Wire-format breaking change; flag-day required.

## 4. Implementation work units

### 4.1 BLS12-381 library vendoring (~3-5 days)

- Vendor `blst` as git submodule, pinned version.
- CMake integration, constant-time build flags.
- C++ wrapper exposing: `Scalar`, `G1Element`, `G2Element`, `Signature`, with key operations (scalar mul, group add, pairing, hash-to-curve).
- Unit tests covering all wrapper ops.

### 4.2 FROST-BLS DKG protocol (~1-1.5 weeks)

- Polynomial sampling, commitment generation.
- Share generation and AEAD-wrapped distribution.
- VSS verification on share receipt.
- Complaint generation and verification.
- Share aggregation and PK derivation.
- Proactive refresh variant (PSS).

### 4.3 Epoch-boundary orchestration (~3-5 days)

- Detect epoch boundary in block-apply path.
- Initiate DKG round (member-side state machine).
- Handle R-block timing per profile.
- Commit `block.epoch_public_key` at finalize block.
- Fallback to previous epoch's keys on failure.

### 4.4 Threshold-signature integration (~3-5 days)

- Replace `creator_dh_secrets` with `creator_partial_sigs` in `ContribMsg`.
- Each member signs `beacon_seed || height` with their epoch share.
- Aggregate any t partials into canonical `R`.
- Replace `delay_output = SHA-256(delay_seed ‖ sort(revealed_secrets))` with `delay_output = R`.

### 4.5 Failure-mode handling (~3-5 days)

- Member-exclusion path during complaint phase.
- Timeout fallback to previous keys.
- Below-threshold detection and fallback.
- Operational metrics: `dkg_round_failures`, `dkg_complaints_filed`, `dkg_excluded_members`, `dkg_phase_latency_ms`.

### 4.6 Regression tests (~3-5 days)

- DKG ceremony completes in R blocks with all-honest committee.
- DKG handles 1 silent member (fallback path).
- Complaint phase: 1 malicious member excluded; remaining committee produces valid PK.
- PSS refresh preserves PK across epoch.
- Threshold signing produces same R from different t-subsets.
- Selective-abort attempt fails (withholding partial doesn't bias R).

### 4.7 Migration / wire-version bump (~2-3 days)

- Bump `ContribMsg` wire version.
- Add genesis fields: `threshold_scheme: "BLS12-381"`, `dkg_protocol: "FROST"`, `dkg_phase_r_blocks: u32` (defaults per profile in `params.hpp`).
- Flag-day height in genesis config.
- Pre-flag-day: legacy commit-reveal; post-flag-day: threshold signatures via DKG-distributed shares.

### 4.8 Documentation refresh (~2-3 days)

- `V2-DESIGN.md` v2.10 update with Option C choice (this pass).
- `WHITEPAPER-v1.x.md` §randomness section update.
- `PROTOCOL.md` wire-format additions.
- `SelectiveAbort.md` final-state update with new threat-model claim.
- This spec doc reviewed and committed.

## 6. Risks and rollback plan

**Risk: BLS12-381 library audit surface.** Adds a new cryptographic primitive family to Determ's stack (currently SHA-256 + Ed25519 only). `blst` is the most-audited BLS12-381 implementation available, but the audit perimeter expands measurably.

*Mitigation.* Vendor `blst` only; do not implement BLS primitives ourselves. Pin to a specific release. Track upstream security advisories. Same supply-chain discipline as libsodium.

**Risk: DKG round failure cascades.** If multiple epochs in succession fail DKG (e.g., persistent network partition), the chain reuses old keys indefinitely. A compromised old key would compromise randomness for the duration.

*Mitigation.* Operational alerting on `dkg_round_failures >= 2`. Below-threshold detection forces a chain pause via governance (v2 mechanism, not yet shipped — for v1.x deployments, manual operator intervention). The fallback-to-old-keys path is a degraded-availability mode, not a normal operating mode.

**Risk: Proactive refresh implementation bug compromises old shares without distributing new ones.** A bug in PSS could leave members without working shares for the new epoch.

*Mitigation.* PSS test coverage: verify old shares are deleted ONLY after new shares are confirmed valid. Two-phase commit on share-update at member level. Rollback path: re-fetch shares from peers via gossip if local PSS state is corrupt.

**Risk: Flag-day migration leaves chain in inconsistent state.** Pre-flag-day blocks use commit-reveal; post-flag-day use threshold signatures. The transition block must handle both correctly.

*Mitigation.* Dry-run validator-side for 100 blocks pre-flag-day (log mismatches without rejecting). DKG ceremony runs at the flag-day epoch boundary; threshold signing activates at the subsequent block. Pre-flag-day commit-reveal continues to validate for old blocks during sync.

**Risk: Spec disagreement during review.** Q5 cascade (BLS12-381 + `blst`) and Q6 cascade (FROST-BLS) are the highest-impact choices. If review prefers a different scheme (e.g., GG20-Schnorr, or building on an existing chain's audited implementation), revise this doc and re-cost.

*Mitigation.* Pre-implementation review per §8 below.

**Rollback plan.** If v2.10 + DKG ships with a bug discovered after the flag-day:
1. Governance pause (v2 mechanism) freezes new block production.
2. Revert flag-day height; new genesis config without v2.10.
3. Re-flag-day to pre-v2.10 behavior; commit-reveal resumes.
4. Cost: chain re-genesis (operator-coordinated). State from pre-flag-day preserved via snapshot.
5. v2.10 re-attempts after bug fix; new flag-day height in subsequent genesis.

## 7. What this enables downstream

v2.10 with the DKG infrastructure described above is a precondition for:

- **Theme 9 DSSO (v2.25 T-OPAQUE).** T-OPRF uses the same BLS12-381 curve. Each committee member's OPRF share derives from the same per-epoch DKG share (different label, same underlying secret-sharing infrastructure). Without v2.10's DKG, T-OPAQUE has no trust-minimized share-distribution path.

- **v2.22 confidential transactions (cascade decision confirmed in `v2.22-PRIVACY-SPEC.md`).** v2.22 confirmed BLS12-381 + Bulletproofs over the same curve. The BLS12-381 library shipped here is shared. Saves ~1 week of curve25519 vendoring effort that v2.22 would otherwise require, and reduces total audit surface vs. introducing curve25519 + BLS12-381 separately.

- **Future v2.x: distributed VRF (v2.9, currently deferred).** A VRF built on BLS12-381 + DKG shares is a natural follow-on; the DKG infrastructure is the long pole.

The DKG infrastructure described here is a **shared foundation** for the threshold-cryptography layer of v2 + Theme 9, not a v2.10-only investment.
