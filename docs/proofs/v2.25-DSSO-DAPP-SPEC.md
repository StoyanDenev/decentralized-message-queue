# v2.25-DSSO-DAPP-SPEC.md — "Sign-In With Determ": the mutual-distrust identity provider (threshold-OPAQUE)

**Status:** AUTHORITATIVE for the DSSO DApp (V1.1-PLAN Bundle A). Single DSSO mechanism doc under the B3 consolidation program. Realizes the design of *Identity provider in an environment of mutual distrust* (academia.edu/80188125) with two owner-chosen improvements — **OPAQUE in place of the paper's SRP**, and a **t-of-n, unordered** threshold in place of the paper's sequential all-node chain. **No FROST, no threshold signature, no block-co-signed assertion** — the paper issues its relying-party token by hash challenge-response over handshake-co-generated keys, so no signature scheme is involved at all.

**Authority:** Stoyan Denev — owner decision: drop FROST, keep the paper's design. This spec supersedes the 2026-07-15 "DLT-B" text, which had re-engineered the assertion into a block-anchored / credential-signature form the paper does not use. Recorded: `DECISION-LOG.md` 2026-07-20.

**Chain surface:** none. DApp-layer protocol over already-shipped primitives; the v1.1 consensus path and the no-migrations lock are untouched.

---

## 0. The paper's design, and the two improvements

The paper builds a federated identity provider from **K mutually-distrusting servers**: a user authenticates to the *set*, and no single server (below the trust threshold) learns the user's password or can impersonate the IdP. The paper's concrete construction is (a) an **SRP** augmented-PAKE whose verifier material is split across the servers, (b) evaluated as a **sequential hash chain** (server A -> B -> ... -> G, every server required, in order), (c) issuing the relying-party token by a **dual-hash challenge-response**: the PAKE handshake co-generates a shared key, and the RP accepts iff `H2 == H2'` where `H2 = H(tenant_key, H1')`.

Determ maps the K servers onto its **K-of-K committee** and adopts the paper's framework verbatim except for two owner-chosen improvements:

1. **OPAQUE in place of SRP.** OPAQUE (CFRG, RFC 9807-track) is the modern UC-secure aPAKE: OPRF blinding hides the password from every server, and the envelope adds precomputation-resistance against a stolen verifier — properties SRP lacks. The paper's framework is PAKE-agnostic, so the substitution is drop-in.
2. **t-of-n and unordered in place of the sequential all-node chain.** The paper needs every server, in a fixed order — a latency chain and a single point of failure. Determ thresholdizes the OPRF (Shamir, **t-of-n**): the user blinds the password, **broadcasts** it, collects **any t** blinded evaluations **in any order**, and recombines by Lagrange interpolation. This is exactly the "not all nodes needed, no predefined order" property.

**FROST is not part of this design and never was.** The paper issues its token by hash challenge-response over co-generated keys, not by a server signature — so there is no threshold-signature step to implement, and nothing here composes with FROST.

## 1. Roles

- **Servers O_1..O_n** — the K-of-K committee members act as the IdP servers. (A deployment MAY register an opt-in server set instead; recorded in DAPP_REGISTER.)
- **User U** — holds a password `pw` and an OPAQUE credential.
- **Relying party (RP)** — an off-chain service; registers via v2.18 DAPP_REGISTER, establishing shared `tenant_key` material with the IdP. Challenges/responses ride v2.19 DAPP_CALL.
- **Threshold t** — how many servers must respond. Default **t = n-1** (tolerates one down/dishonest server; below-t collusion cannot recover the password). Per-deployment; recorded at registration.

## 2. Primitives — all already shipped

| Primitive | Role | Ref |
|---|---|---|
| P-256 RFC 9497 OPRF/VOPRF | the threshold OPRF substrate (prime-order group + hash-to-curve + DLEQ) | CRYPTO-C99 §3.9b |
| Shamir over the P-256 scalar field | user-dealt t-of-n shares of the OPRF key | shipped (`test_wallet_shamir*`) |
| OPAQUE envelope (AEAD + HKDF) | seals the user credential; unsealed by the OPRF output | ChaCha20-Poly1305 / AES-GCM + HKDF (§3.1/§3.4/§3.5) |
| SHA-256 / HKDF | the dual-hash challenge-response (`H1`/`H2`) + key derivation | §3.1 |
| Ed25519 | user credential key; committee identity keys | §3.2 |
| v2.18 DAPP_REGISTER / v2.19 DAPP_CALL | RP registration; challenge/response transport | shipped |
| Light-client committee resolution | RP resolves the on-chain committee (IdP server) set | shipped |

Zero new primitives, zero new hardness assumptions. The threshold OPRF runs on the shipped P-256 RFC 9497 stack because that is the shipped stack with the required group operations (point add, scalar inverse, hash-to-curve); X25519 lacks them. This is an implementation substrate, not a change to the paper's design.

## 3. Registration

1. U picks `pw` and generates an OPAQUE credential (Ed25519 keypair `cred`).
2. U computes the OPRF key `k` and **deals it as t-of-n Shamir shares** `k_i` to the servers (user-as-dealer — no DKG, no server-side ceremony; an inconsistent dealing can only self-DoS U's own account). Each server stores its `k_i`; the share commitments `PK_i = k_i·G` are published so responses can be checked (§4).
3. `y = OPRF_k(pw)` computed locally; `envelope = AEAD_{HKDF(y)}(cred_sk)`. The envelope is stored (client-side and/or on-chain).
4. RP registration (once per RP): v2.18 DAPP_REGISTER establishes the RP's identity and shared `tenant_key` material with the IdP.

## 4. Login (t-of-n threshold OPRF — unordered)

1. U blinds: `r` random, `B = r·H2C(pw)`.
2. U **broadcasts** `B` to the servers — no ordering, no server-to-server communication, no per-server state beyond the stored share.
3. Each responding server returns `Z_i = k_i·B` with a DLEQ proof `pi_i` (shipped P-256 VOPRF) that `Z_i` is consistent with its published `PK_i`.
4. On **any t** valid responses, in **any order**: U discards any response whose `pi_i` fails, Lagrange-combines `Z = sum(lambda_i·Z_i) = k·B`, unblinds `N = r^{-1}·Z = k·H2C(pw)`, finalizes `y`.
5. `y` unseals the envelope -> `cred`. The OPAQUE handshake **co-generates a shared session key** between U and the IdP (the paper's "co-generation of a common key"). No server saw `pw`; below-t servers cannot combine.

## 5. Assertion to the RP (the paper's dual-hash challenge-response — no signature)

Following the paper: the handshake yields `sso_key` (user<->IdP) and the RP holds `tenant_key` from its registration. For a login to RP with a challenge/nonce:

- U computes `H1' = H(sso_key, challenge)` and `H2 = H(tenant_key, H1')`, binding the SIWE-class claim `(iss=chain_id, sub=U, aud=rp, iat, exp, nonce)`.
- The RP computes `H2'` from the same material and **accepts iff `H2 == H2'`**.

No committee signature, no FROST, no block co-sign. The token is a keyed hash that the user (via the OPRF-unsealed credential) and the RP (via its registered `tenant_key`) can each derive. Exact `H1`/`H2` wire format follows the paper's construction, pinned at implementation.

*Optional, off by default:* a deployment wanting a public login audit trail MAY additionally emit a DAPP_CALL record of `(sub, aud, iat)` — an audit anchor only, not part of the authentication, and not the assertion mechanism.

## 6. Security claims -> backing

| # | Claim | Backing |
|---|---|---|
| C1 | <= t-1 colluding servers learn nothing about `pw` beyond online guessing | Shamir (information-theoretic) + threshold 2HashDH OPRF (JKK 2014; TOPPSS threshold form JKKX 2017), Gap-OMDH/ROM; RFC 9497 P256-SHA256, KAT-gated (§3.9b) |
| C2 | No single server (below t) can impersonate the IdP | The paper's mutual-distrust property: the shared key needs the user's OPAQUE credential, recoverable only via >= t combined OPRF responses |
| C3 | Offline dictionary attack needs >= t shares | Shamir + OMDH |
| C4 | A byzantine server cannot corrupt the login undetected | Per-response DLEQ vs the published `PK_i` (RFC 9497 VOPRF soundness) |
| C5 | OPAQUE upgrades over the paper's SRP | precomputation-resistance + verifier-secrecy (OPAQUE, JKX 2018) |
| C6 | RP assertion soundness | keyed-hash challenge-response (PRF security of H) over PAKE-authenticated keys — the paper's construction |
| C7 | Liveness: any t honest reachable servers + the transport complete a login | protocol structure; unordered by construction |

Online guessing is metered as in the paper (per-account attempt limits); with t-of-n the cap must be aggregate (cooperative counters, or fee-metered DAPP_CALL) since an attacker can rotate subsets — deployment choice.

## 7. Explicit non-goals

- **No FROST / threshold signature / DKG** — the paper co-signs nothing; the OPRF key is user-dealt.
- **No block-anchored assertion** — the token is the paper's hash challenge-response (an earlier draft's block-anchoring is dropped).
- **No post-quantum claim; no assertion-traffic anonymity; no phishing defense** — out of scope.

## 8. Recovery / rotation

- Password change / share refresh / server churn: U re-deals fresh shares + envelope (user-as-dealer; no DKG), authorized under U's chain identity key.
- Chain-identity key loss/compromise: v2.26 ROTATE_KEY (unchanged precondition for production).

## 9. Green gates before Bundle A merge

- **G1** t-of-n identity: Lagrange over every t-subset == direct OPRF evaluation.
- **G2** per-response DLEQ reject paths (tampered response rejected).
- **G3** envelope AEAD/HKDF KATs.
- **G4** end-to-end: register -> login (t-of-n; one server down; one byzantine) -> dual-hash assertion -> RP accepts.
- **G5** constant-time review of secret scalar paths; **G6** zeroization (`k`, coefficients, `k_i`, `r`, `y`, keys).

## 10. Relation to earlier drafts

The 2026-06-07 "DLT-A" and 2026-07-15 "DLT-B" drafts re-engineered the assertion (block-anchored DAPP_CALL + committee/user signatures) — a Determ-specific mechanism the paper does not use. This spec drops that: the assertion is the paper's dual-hash challenge-response, and the "DLT-A/-B" labels are retired. What carries over from those drafts (and is correct) is the t-of-n OPRF on the shipped P-256 stack. FROST removal is unchanged (`FROST_DEVIATION_NOTICE.md`) — this design needs no threshold signature.
