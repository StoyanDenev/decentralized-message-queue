## 0. What DSSO is

The exogenous framework is the mutual-distrust identity-provider architecture ("identity provider in an environment of mutual distrust", PAKE-as-black-box): K cooperating-but-untrusting operators jointly act as an IdP so that no single operator can impersonate the IdP or learn user secrets. Determ instantiates the operator group with DApp instances run by the K-of-K committee members and fills the aPAKE role with a **T-OPAQUE-shaped composition**: a threshold OPRF evaluates the password jointly, the OPRF output unseals an envelope holding the user's credential key (OPAQUE's envelope construction, Jarecki–Krawczyk–Xu 2018), and — instead of OPAQUE's server-side AKE — the **chain itself** anchors the resulting assertion. "Sign-In With Determ."

The original paper's sequential nested-hash evaluation (H1→H2→…) is not used: blinded OPRF evaluation is broadcast-parallel, stateless per operator, and order-free by construction (§4).

## 1. Composition (DLT-B)

**DLT-B = DLT-A with the T-OPRF leg re-based; assertion leg kept.** Two legs:

1. **T-OPRF leg (password → credential).** t-of-n verifiable threshold OPRF over **NIST P-256, RFC 9497 P256-SHA256, VOPRF mode (0x01) throughout**. Per-user OPRF key, user-dealt Shamir shares (§3), per-response DLEQ verification, client-side Lagrange recombination (§4). The OPRF output unseals an AEAD envelope holding the user's DSSO credential key `cred_sk` (Ed25519).
2. **Assertion leg (credential → assertion), unchanged from DLT-A.** The user signs the RP challenge with `cred_sk`; a DSSO instance emits the assertion as a DAPP_CALL tx; the RP light-client-verifies block inclusion and the **user's own signature** (§5). The K-of-K block signature attests **inclusion/ordering only**; assertion truth is anchored to the user's credential signature, so **no DSSO operator is trusted for assertion correctness** and emission is permissionless among instances.

**Primitive inventory (all shipped; zero new primitives, zero new assumptions):**

| Primitive | Where used | Spec ref |
|---|---|---|
| P-256 group ops (`base_mul`, `point_mul`, `point_add`, SEC1 compress/decompress) | shares, Lagrange, DLEQ | CRYPTO-C99 §3.8c/§3.9b |
| RFC 9380 hash-to-curve P256_XMD:SHA-256_SSWU_RO_ | password → group element | §3.9b |
| mod-n scalar ops (`scalar_mul_mod_n`, `_inv_mod_n`, add/negate) | blinding, unblinding, Shamir, Lagrange | §3.9b |
| RFC 9497 OPRF/VOPRF (`blind`/`evaluate`/`finalize`; `voprf_prove`/`_verify`) | T-OPRF core; per-share DLEQ | §3.9b |
| Ed25519 | `cred_sk` signatures; operator DApp keys | §3.2 |
| X25519 + AEAD (XChaCha20-Poly1305 MODERN / AES-256-GCM FIPS via P-256 ECDH) | share-envelope transport at registration only | §3.3–§3.5 |
| SHA-256 / HKDF | envelope-key derivation | §3.1 |
| Argon2id (MODERN) / PBKDF2 (FIPS) | optional password prehash | §3.6/§3.8b |
| v2.18 DAPP_REGISTER, v2.19 DAPP_CALL | RP + DSSO registration; messaging; assertions | shipped |
| Light-client suite (headers → block sigs → Merkle state/tx proofs) | RP verification | shipped |
| v2.26 ROTATE_KEY | chain-identity recovery precondition | `v2.26-ROTATION-SPEC.md` |

The hardness assumption set is exactly §3.9b's: ECDLP + Gap-One-More-DH in the ROM. Same claim form as §3.23 LSAG/CLSAG: built entirely on the shipped P-256 public API, no new hardness assumption.

**Profile note.** The DSSO OPRF is **profile-agnostic P-256** — the same posture as the §3.22 shielded pool and an extension of DECISION-LOG 2026-07-07 D3's reuse-P-256 logic. MODERN vs FIPS differ only in the share-envelope AEAD/KX and the optional password prehash. This supersedes the 2026-07-07 D1 per-profile split (MODERN X25519 T-OPRF / FIPS P-256).

## 2. Roles and parameters

- **Operators O_1…O_n** — DSSO DApp instances. Default n = K (the chain committee members); a DSSO DApp MAY register with an opt-in operator set instead (deployment choice recorded in its DAPP_REGISTER record).
- **Threshold t** — the secrecy/liveness dial, fixed per registration record:

| t | Login liveness | Password secrecy (offline dictionary requires) |
|---|---|---|
| t = n | all n must respond | collusion of all n |
| **t = n−1 (RECOMMENDED default)** | tolerates 1 unreachable/malicious operator | collusion of n−1 |
| t < n−1 | tolerates n−t faults | collusion of t (explicit per-deployment decision) |

Any t colluding operators reconstruct that user's OPRF key and can dictionary-grind that user's password offline (§6 C2). t is therefore a *stated* trade-off, not a free win; deployments choosing t < n−1 must record the rationale. Changing t or the operator set requires a new registration record (§8) — records are immutable once block-anchored.

## 3. Registration

User U, password `pw`; fresh credential keypair (`cred_sk`, `cred_pk`) — Ed25519, generated client-side, distinct from U's chain identity key (key-separation: wallet key never leaves the wallet).

1. `k ←$ Z_n^*` (per-user OPRF key). Shamir polynomial `f(x) = k + a_1·x + … + a_{t−1}·x^{t−1} mod n`; shares `k_i = f(i)`, `i = 1…n`.
2. Share pubkeys `PK_i = k_i·G`; group key `PK = k·G`.
3. Deliver `k_i` to O_i sealed under the profile KX + AEAD (transport: DAPP_CALL or the operator's registered endpoint). O_i verifies `k_i·G == PK_i` and rejects otherwise (self-protection). Public consistency — any t of {PK_i} Lagrange-combine in the exponent to PK — is checkable by anyone, once. **No VSS complaint round is needed:** the user deals shares of *their own* key, so an inconsistent dealing can only self-DoS that one account; no honest party is harmed. User-as-dealer is what removes the DKG — and with it every `FROST_DEVIATION_NOTICE.md` exposure.
4. `y = F_k(pw)` computed locally via the direct-Evaluate path (the §3.9b blind/evaluate/finalize == direct-Evaluate identity). `K_env = HKDF-SHA-256(y, info = "determ/dsso/env/v1")`; `envelope = AEAD_{K_env}(cred_sk)`.
5. Registration record emitted via DAPP_CALL and block-anchored: `{U, suite_id, n, t, PK, PK_1…PK_n, envelope, cred_pk, prehash_params}`. Block inclusion makes the `cred_pk ↔ U` binding immutable and light-client-verifiable — this binding is what the assertion leg verifies against.
6. Zeroize `k`, the polynomial coefficients, and every `k_i` (`determ_secure_zero`).

*Optional password prehash:* `pw' = Argon2id(pw, salt=U)` (MODERN) / PBKDF2 (FIPS) before hash-to-curve; parameters pinned in `prehash_params`. Raises the per-guess cost of the ≥t-collusion offline grind (§6 C2); does not change any other claim.

## 4. Login

1. Fetch U's registration record (light-client-verified against a signed state root / inclusion proof).
2. Blind: `r ←$ Z_n^*`, `B = r·H2C(pw)` (`determ_p256_oprf_blind`, VOPRF-mode context).
3. **Broadcast** `(U, B)` to the operators. No ordering, no inter-operator communication, no per-operator state beyond the stored share.
4. Each responding O_i returns `Z_i = k_i·B` plus DLEQ proof `π_i` — the shipped single-element `determ_p256_voprf_prove` with the **share** key in the key slot, statement `log_G(PK_i) = log_B(Z_i)`.
5. U verifies each `π_i` against the record's `PK_i` (shipped `_verify`; invalid → discard and record operator fault). On any t valid responses S, in any arrival order: Lagrange coefficients `λ_i^S` at 0 (mod-n ops), `Z = Σ_{i∈S} λ_i·Z_i` (`point_mul` + `point_add`), unblind `N = r^{-1}·Z` (`_inv_mod_n` + `point_mul`), `y = Finalize(pw, N)` (RFC 9497).
   *Correctness:* `Z = (Σ_S λ_i k_i)·B = k·B = k·r·H2C(pw)` ⇒ `N = k·H2C(pw)` — byte-identical to the registration-time direct evaluation, so `Finalize` matches (gate G1, §10).
6. `K_env = HKDF(y)`; AEAD-open the envelope → `cred_sk`. Tag failure = wrong password or corrupted record — explicit rejection, no partial information.
7. Session: sign the RP challenge with `cred_sk` (§5). `cred_sk` and `y` are held in memory only and zeroized after the session key handoff.

Both conditions the design targets hold by construction: **not all nodes are needed** (any t of n), and **there is no predefined evaluation order** (responses combine commutatively; Lagrange coefficients depend only on *which* t responded, not in what order).

## 5. Assertion (leg 2)

Assertion tx payload (DAPP_CALL from the DSSO DApp):

```
{ iss = dsso_dapp_id, sub = U, aud = rp_id,
  cred_pk, challenge_hash, sig_U,
  iat, exp, nonce, height, state_root }
```

`sig_U` = Ed25519 signature under `cred_sk` over `(chain_id ‖ dsso_dapp_id ‖ rp_id ‖ challenge_hash ‖ nonce ‖ exp)`. Any instance may emit — emission is permissionless among instances because verification does not trust the emitter.

RP verification (all via the shipped light-client suite, no trusted node):
(a) block containing the assertion tx carries valid K-of-K committee signatures;
(b) Merkle inclusion proof of the tx in that block;
(c) `sig_U` verifies under the `cred_pk` bound in U's **block-anchored registration record** (fetched + inclusion-proved the same way);
(d) freshness: `exp` not passed, `nonce` is the RP's own challenge, single-use.

**The precise attestation claim.** The K-of-K block signature attests **existence, ordering, and timestamp** of the assertion tx — *not* the truth of the authentication. Validators do not (and must not be assumed to) evaluate DSSO semantics. Assertion truth reduces to Ed25519 EUF-CMA under `cred_pk` (§6 C4). Consequence: a fully malicious operator set can **censor** (refuse to emit/serve — liveness, §6 C6) but cannot **mint** an assertion for U. This corrects the prior "the block signature IS the threshold attestation" phrasing, which conflated inclusion-attestation with authentication-attestation.

## 6. Security claims → backing (the proof map)

Exactly these claims are made, and no others.

| # | Claim | Reduces to | Backing |
|---|---|---|---|
| C1 | ≤ t−1 colluding operators, observing all transcripts, learn nothing about `pw` beyond what online guessing (C7) yields | Blinded requests perfectly hiding (uniform `r`, multiplicative blinding); ≤ t−1 Shamir shares independent of `k`; 2HashDH T-OPRF security | Shamir (information-theoretic); 2HashDH: Jarecki–Kiayias–Krawczyk 2014, threshold form: Jarecki–Kiayias–Krawczyk–Xu (TOPPSS) 2017 — Gap-OMDH, ROM. Instantiation is RFC 9497 P256-SHA256, KAT-gated (§3.9b) |
| C2 | Offline dictionary attack requires ≥ t shares | `k` reconstructible only from ≥ t shares; with `k`, each guess costs H2C + `point_mul` (+ prehash if pinned) | Shamir + OMDH; prehash per §3.6/§3.8b |
| C3 | A byzantine operator cannot corrupt the login result undetected | Per-response DLEQ `log_G(PK_i) = log_B(Z_i)` vs the block-anchored `PK_i` | RFC 9497 VOPRF proof soundness (ROM); shipped reject-path gates (§3.9b) |
| C4 | Assertion unforgeability: no coalition of operators (any size) can mint an assertion for U | Ed25519 EUF-CMA under `cred_pk` ∧ block-anchored `cred_pk ↔ U` binding ∧ Merkle/state-proof soundness | §3.2; the already-locked chain FV surface (block-sig + inclusion soundness docs) — unchanged by this spec |
| C5 | No operator ever sees `pw`, `y`, `K_env`, or `cred_sk` | Operator I/O is `(B, Z_i, π_i)` only | Corollary of C1 |
| C6 | Liveness: any t honest reachable operators + a live chain complete a login | Protocol structure; invalid responses filtered by C3 | — |
| C7 | Online guessing is metered | Each guess requires t fresh evaluations; throttling per §7 | Stated policy, not a cryptographic reduction — see §7 |

**Deliberately NOT claimed (dropped with the AKE):** mutual authentication with an operator, channel binding, and OPAQUE's session-key indistinguishability — the chain-anchored assertion replaces the server-side AKE, and no published proof covers a threshold AKE; claiming it would violate the B3 nothing-aspirational rule. The exogenous framework's guarantees are consumed only through the black-box property this composition does provide: password-authenticated issuance where no single (indeed no < t) operator can evaluate, and no coalition can forge (C4).

## 7. Online guessing / rate limiting

Each online guess costs one T-OPRF evaluation accepted by ≥ t operators. Per-operator throttling alone is subset-rotatable (an attacker can spread guesses across the C(n,t) subsets); the aggregate cap must therefore be enforced either (i) cooperatively — operators share guess counters per U via DAPP_CALL telemetry — or (ii) structurally — the deployment pins eval-requests-via-DAPP_CALL in the registration record, making every guess a fee-bearing on-chain tx (chain-metered, audit-trailed, higher latency). Deployment choice; the reference deployment uses (i) with conservative per-U counters. C7 is an operational claim, not a reduction — stated as such.

## 8. Recovery and rotation

- **Password loss / share refresh / operator churn:** emit a new registration record (fresh `k`, shares, envelope; same `cred_pk` if `cred_sk` is still held, otherwise fresh credential keypair), authorized by a signature under U's **chain identity key**; latest block-anchored record at height wins. Re-dealing is user-as-dealer again — no PSS, no DKG.
- **Chain-identity key loss/compromise:** v2.26 ROTATE_KEY (unchanged precondition for production DSSO).
- **`cred_sk` compromise:** new registration record with fresh credential keypair (above); RPs verify against the latest record, so the old `cred_pk` dies with it.

## 9. Explicit non-goals

- **No FROST, no threshold signing of any kind** — nothing in DLT-B signs under a shared key; `FROST_DEVIATION_NOTICE.md` §3 is not engaged, and this spec does not weaken it.
- **No threshold AKE** — no published proof exists; replaced by chain anchoring (§5, §6).
- **No post-quantum claim** — discrete-log stack, same posture and same migration path as the chain (§7.5.6/§7.5.7 discriminators).
- **No assertion-traffic anonymity** — `V2-DAPP-DESIGN.md` §10 linkability caveats apply unchanged.
- **No UI/phishing defenses** — out of scope for the cryptographic layer.

## 10. Proof obligations before Bundle A merge (green gates)

- **G1** — t-of-n identity: Lagrange recombination over every |S| = t subset equals the direct-Evaluate output (new test over the shipped API; RFC 9497 A.3 vectors reusable per share).
- **G2** — per-share DLEQ reject paths: tampered `c`/`s`/`Z_i`/`PK_i`/wrong-mode all rejected (extends the existing §3.9b gates).
- **G3** — envelope KATs per profile AEAD + HKDF `info`-string domain-separation review (`determ/dsso/env/v1`).
- **G4** — end-to-end reference RP flow incl. light-client verification of the registration record and assertion (Bundle A work unit).
- **G5** — constant-time review of secret-adjacent scalar paths (registration `k`/coefficients, login `r`/`r^{-1}`, operator `k_i`) against `ConstantTimeInventory.md`.
- **G6** — zeroization audit: `k`, coefficients, `k_i`, `r`, `y`, `K_env`, `cred_sk` (`determ_secure_zero`).

## 11. Relation to the prior direction (DLT-A) — why re-based

DLT-A (DECISION-LOG 2026-06-07, reaffirmed 2026-07-07 D1) specified the T-OPRF leg as "X25519 threshold DH, additive n-of-n shares, aggregate via group multiplication, zero new primitives." Re-based because: (i) **not implementable as specified** — the shipped X25519 module (`include/determ/crypto/x25519/x25519.h`) exposes exactly two clamped, x-only scalar-mult functions; additive-share aggregation needs point addition, unblinding needs unclamped multiplication by `r^{-1} mod ℓ`, and "blinded password as an X25519 point" needs a hash-to-curve — none exist there, so the "zero new primitives" claim held only for a stack that could not run the protocol; (ii) **no proof** — an OPRF over a cofactor-8, x-only group has no published security argument, and RFC 9497 restricts OPRF suites to prime-order group abstractions for exactly this reason; (iii) **no verifiability, no liveness slack** — additive n-of-n lets a single byzantine operator silently corrupt the output and a single offline operator halt all logins. DLT-B keeps DLT-A's assertion leg and its "already-shipped primitives only" property — on the stack where that property is actually true — and sharpens the assertion-attestation claim (§5).
