# CrossBinaryCanonicalFormat — why the three Determ binaries agree on every canonical form

**Status: structural-equivalence soundness analysis.** Sibling capstone:
[LightClientCompositionMap.md](LightClientCompositionMap.md).

Determ ships **three** executables — `determ`, `determ-wallet`, `determ-light` — that must
agree, *byte-for-byte*, on a small set of **canonical forms**:

- the **Transaction** `signing_bytes` and its `tx_hash = SHA-256(signing_bytes)`
  (`src/chain/block.cpp:17-34` — `Transaction::signing_bytes` + `Transaction::compute_hash`),
- the **Block** `signing_bytes` / `compute_hash` / the committee **block digest**
  (`src/chain/block.cpp:235-364`; the daemon digest the committee signs lives in
  `src/node/producer.cpp::compute_block_digest`),
- the **genesis hash** (`src/chain/genesis.cpp:458-461` — `compute_genesis_hash`),
- the **anon-address derivation** (`include/determ/types.hpp:153-155` — `make_anon_address`).

Cross-binary interop *depends* on this agreement. A tx signed by `determ-light sign-tx` must
verify and apply on a `determ` daemon; a Merkle proof generated by `determ-wallet merkle-proof`
must verify under `determ-light verify-state-proof`; a genesis hash a light-client pins must
equal the one the daemon computes. If any binary computed a *different* canonical form, the
light-client's verification soundness (`LightClientThreatModel.md`) would be vacuous: it would
"verify" against a form no honest daemon ever produced.

This document is the **structural-equivalence proof**: it accounts for *why* agreement holds —
and pinpoints, with code citations, exactly where agreement is **shared by construction** (one
`src/` definition linked into multiple binaries) versus **re-implemented and kept in sync by
discipline + tests** (the divergence-risk surfaces). It is a new subject: no existing proof
states the build-time reuse structure as its own theorem.

**Relationship to the rest of the proof corpus.** Every per-format soundness proof — the
Merkle family (`MerkleTreeSoundness.md`), the digest binding (`PartnerSubsetDigestBindingSoundness.md`,
`TimestampReconciliationSoundness.md`), the wire codec (`BinaryCodecRoundTripSoundness.md`,
`WireFormatBackwardCompat.md`), the genesis/state-root anchoring (`StateRootAnchorSoundness.md`) —
proves that *one* implementation of *one* format is **correct/sound**. None of them proves that
the *three binaries* compute *the same bytes*. That gap is this document. CBF is **orthogonal**
to soundness: it establishes **agreement**, not **correctness** (CBF-4).

## 1. Theorem statements

- **CBF-1 (single-implementation structural equivalence — the shared-by-linking regime).**
  For every canonical form whose definition lives in a `src/` module that is linked into more
  than one binary, there is **exactly one** machine-code implementation per binary, compiled
  from **the same source file**. Two binaries that both link `src/chain/block.cpp`,
  `src/chain/genesis.cpp`, `src/crypto/merkle.cpp`, `src/crypto/keys.cpp`, `src/crypto/sha256.cpp`
  therefore compute identical bytes **by construction**, not by coincidence. A divergence could
  only arise from a *build* that fails to relink the shared module (a stale object file, a
  partial rebuild) — never from the source, because there is no second source to disagree with.

- **CBF-2 (the re-implemented-mirror regime — the divergence-risk surfaces).**
  Two canonical computations are **NOT** shared by linking and are instead *re-implemented* in
  each binary that needs them, so agreement rests on byte-for-byte mirroring kept honest by the
  parity tests (CBF-3), not by CBF-1's "one source" guarantee:
  1. the **Transaction `signing_bytes`** is canonical in `src/chain/block.cpp:17-29` (linked into
     `determ`), but re-implemented in `light/sign_tx.cpp:37-62` (`compute_signing_bytes`) and
     again inline in `wallet/main.cpp` (`tx-sign-verify` at `:8224-8244`, `cold-sign` at
     `:9543-9562`) — `determ-wallet` links **neither** `src/chain/block.cpp` nor the canonical
     routine (its sources are `wallet/*.cpp` only; CMakeLists.txt:232-233);
  2. the **committee block digest** is canonical in `src/node/producer.cpp::compute_block_digest`,
     but re-implemented in `light/verify.cpp:57-92` (`light_compute_block_digest`) because the
     light-client does **not** link `src/node/producer.cpp` (it would drag in the whole daemon).
     `light_compute_block_digest` is the single highest-stakes divergence-risk surface: it is
     the verification anchor of the entire light-client proof family (`LightClientCompositionMap.md`
     T-L2). Its non-trivial conditional appendices — `partner_subset_hash` (FB55/S-030-D2) and the
     reconciled-timestamp dimension (FB56/S-030-D2) — must mirror the producer's trailing
     conditional binds exactly (cross-ref `PartnerSubsetDigestBindingSoundness.md`,
     `TimestampReconciliationSoundness.md`).

- **CBF-3 (the runtime parity tests are the empirical relink guard).**
  The cross-binary parity tests in `tools/` are the **empirical** check that a given build
  actually relinked the shared modules (CBF-1) **and** that the re-implemented mirrors have not
  drifted (CBF-2). They make *another binary* the reference rather than a hand-rolled algorithm
  copy, so correctness is judged by **agreement between two independent processes**:
  `tools/test_merkle_cross_binary_fuzz.sh` (wallet generates → light verifies, 24 random leaf
  sets), `tools/test_light_sign_tx.sh` (light signs → wallet `validate-tx` accepts),
  `tools/test_wallet_tx_hash_cross_command_fuzz.sh`, the `tools/test_*_determinism.sh` suite, and
  the byte-layout lock `tools/test_tx_signing_bytes.sh`. A build that linked a stale object, or a
  mirror that drifted by one byte, turns these RED.

- **CBF-4 (boundary: structural, not cryptographic).**
  CBF is a **build-time / structural** guarantee. It says the binaries **agree** on the canonical
  form; it does **NOT** say the form is itself **sound**. Whether `signing_bytes` is
  collision-resistant, whether the digest binds every consensus-critical field, whether the
  Merkle root is second-preimage-resistant — those are the per-format proofs' job
  (`MerkleTreeSoundness.md`, `BinaryCodecRoundTripSoundness.md`, the digest-binding proofs,
  `Preliminaries.md` §2). CBF and the per-format proofs **compose**: CBF says "all three speak
  the same language"; the per-format proofs say "that language is sound."

## 2. Assumptions and threat model

### 2.1 The standing assumption: a correct build

CBF's entire force rests on **one** assumption:

> **(B0) Honest build.** All three binaries under test were compiled from **the same source
> tree**, with a build that relinked every shared `src/` module into every binary that lists it.

This is not a cryptographic assumption; it is a **provenance** assumption. CBF is the statement
"*given* the binaries came from one tree and the linker did its job, they agree." It deliberately
does **not** defend against an adversary who ships a *tampered binary* (that is `A_local` /
`A_genesis` in `LightClientThreatModel.md` §2.2 — explicitly out of scope for the light-client
soundness family, and equally out of scope here). The value CBF adds *over* B0 is precise:

- For the **shared-by-linking** forms (CBF-1), B0 plus "one source file" *is* the whole proof —
  there is no second implementation that could disagree, so agreement is automatic. The only
  residual failure mode is a **broken build** (stale relink), which CBF-3's tests catch.
- For the **re-implemented mirrors** (CBF-2), B0 is **necessary but not sufficient**: two honest
  sources can still disagree if a maintainer edits one mirror and not the other. Here the guarantee
  degrades from "by construction" to "by discipline + CBF-3 tests," and the doc says so honestly.

### 2.2 What CBF does NOT cover

1. **Format soundness** — see CBF-4. CBF says the binaries agree; it never claims the agreed-upon
   format resists collision/preimage/forgery. Those are `MerkleTreeSoundness.md`,
   `BinaryCodecRoundTripSoundness.md`, `Preliminaries.md` §2, and the digest-binding proofs.
2. **A tampered or mismatched binary** — defeating B0 (an attacker who replaces `determ-light.exe`
   with a doctored build) is `A_local`, out of scope (it is out of scope for the entire
   light-client family; `LightClientThreatModel.md` §2.2 F-local).
3. **Toolchain-level miscompilation** — a compiler bug that miscompiles the *one* shared source
   into divergent code on two targets is outside CBF (it is a compiler-correctness assumption,
   adjacent to `Preliminaries.md`'s implicit "the C++ compiler is correct").
4. **Non-canonical / display surfaces** — JSON key ordering, pretty-print whitespace, human-readable
   `type_name` mnemonics (`light/sign_tx.cpp:111`) are **not** canonical forms; only the bytes that
   feed a hash or a signature are. CBF scopes exactly the four canonical forms in the intro.
5. **The wire/transport codec** — the binary-envelope framing/round-trip is
   `BinaryCodecRoundTripSoundness.md` (A3); CBF is about the *content* hashed inside, not the
   network frame around it.

## 3. Ground truth — the CMake reuse lists (the structural backbone)

The three `add_executable` declarations in `CMakeLists.txt` are the load-bearing structural fact.
Cited line ranges are exact (read at authoring time).

### 3.1 `determ` (the daemon) — `CMakeLists.txt:74, 136`

```cmake
file(GLOB_RECURSE SOURCES "src/*.cpp")     # CMakeLists.txt:74
...
add_executable(determ ${SOURCES})           # CMakeLists.txt:136
```

`determ` links **every** `src/*.cpp`, so it holds the **canonical** definition of all four forms:
`Transaction::signing_bytes`/`compute_hash` (`src/chain/block.cpp`), `Block` digest paths
(`src/chain/block.cpp` + `src/node/producer.cpp`), `compute_genesis_hash` (`src/chain/genesis.cpp`),
and `make_anon_address` (header `include/determ/types.hpp`, included everywhere). `determ` is the
**reference oracle**: the canonical form *is* whatever `determ` computes.

### 3.2 `determ-light` (the light client) — `CMakeLists.txt:262-282`

The light-client target is an **explicit, non-glob** source list. The decisive lines:

```cmake
add_executable(determ-light
  light/main.cpp
  ...
  light/sign_tx.cpp            # CMakeLists.txt:268  (RE-IMPLEMENTS signing_bytes)
  light/verify.cpp             # (RE-IMPLEMENTS the committee block digest)
  ...
  # Reused from src/ (explicit list — no glob):       CMakeLists.txt:276
  src/crypto/merkle.cpp        # CMakeLists.txt:277   (SHARED by linking)
  src/crypto/keys.cpp          # CMakeLists.txt:278   (SHARED by linking)
  src/crypto/sha256.cpp        # CMakeLists.txt:279   (SHARED by linking)
  src/chain/block.cpp          # CMakeLists.txt:280   (SHARED by linking — Block + Transaction)
  src/chain/genesis.cpp        # CMakeLists.txt:281   (SHARED by linking — genesis hash)
)
```

So `determ-light` **shares by linking** (CBF-1): the Merkle primitive, the Ed25519 key/sign/verify
helpers, the SHA-256 builder, the `Block` struct + its `signing_bytes`/`compute_hash`, and
`compute_genesis_hash`. It **re-implements** (CBF-2): the Transaction `signing_bytes`
(`light/sign_tx.cpp`, because it never signs through `Transaction` — it builds bytes directly from
CLI args) and the committee **block digest** (`light/verify.cpp`, because linking
`src/node/producer.cpp` would pull in the whole daemon). The explicit list is *deliberate*: the
CMake comment at `:258-261` states the binary "Explicitly NO ... node/producer.cpp ... reuses the
standalone crypto + chain primitives from src/ via an EXPLICIT file list (not glob) so the linker
can't accidentally pull in daemon code."

### 3.3 `determ-wallet` (offline key ops) — `CMakeLists.txt:232-233`

```cmake
file(GLOB_RECURSE WALLET_SOURCES "wallet/*.cpp")   # CMakeLists.txt:232
add_executable(determ-wallet ${WALLET_SOURCES})     # CMakeLists.txt:233
```

`determ-wallet` links **only** `wallet/*.cpp`. It links **none** of `src/chain/*` or
`src/node/*`. Consequently the wallet's Transaction `signing_bytes` and `tx_hash` are
**re-implemented inline** in `wallet/main.cpp` (CBF-2) — there is *no* shared linkage to the
canonical `src/chain/block.cpp` routine. (The wallet *does* share the C99 crypto `.c` files and
the envelope/shamir `.cpp` with `determ` via the explicit `list(APPEND SOURCES ...)` at
`CMakeLists.txt:82-134`, but that is the *primitive* layer — SHA-256/AES/Argon2id — not the
canonical-form layer.)

### 3.4 The reuse matrix

| Canonical form | `determ` | `determ-wallet` | `determ-light` | Regime |
|---|---|---|---|---|
| `Transaction::signing_bytes` / `tx_hash` | `src/chain/block.cpp` (canonical) | re-impl `wallet/main.cpp` | re-impl `light/sign_tx.cpp` | **CBF-2 mirror** (3 impls) |
| `Block::signing_bytes` / `compute_hash` | `src/chain/block.cpp` | — (wallet has its own block-tx-root helper) | **links** `src/chain/block.cpp` | **CBF-1 shared** (determ↔light) |
| committee block **digest** | `src/node/producer.cpp` (canonical) | — | re-impl `light/verify.cpp` | **CBF-2 mirror** (2 impls) |
| `compute_genesis_hash` | `src/chain/genesis.cpp` | — | **links** `src/chain/genesis.cpp` | **CBF-1 shared** (determ↔light) |
| Merkle root / proof / verify | `src/crypto/merkle.cpp` | **links** `src/crypto/merkle.cpp` | **links** `src/crypto/merkle.cpp` | **CBF-1 shared** (all three) |
| `make_anon_address` | header `types.hpp` | header (inline) | header (inline) | **CBF-1 shared** (header, one definition) |
| Ed25519 sign/verify (`crypto::sign`) | `src/crypto/keys.cpp` | libsodium/OpenSSL path | **links** `src/crypto/keys.cpp` | **CBF-1 shared** (determ↔light) |

Reading the matrix: the *structure* gives CBF-1 strength to the Block, genesis, Merkle, key, and
anon-address forms across determ↔light, and to Merkle across **all three** (the wallet shares
`src/crypto/merkle.cpp` even though it does not share `src/chain/block.cpp`). The Transaction
`signing_bytes` and the committee digest are the two CBF-2 mirror surfaces that the parity tests
must guard.

## 4. The shared-by-linking forms (CBF-1) — proof

The CBF-1 argument is short because the structure does the work:

1. **One source.** Each form has exactly one definition in a `src/` `.cpp` (or a header with a
   single `inline` definition): Merkle in `src/crypto/merkle.cpp`, the `Block` paths +
   `Transaction` in `src/chain/block.cpp:17-364`, genesis in `src/chain/genesis.cpp:458-461`,
   anon-address in `include/determ/types.hpp:153-155`.
2. **Same source compiled into each binary.** The CMake lists (§3.2) name those exact files in
   `determ-light`'s explicit list, and the glob (§3.1) pulls them into `determ`. The translation
   unit is byte-identical; the include paths resolve to the same `include/determ/...` headers
   (`CMakeLists.txt:138-143, 283-287`).
3. **Determinism of each form.** Each form is a pure function of its struct fields — no clock, no
   RNG, no global mutable state. (Block `signing_bytes` determinism is locked in-process by
   `determ test-block-hash` / `tools/test_block_hash.sh`; Transaction by
   `tools/test_tx_signing_bytes.sh`; genesis by `tools/test_genesis_determinism.sh`; the anon
   address is a pure `"0x" + to_hex(pk)`.) A pure function compiled from one source into two
   binaries returns the same bytes on the same input — **by construction**.

Therefore the only way two CBF-1 binaries can disagree is a **build defect**: a stale `.o` left
over from a partial rebuild so that one binary embeds an old version of the shared source. That is
not a source-level divergence; it is a relink failure, and it is exactly what CBF-3 detects (a
fresh full build that the parity tests pass against is the certificate that no stale relink
occurred). There is no source-level coincidence to rely on, because there is no second source.

**Backward-compatible conditional binds preserve CBF-1.** Several CBF-1 forms append fields only
when non-zero (the `Block::signing_bytes` conditional binds: `partner_subset_hash` at
`block.cpp:329-334`, `state_root` at `:345-350`, region at `:317-320`; the genesis hash's
"only-when-non-default" mixes at `genesis.cpp:392-423`). Because the **same** conditional source
compiles into both determ and light, the *conditions* are evaluated identically too — the
backward-compat guards do not create a divergence surface between CBF-1 binaries; they create one
only against the CBF-2 mirror (§5), which is why `light_compute_block_digest` must replicate the
conditions explicitly.

## 5. The re-implemented mirrors (CBF-2) — where vigilance lives

Two surfaces are **not** protected by CBF-1 and carry the divergence risk.

### 5.1 Transaction `signing_bytes` (three independent implementations)

| Impl | Location | Notes |
|---|---|---|
| canonical | `src/chain/block.cpp:17-29` | `Transaction::signing_bytes`; linked into `determ` and (via the shared `block.cpp`) into `determ-light` for the *struct* path |
| light mirror | `light/sign_tx.cpp:37-62` | `compute_signing_bytes(type, from, to, amount, fee, nonce)` — builds bytes directly from CLI args (the light client never instantiates a `Transaction`); its own comment at `:43` says "matches `src/chain/block.cpp::Transaction::signing_bytes`" |
| wallet mirror | `wallet/main.cpp:8224-8244` (`tx-sign-verify`), `:9543-9562` (`cold-sign`) | identical byte layout reconstructed inline; comment at `:8224` says "byte-for-byte identical to what `src/chain/block.cpp Transaction::signing_bytes` produces" |

The byte layout all three must hit (locked by `tools/test_tx_signing_bytes.sh` golden vectors):
`u8(type) ‖ from ‖ 0x00 ‖ to ‖ 0x00 ‖ u64_be(amount) ‖ u64_be(fee) ‖ u64_be(nonce) ‖ payload`.
The light/wallet mirrors omit `payload` because the light/wallet tx types carry an empty payload
(`light/sign_tx.cpp:50` comment) — a *scope* restriction, not a layout divergence: for the tx
types they sign, the trailing `payload` is the empty string and contributes zero bytes, so the
mirror is exact for its domain.

**Why a mirror at all?** Both the light client and the wallet sign *without* the daemon's account
model — they take raw CLI/keyfile inputs and must not pull `src/chain/block.cpp`'s broader
dependencies (chain state, JSON validation) into a stripped binary. The cost is that three byte
layouts must stay identical by discipline; the guard is CBF-3.

### 5.2 The committee block digest (`light_compute_block_digest`)

The single highest-stakes mirror. Canonical: `src/node/producer.cpp::compute_block_digest`
(the bytes the K-of-K committee signs in Phase 2). Mirror: `light/verify.cpp:57-92`. The file's
own header comment (`light/verify.cpp:32-56`) is explicit: "COPY OF producer.cpp:577-591 — keep in
sync ... If the upstream byte-order or field set ever changes, mirror it here." The mirror
replicates the producer's two trailing **conditional** appends:

- `partner_subset_hash` bound only when non-zero (`light/verify.cpp:75-78`, mirroring producer) —
  the merged-signing dimension; soundness in `PartnerSubsetDigestBindingSoundness.md` (FB55).
- the canonical `timestamp` bound only when the header carries `creator_proposer_times`
  (`light/verify.cpp:88-90`, mirroring producer) — the reconciled-timestamp dimension; soundness
  in `TimestampReconciliationSoundness.md` (FB56).

This mirror is the verification anchor of the **entire** light-client proof family: T-L2 (committee
head trust) recomputes this digest and checks K Ed25519 signatures against it
(`LightClientCompositionMap.md` §3.1, L-2). If the mirror drifts from the producer by one byte,
every committee signature fails to verify and the light client fail-closes (a false-negative, never
a false-positive — the failure mode is *safe* but breaks liveness). The light digest **deliberately
does not** bind the four rpc-headers-stripped heavy collections (it fail-closes on F2 / cross-shard
blocks instead; `light/verify.cpp:44-55`), so the mirror is *narrower* than the producer by design,
not by drift — and that narrowing is sound precisely because it only ever refuses, never
over-accepts.

## 6. The empirical relink guard (CBF-3) — test surface

The parity tests make **a second Determ process the reference**. This is the antidote to the
classic fuzz-test failure mode (a hand-rolled reference re-implementation can itself be wrong);
judging by *agreement between two independent binaries* means the test passes only if both
implementations land on the same bytes. The empirically-verified self-test for this document
(run on this Windows host, all three release binaries):

```
light  (light/sign_tx.cpp signing_bytes):      21b8e8be…865f67
wallet (wallet/main.cpp signing_bytes):        21b8e8be…865f67
determ (src/chain/block.cpp compute_hash):     21b8e8be…865f67
PASS: 3 independent signing_bytes impls agree byte-for-byte on tx_hash
```

The three numbers are produced by three **different** code paths — `determ-light sign-tx`
(re-impl), `determ-wallet tx-sign-verify` (re-impl), `determ tx-hash` (canonical
`Transaction::compute_hash`) — over one signed TRANSFER. Their equality is CBF-1+CBF-2 holding on
a real build.

| Test (`tools/`) | What it cross-checks | CBF mapping |
|---|---|---|
| `test_merkle_cross_binary_fuzz.sh` | wallet `merkle-proof`/`merkle-root` **generates**, light `verify-state-proof` **verifies**; 24 random leaf sets, sizes 1..N; tamper-rejection on sibling / value_hash / root | CBF-1 (shared `src/crypto/merkle.cpp` in **both** wallet and light) + CBF-3 |
| `test_light_sign_tx.sh` | light `sign-tx` (re-impl `signing_bytes`) → wallet `validate-tx` accepts (re-impl verifier) → proves cross-binary `signing_bytes` parity | CBF-2 (Transaction mirror) + CBF-3 |
| `test_wallet_tx_hash_cross_command_fuzz.sh` | wallet `derive-tx-hash` vs its own sign path, cross-command tx_hash stability | CBF-2 (wallet mirror self-consistency) |
| `test_tx_signing_bytes.sh` | golden-vector byte-layout lock for `Transaction::signing_bytes` (the layout all three mirrors must hit) | CBF-2 reference layout |
| `test_tx_signing_determinism.sh` | in-process `determ` determinism of `Transaction::signing_bytes` (100 calls stable) | CBF-1 determinism premise (§4.3) |
| `test_genesis_determinism.sh` | in-process `determ` `compute_genesis_hash` determinism | CBF-1 determinism premise (genesis) |
| `test_block_hash.sh` | in-process `Block::signing_bytes`/`compute_hash` determinism | CBF-1 determinism premise (block) |
| `test_anon_address_case.sh` | `make_anon_address` canonical lowercase + `normalize_anon_address` round-trip | CBF-1 anon-address form |

**Windows-host caveat (B0 portability).** On this host the `compute_genesis_hash` path has a known
edge, so genesis-hash-dependent cross-binary checks **SKIP** here (per `tools/common.sh`'s
SKIP-when-missing / SKIP-on-edge pattern). The Transaction `signing_bytes` / `tx_hash` and the
block-digest paths **do** work on Windows (demonstrated above), so every-host CBF-2 coverage rides
on those; the genesis CBF-1 leg is exercised on the Linux/WSL2 cluster runs. This is a *test-host*
limitation, not a CBF gap: CBF-1 for genesis still holds by construction (one shared
`src/chain/genesis.cpp`); only the empirical confirmation defers to a non-Windows host.

## 7. Composition with the rest of the corpus

CBF sits **beneath** the light-client family and **beside** the per-format proofs:

- **CBF → `LightClientCompositionMap.md` / `LightClientThreatModel.md`.** The map's L-1
  ("genesis-encoding determinism") and L-2 ("`light_compute_block_digest` ≡ producer digest,
  byte-for-byte") are *exactly* the CBF claims specialized to the light binary. CBF-1 supplies L-1
  (shared `genesis.cpp`); CBF-2 + CBF-3 supply L-2 (the digest mirror + its parity tests). T-L5's
  `sign-tx` leg ("`compute_signing_bytes` byte-identical to chain `Transaction::signing_bytes`",
  map §5 row) is CBF-2 for the Transaction form.
- **CBF ⟂ the per-format soundness proofs (CBF-4).** `MerkleTreeSoundness.md` (MT-1..MT-5) proves
  the Merkle form is collision/preimage-sound; CBF-1 proves all three binaries compute *that same*
  form. `BinaryCodecRoundTripSoundness.md` proves the wire codec round-trips; CBF is about the
  hashed content, not the frame. `PartnerSubsetDigestBindingSoundness.md` /
  `TimestampReconciliationSoundness.md` prove the two conditional digest dimensions *bind*; CBF-2
  proves the light mirror *replicates* those binds. The two layers multiply: agreement × soundness
  = trustworthy cross-binary interop.
- **CBF assumes (does not re-prove) {B0}.** The honest-build assumption is the analog, at the build
  layer, of `LightClientThreatModel.md`'s exclusion of `A_local` at the runtime layer.

## 8. Status

- **New subject.** No prior proof states the cross-binary build-time reuse structure as its own
  theorem. `LightClientCompositionMap.md` *uses* the fact (L-1/L-2) without proving it as a
  standalone structural claim; `BinaryCodecRoundTripSoundness.md` and `WireFormatBackwardCompat.md`
  cover the wire *codec*, a disjoint subject (network frame, not hashed content); the
  `test_*_determinism.sh` suite is the empirical surface but had no proof-level home. CBF is that
  home.
- **Honest two-regime account.** The headline "all three binaries link the same `src/` modules" is
  **true for the Merkle / Block / genesis / key / anon-address forms** (CBF-1, shared by linking)
  and **deliberately false for the Transaction `signing_bytes` and the committee digest** (CBF-2,
  re-implemented mirrors). This document states both precisely rather than over-claiming a uniform
  "one implementation everywhere," because the wallet links only `wallet/*.cpp` (CMakeLists.txt:232)
  and the light client re-implements two forms by necessity. The strength of the guarantee is
  exactly as strong as the structure, no stronger.
- **Empirically verified on this host.** The three-binary `tx_hash` parity (§6) was run live: all
  three independent `signing_bytes` implementations produced the identical hash
  `21b8e8bec9f73b65d21d91cd46b247de168bcba48beb3e1bdbb7922ebd865f67` over one signed TRANSFER.
- **Assumption.** B0 (honest build from one tree, relinked). CBF defends only the agreement *given*
