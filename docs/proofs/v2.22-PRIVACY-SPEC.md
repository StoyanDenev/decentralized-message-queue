## 1. Scope and model

Determ ships **amount privacy** via a **named-input confidential-transaction pool** over NIST P-256 — Pedersen commitments (`C = v·G + r·H`, `H` nothing-up-my-sleeve per RFC 9380) plus aggregated Bulletproofs range proofs and Schnorr balance proofs (§3.19). The pool is **profile-agnostic**: one binary, available under every deployment profile (CryptoProfile is a posture label, not a code switch).

**The named-input model.** Confidential value lives as unspent commitments ("notes") in the `cn:` state namespace. Inputs reference prior output commitments **by name**; membership = presence in the unspent set; **the commitment IS its own nullifier** — spending removes it, so a note is spendable at most once with no separate nullifier set. Three transaction types:

| Op | TxType | Move | Amount visibility |
|---|---|---|---|
| SHIELD | 12 | transparent → pool | PUBLIC (deposit `A` in the clear) |
| UNSHIELD | 13 | pool → transparent | PUBLIC (exit `A` in the clear) |
| CONFIDENTIAL_TRANSFER | 14 | pool → pool | **HIDDEN** (only the fee is public) |

Amount privacy exists **in motion** (CONFIDENTIAL_TRANSFER) and **at rest** (pool notes are hiding commitments). It does NOT extend to the transaction graph — see §4.

## 2. As-built consensus surface (verified)

### 2.1 SHIELD — transparent → confidential deposit

- `TxType::SHIELD = 12` — [block.hpp:198](../../include/determ/chain/block.hpp). `tx.amount = A` is the PUBLIC deposit; `tx.payload = C(33 SEC1) ‖ balance_proof(65)`, exactly 98 bytes, part of `signing_bytes()` (the depositor's signature authenticates `C`).
- **Accept-rule** `determ_shield_verify` — [ctxbundle.c:85](../../src/crypto/pedersen/ctxbundle.c): computes the excess `E = C − A·G` (`determ_p256_balance_excess` with one input, zero outputs, fee = `A`) and requires a verifying Schnorr PoK that `E = r·H`. A verifying proof forces `C` to commit to **exactly** `A` (a non-zero `G`-component in `E` has no proof under unknown `log_G(H)` — CTB-3), so a depositor cannot mint by over-committing.
- **Apply** — [chain.cpp:802](../../src/chain/chain.cpp): checks `balance ≥ A + fee`, payload length, re-verifies the proof, rejects a duplicate commitment ([chain.cpp:814](../../src/chain/chain.cpp) — no double-mint), then debits `A + fee`, inserts the note, and `accumulated_shielded_ += A` ([chain.cpp:818](../../src/chain/chain.cpp)).

### 2.2 UNSHIELD — confidential → transparent withdraw

- `TxType::UNSHIELD = 13` — [block.hpp:207](../../include/determ/chain/block.hpp). Same 98-byte payload shape; `tx.amount = A` is the PUBLIC exit amount; `A ≥ fee` required ([chain.cpp:833](../../src/chain/chain.cpp)).
- **The context-bound proof (the load-bearing property).** A bare PoK of the note blinding is replayable — a mempool observer could copy `C‖proof` into their own tx and redirect the credit. The fix: the proof's Fiat-Shamir challenge hashes `E ‖ T ‖ ctx32` (98 bytes, vs the unbound 66 — domain-separated, so SHIELD and UNSHIELD proofs never cross-verify) — `determ_p256_balance_verify_bound` [balance.c:133](../../src/crypto/pedersen/balance.c) — where `ctx32 = SHA-256("determ-unshield-v1" ‖ len64(from) ‖ from ‖ len64(to) ‖ to ‖ nonce ‖ amount)`, length-prefixed so distinct `(from,to)` never alias. The digest comes from the single shared helper `unshield_spend_ctx_hash` — [shielded.hpp:28](../../include/determ/chain/shielded.hpp) — used by the client prover, the validator, and apply (the S-043 one-formula-one-function rule). Any change to from/to/nonce/amount invalidates a captured proof.
- **Apply** — [chain.cpp:823](../../src/chain/chain.cpp): note must exist ([chain.cpp:843](../../src/chain/chain.cpp) — also blocks double-spend), bound proof re-verified via `determ_unshield_verify` [ctxbundle.c:96](../../src/crypto/pedersen/ctxbundle.c), then the note is **erased** (its own nullifier, [chain.cpp:851](../../src/chain/chain.cpp)), `accumulated_shielded_ -= A`, and `tx.to` is credited `A − fee` with an overflow-checked add ([chain.cpp:853](../../src/chain/chain.cpp)). The withdrawer's transparent balance is untouched — value comes from the pool.
- **Cross-shard rejection.** A cross-shard UNSHIELD (`tx.to` routing off-shard) is rejected at ALL THREE sites — validator [validator.cpp:1120](../../src/node/validator.cpp), producer [producer.cpp:1047](../../src/node/producer.cpp), apply [chain.cpp:840](../../src/chain/chain.cpp) — a no-op that does NOT spend the note. Without it, a cross-shard credit would silently break the K-shard aggregate supply identity (HIGH adversarial-audit finding, remediated 2026-07-06).

### 2.3 CONFIDENTIAL_TRANSFER — confidential → confidential

- `TxType::CONFIDENTIAL_TRANSFER = 14` — [block.hpp:216](../../include/determ/chain/block.hpp). `tx.payload` = the **DCT1 bundle**: `"DCT1" ‖ n_in(1) ‖ m(1) ‖ n(1) ‖ fee(8 BE) ‖ C_in[n_in·33] ‖ C_out[m·33] ‖ agg_rangeproof ‖ balance_proof(65)` ([ctxbundle.c:31](../../src/crypto/pedersen/ctxbundle.c); `n_in, m ∈ [1,255]`, `n` a power of 2 ≤ 64, `m·n ≤ 256`).
- **Accept-rule** `determ_ctx_bundle_verify` — [ctxbundle.c:58](../../src/crypto/pedersen/ctxbundle.c): ONE aggregated Bulletproofs range proof over `C_out` directly (each hidden output `∈ [0, 2^n)`; the composition identity is structural, no splice) AND the balance proof against the verifier-recomputed excess `E = ΣC_in − ΣC_out − fee·G` (`Σv_in = Σv_out + fee`, fee PUBLIC). `determ_ctx_bundle_header` ([ctxbundle.c:41](../../src/crypto/pedersen/ctxbundle.c)) validates magic/params and that `len` equals the exact expected size, so the commitment slices are provably in-bounds (SP-13).
- **The inflation guard (the load-bearing consensus check).** The crypto cannot tell that two *listed* inputs are the *same* pool note — listing one note worth `V` twice would prove `Σv_in = 2V` and mint `2V − fee`. Apply defeats this with an **all-or-nothing dedup**: every input+output key goes through one `std::set`, gathered BEFORE any mutation; a repeated key, a spent/unknown input, or an output colliding with an existing note skips the whole tx — [chain.cpp:886](../../src/chain/chain.cpp)–899 (SP-11).
- **Apply** — [chain.cpp:865](../../src/chain/chain.cpp): `tx.fee` must equal the bundle's public fee ([chain.cpp:877](../../src/chain/chain.cpp)), inputs erased, outputs inserted, `accumulated_shielded_ -= bundle_fee` ([chain.cpp:904](../../src/chain/chain.cpp)) — only the public fee leaves the pool, to creators. **Pool→pool**: no transparent `tx.to` credit, hence no cross-shard vector (producer accounting is a no-op — [producer.cpp:1051](../../src/node/producer.cpp)).

### 2.4 Supply identity, state-root additivity, snapshots

- **Supply.** `expected_total() = genesis + subsidy + inbound − slashed − outbound − accumulated_shielded_` — [chain.hpp:462](../../include/determ/chain/chain.hpp). The **always-on A1 assertion** after every block apply compares it to the live transparent sum and **throws** on any drift — [chain.cpp:1535](../../src/chain/chain.cpp)–1537 (fail-closed; not debug-only). Total real supply = `live_total_supply() + accumulated_shielded_`. Pedersen binding guarantees the exit/fee amounts equal the shield-time values, so the counter cannot underflow (SP-7/SP-12).
- **State root.** The `c:accumulated_shielded` counter leaf is emitted only when non-zero ([chain.cpp:416](../../src/chain/chain.cpp)) and the `cn:` note leaves (`value = SHA256(commitment ‖ height)`, sorted-map deterministic order) only for a non-empty pool ([chain.cpp:422](../../src/chain/chain.cpp)) — so a chain that never SHIELDs has a **byte-identical** state root to a pre-§3.22 chain. The FAST golden state-root corpus, green with this code compiled in, IS the invariance witness (SP-3).
- **Snapshots.** Serialize emits the counter/pool only when non-zero/non-empty ([chain.cpp:1767](../../src/chain/chain.cpp)); restore defaults to zero/empty ([chain.cpp:1897](../../src/chain/chain.cpp)) — shield-free snapshot byte-identity + exact round-trip (SP-5).

### 2.5 Enforcement sites and gates

Every accept-rule is enforced at the **validator** (authoritative submit-time gate: SHIELD [validator.cpp:1093](../../src/node/validator.cpp), UNSHIELD [validator.cpp:1106](../../src/node/validator.cpp), CONFIDENTIAL_TRANSFER [validator.cpp:1132](../../src/node/validator.cpp)) AND re-verified at **apply** (belt-and-suspenders against a hostile producer), with the **producer** doing provisional accounting ([producer.cpp:1033](../../src/node/producer.cpp)). Anonymous (bearer) accounts may use all three ops ([validator.cpp:596](../../src/node/validator.cpp) — a bridge permission, not graph privacy). Gates: `test-shield` / `test-unshield` / `test-confidential-transfer` via [tools/test_shield.sh](../../tools/test_shield.sh), [tools/test_unshield.sh](../../tools/test_unshield.sh), [tools/test_confidential_transfer.sh](../../tools/test_confidential_transfer.sh); each increment cleared a refute-by-default adversarial audit.

## 3. Threat model and invariants

Adversary: a malicious tx author, a mempool observer (front-runner), and a buggy/hostile block producer, all classical (see NC-4). What holds, with the proving claim in [`ShieldedPoolSoundness.md`](ShieldedPoolSoundness.md):

| Attack | Defense | Claim |
|---|---|---|
| Mint on deposit (commit to more than debited) | `E = C − A·G` must open to zero on `H` | SP-1 |
| Double-mint (re-SHIELD the same `C`) | duplicate-commitment no-op at apply | SP-4 |
| Front-run/redirect a withdraw (`C‖proof` theft) | context-bound Fiat-Shamir challenge over `E‖T‖ctx32` | SP-6 |
| Double-spend a note | commitment-as-nullifier: erased on spend, absent = no-op | SP-8 |
| Inflation via duplicate-listed inputs | all-or-nothing dedup across inputs+outputs, gather-before-mutate | SP-11 |
| Overflow/negative hidden outputs | aggregated range proof, each output `∈ [0, 2^n)` | SP-10 |
| Net supply drift (any op) | `expected_total()` identity + always-on A1 throw | SP-2/7/12 |
| Cross-shard supply break via UNSHIELD | rejected at validator+producer+apply | NC-5/SP-13 |
| Malformed-payload memory unsafety | exact-length header validation before slicing | SP-13 |
| Accept-rule drift across enforcement sites | shared helpers (S-043 rule): one verify fn, one ctx fn | SP-9 |
| Hostile producer including an invalid tx | apply-side re-verify of every accept-rule | SP-4/9 |

**Assumptions (inherited, not re-proven here):** P-256 ECDLP + Fiat-Shamir ROM (the CTB-* reductions in [`ConfidentialTxBalanceSoundness.md`](ConfidentialTxBalanceSoundness.md)); Pedersen binding computational / hiding information-theoretic ([`PedersenCommitmentSoundness.md`](PedersenCommitmentSoundness.md)); SHA-256 collision resistance for the ctx digest. The tx-bound proof is a new, owner-authorized (2026-07-06), session-audited construction — not externally reviewed (L-5). Conformance is fixed witnesses + the golden corpus, not the input space (L-2). Provers are constant-time-hardened in isolation ([`ConstantTimeInventory.md`](ConstantTimeInventory.md)); extending the timing probe to the integrated path is decided verification work (register D2). The dedicated adversarial audit + beta soak of this whole surface is register item D1 (§6).

## 4. Non-goals

- **Input-unlinkability / graph privacy — OUT.** Inputs are NAMED: an observer follows the note graph (which notes fed which transfer) even though transferred values are hidden (NC-7); UNSHIELD links the spent note to a public recipient (NC-6); `tx.from`/`tx.to` and the SHIELD/UNSHIELD amounts are public (NC-2/NC-3). The LSAG/CLSAG/RingCT library that would close this was **removed from the tree by owner decision 2026-07-09** (register B2, jointly A7): graph privacy is not a launch requirement, and under no-migrations every unwired primitive is permanent audit surface — the same razor that deleted secp256k1, the Z_p* backend, and libsodium. The three soundness docs ([`LsagRingSignatureSoundness.md`](LsagRingSignatureSoundness.md), [`ClsagRingSignatureSoundness.md`](ClsagRingSignatureSoundness.md), [`RingCTSpendCompositionSoundness.md`](RingCTSpendCompositionSoundness.md)) + git history remain the design record; re-adding is a normal reviewed feature if ever required. *(Execution note: the removal EXECUTED 2026-07-09 — the FROST and ringsig library files, their five test surfaces, vectors, and python oracles are deleted from the tree; FAST is 202.)*
- **On-chain output-secret delivery — OUT (NC-8).** A transfer's recipient must learn `(value, blinding)` for their output note **off-chain**; there is no encrypted-memo / stealth-address channel. The §5 view-key layer adds audit *disclosure*, not on-chain delivery.
- **Not post-quantum (NC-4).** Soundness rests on P-256 ECDLP. The PQ signature track (§3.21 `PQ_TRANSFER`) is separate.
- **Single-shard (NC-5).** Cross-shard UNSHIELD is rejected in code; a receipted cross-shard confidential withdraw is a separate, owner-gated step.

## 5. TO-BUILD — the view-key / audit / light layer (DECIDED 2026-07-09, not yet built)

Everything in this section is **decided but unimplemented** — no view-key, audit, or light-CT-verification code exists in the tree at rewrite time. Authority: pre-launch decision register items A1, A2, A3 (owner, 2026-07-09). Build order: pre-beta critical path.

### 5.1 View keys — per-epoch HKDF derivation (register A1, Option C ratified)

Each account holds a long-term `view_master` keypair; per-epoch view keys derive client-side:

```
vk_epoch_n = HKDF(view_master_sk, "VK" || chain_id || account_addr || epoch_n)
```

Bounded per-epoch exposure of a leaked key, **zero on-chain cost** for rotation (no rotation tx, no "forgot to rotate" failure mode), and epoch windows map directly onto quarterly/annual audit scopes. Reuses the shipped HKDF-SHA-256 primitive ([hkdf.c](../../src/crypto/sha2/hkdf.c)) — no new cryptography. Rejected cadences (genesis-pinned, per-tx, hybrid) are recorded in the register and git history.

### 5.2 Audit integration — FULL (register A2, composes with v2.24)

- **Dual-mode disclosure:** `view_master_sk` (full account history — in-house compliance) or `vk_epoch_n` (one epoch window — external regulator), disclosed off-chain; both reuse the §5.1 derivation.
- **Baked-in audit key:** an optional `audit_view_master_pk` account field giving a named auditor standing derivation rights, rotated via a `ROTATE_AUDIT_KEY` tx (the v2.24/v2.26 rotate pattern).
- **On-chain audit trail:** a `LOG_AUDIT_ACCESS` tx recording disclosure events, for deployments that must audit the auditors.

FULL (not minimal) was chosen because the first-wave DApp is a regulated one — a provably-fair lottery remake of the Liberty Bell slot machine (owner requirement, 2026-07-09) — consuming confidential stakes/payouts + scoped regulator disclosure end-to-end.

### 5.3 Light-client verification — FULL (register A3)

The daemon's cleartext-balance cross-check cannot cover hidden value, so `determ-light` gets a confidential read path: verify commitment **inclusion** in the committee-signed `state_root` (the `cn:` leaves, via the existing state-proof path in [trustless_read.cpp](../../light/trustless_read.cpp)) **AND verify the range/balance proofs client-side** — a light client does not trust the committee for CT validity. Chosen over inclusion-only because Liberty Bell players are light-client users.

## 6. Flag-gate and rollback (register D1 — decided; flag not yet built)

Confidential transactions ship **behind a per-deployment disable flag**: a deployment launched with the flag off has its validator reject TxTypes 12/13/14 at submit-time, decoupling mainnet launch from CT maturity; the flag is flipped on once the D1 adversarial audit + beta soak clear. No such flag exists in code yet — it is a decided pre-beta build item. Still-accurate rollback facts carried from the prior design: a disabled pool leaves any existing commitments intact in chain history (unspendable while disabled, no value destroyed — the A1 identity still books `accumulated_shielded_`), and the governance pause remains the emergency backstop for a live consensus bug. Under no-migrations there is no flag-day machinery — the flag is per-deployment configuration, not a protocol phase.
