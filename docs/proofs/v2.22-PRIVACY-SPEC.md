## 1. Scope

**Profile applicability.** Confidential transactions are available in BOTH profiles — via the §3.22 P-256 shielded pool (Pedersen + Bulletproofs over NIST P-256), which is profile-agnostic. The FIPS constructions are built on FIPS-approved primitives (P-256 + SHA-256), but the ZK range-proof / ring-signature CONSTRUCTIONS are not themselves FIPS-validated algorithms (NIST has not standardized them); a deployment requiring per-operation CMVP validation treats them as out-of-module.

| Profile | Confidential tx availability |
|---|---|
| `cluster` (FIPS — in-house enterprise / financial services / regulated) | ✅ Available (§3.22 P-256 shielded pool; FIPS-aligned primitives, not a FIPS-validated algorithm) |
| `tactical` (FIPS — military / defense / embedded) | ✅ Available (§3.22 P-256 shielded pool; FIPS-aligned primitives, not a FIPS-validated algorithm) |
| `web` (MODERN — web-scale, commercial single-cluster) | ✅ Available |
| `regional` (MODERN — regional sharded) | ✅ Available |
| `global` (MODERN — global-scale) | ✅ Available |

FIPS-profile deployments can use the §3.22 P-256 confidential-tx shielded pool with view-key disclosure (identical to MODERN), OR — where per-operation CMVP validation is mandatory — fall back to clear-amount TRANSFER + v2.24 audit hooks. This is a deployment/compliance choice, not a chain limitation.

v2.22 confidential transactions ship as documented below. The wired implementation is the §3.22 P-256 shielded pool (profile-agnostic — used by both FIPS and MODERN). MODERN adds no separate ZK curve; the big-prime Z_p* §3.20 stack is an unwired library primitive, not a profile backend.

This spec covers ONLY confidential-amount transactions and their view-key infrastructure. It does NOT cover:
- Transaction-graph privacy (sender / recipient unlinkability) — out of scope for v2.22; the chain still records `tx.from` and `tx.to` publicly. Adding graph privacy is a v2.x follow-on that requires deeper architectural changes (ring signatures, stealth addresses, or zk-SNARK shielded pools).
- Smart-contract privacy — Determ has no contract VM; computation privacy is delivered via the God Stack pattern (zk-VM L2), out of v2.22's scope.
- Bridge interoperability with other confidential-amount chains (Zcash, Monero) — out of scope; v2.23's bridge ships clear-amount cross-chain transfers.

The artifacts produced by v2.22:
- **Pedersen-commitment amounts** in TRANSFER (and any future confidential-amount tx type)
- **Bulletproof range proofs** ensuring committed amounts are in `[0, 2^64)` (no underflow)
- **Balance-conservation proofs** ensuring sum(inputs) = sum(outputs) without revealing values
- **Per-epoch view-key derivation** for amount recovery by recipient, sender, and designated auditors
- **Sender-recipient ephemeral-DH handshake** for per-tx amount-decryption key derivation
- **Audit-mode disclosure tooling** (composes with v2.24)

## 2. Design decisions

### Q1: View-key rotation cadence

**Decision: per-epoch automatic rotation via HKDF hierarchical derivation (Option C).**

Each account has a long-term `view_master` keypair. Per-epoch view keys derive deterministically:

```
vk_epoch_n = HKDF(view_master_sk, "VK" || chain_id || account_addr || epoch_n)
pk_epoch_n = vk_epoch_n · G   (point on secp256k1)
```

Where `epoch_n` is the epoch number at tx-apply time. The recipient (or auditor with view_master) recomputes the same `vk_epoch_n` to decrypt amounts within epoch n.

**Rationale.**
- **Bounded exposure per epoch.** A compromised `vk_epoch_n` exposes only epoch n's amounts; subsequent epochs unaffected by HKDF forward secrecy.
- **Zero on-chain cost for rotation.** Derivation is purely client-side. No `ROTATE_VIEW_KEY` tx, no gas burn.
- **Maps to regulator cadence.** Quarterly/annual audits → grant per-epoch keys for the audit window only. No leakage of out-of-scope periods.
- **No rotation discipline required.** Happens automatically at every epoch boundary; eliminates "forgot to rotate" OPSEC failure.
- **Composes with v2.10's epoch infrastructure.** Epoch boundaries are already a protocol sync point.

**Alternatives rejected:**
- *Genesis-pinned (Option A):* Catastrophic forward-secrecy failure — compromise = permanent exposure for account lifetime.
- *Per-account mutable via tx (Option B):* Burden-heavy (user rotation discipline; gas cost); audit re-disclosure friction.
- *Per-tx ephemeral (Option D):* Auditor disclosure becomes expensive (per-tx key derivation for account-level audits typical in regulation).
- *Hybrid master + per-tx (Option E):* Over-engineered for v2.22's scope; could be a v2.x follow-on if specific deployments demand it.

**Master compromise mitigation.** A `ROTATE_VIEW_MASTER` tx (analogous to v2.26 `ROTATE_KEY`) allows one-time recovery from catastrophic master compromise. Not for routine rotation (which Option C handles via HKDF) — only for emergency.

**Optional hierarchical HKDF for stronger forward secrecy (recommended).** The flat per-epoch derivation `vk_epoch_n = HKDF(view_master, "VK" || chain_id || account_addr || epoch_n)` gives epoch-level PFS — a compromised epoch key reveals only that epoch's amounts. But compromise of `view_master` itself exposes ALL past and future per-epoch keys. To shrink the master-compromise blast radius from "all history" to "one period of history," derive via a hierarchical tree:

```
view_master_sk   (long-term, hardware-isolated; never network-touched)
    │
    └─ period_sk_p = HKDF(view_master_sk, "PERIOD" || chain_id || account_addr || period_p)
              │
              └─ vk_epoch_n = HKDF(period_sk_p, "VK" || epoch_n)
                              for epoch_n in [p * period_blocks, (p+1) * period_blocks)
```

Where `period_blocks` is a genesis-pinned parameter (default: `view_master_period_blocks = 10000` ≈ 100 epochs at `epoch_blocks=100`). Each period derives a `period_sk_p` from the master; per-epoch keys then derive from `period_sk_p`. The user (or auditor with the master) recomputes any `vk_epoch_n` via the two-level derivation.

**What this delivers:**
- **Period-level PFS against master compromise** — a compromised `view_master_sk` at time T exposes only the *current and future* periods' keys. *Past* periods' derived keys are not recoverable from a stolen master alone — the user must have rotated to a new `view_master_sk` between periods via `ROTATE_VIEW_MASTER`.

Wait — that last point isn't quite right. Hierarchical HKDF alone doesn't give past-period PFS against master compromise (HKDF is one-way derivation, not one-way evolution). The master can derive any past period's key just as easily as future ones.

**Honest re-statement of what hierarchical HKDF actually delivers:**

The hierarchical structure does NOT close the master-compromise gap on its own. It enables a separate operational protection: **the user can keep `view_master_sk` in cold/hardware-isolated storage and only derive `period_sk_p` for the current period for hot wallet use.** A compromise of the hot-wallet `period_sk_p` then exposes only that period — not all history. The master stays cold; an attacker compromising the hot wallet doesn't get the master.

This is a **cold/hot key separation** pattern, made structurally possible by the hierarchical derivation. Without the hierarchy, the user must keep `view_master_sk` available for every per-epoch derivation; with the hierarchy, the user can derive one `period_sk_p` per period from cold storage, then use it from hot storage for `period_blocks` epochs before refreshing.

**Recommended for high-stakes deployments** (regulated gambling, B2B payment, healthcare) where hardware-isolation of the master is operationally feasible. Not necessary for low-stakes deployments where the user must keep the master online anyway.

**Genesis fields (when adopted):**
- `view_master_period_blocks: u64` — 0 = flat derivation (Option C baseline); >0 = hierarchical derivation with this period length

**Cost:** ~50 LOC. Adds one HKDF level to the derivation; one new genesis field; documentation update to v2.22 + v2.24 (auditor disclosure can now be at master, period, or epoch granularity). Compatible with the rest of v2.22's design.

### Q2: Range-proof construction

**Decision: Bulletproofs over secp256k1 via libsecp256k1-zkp.**

**Rationale.**
- **Most-production-tested C99 Bulletproof implementation available.** libsecp256k1-zkp (maintained by Blockstream / Mimblewimble-Grin) has been running Bulletproofs in production since 2018-2019 in Liquid (Blockstream's Bitcoin sidechain) and Grin. ~5+ years of production exposure of Bulletproofs-over-secp256k1 specifically.
- **C99 native, libsodium-independent.** Aligns with the CRYPTO-C99-SPEC.md strategy of full libsodium removal. libsecp256k1 + libsecp256k1-zkp ship as vendored C99 modules. No external dependency.
- **Bitcoin-grade audit pedigree.** libsecp256k1 underlies Bitcoin Core (~$1T+ secured); audited multiple times; constant-time discipline reviewed by cryptographers including Pieter Wuille, Greg Maxwell, Jonas Nick.
- **Prime-order group natively.** secp256k1 has cofactor 1 — no cofactor-clearing complications. Same property as ristretto255.
- **No trusted setup.** Bulletproofs require no trusted setup ceremony. Matches Determ's mutual-distrust posture.
- **Aggregation friendly.** Bulletproofs aggregate naturally — a single proof can cover N range claims with O(log N) overhead. Critical for TRANSFER txs with multiple outputs.

**Trade-offs acknowledged.**
- **Adds secp256k1 as a second curve family.** Determ's cryptographic stack becomes: SHA-256 + curve25519 family (Ed25519 sigs, X25519 KX, FROST-Ed25519 threshold) + secp256k1 family (Bulletproofs, OPRF, prime-order needs). Three curves total; two underlying mathematical families. Documented as deliberate trade-off in `CRYPTO-C99-SPEC.md` §2.Q1.
- Proof size: ~600-1500 bytes per output (acceptable; <0.5% of typical block bandwidth at tactical profile).
- Verification cost: O(N log N) for N outputs. Aggregation helps; cost is bounded.
- Larger than Plonk/SNARK proofs but no trusted setup — the right trade for Determ.

**Alternatives considered:**
- *Bulletproofs over curve25519/ristretto255 (dalek-cryptography reference):* Earlier draft chose this. Reverted because the only mature C99 ristretto255 implementation is in libsodium; vendoring libsodium-derived ristretto255 source conflicts with the libsodium-removal goal in CRYPTO-C99-SPEC.md. libsecp256k1-zkp delivers Bulletproofs without any libsodium dependence.
- *Bulletproofs over BLS12-381 (blst):* Adds pairing-friendly curve to the stack; previously rejected for "two primitives" reasons. Pairings not needed (God Stack pattern offloads pairing-based ZK to external L2).
- *BBS+ signatures over BLS12-381:* Designed for selective-disclosure signatures, not range proofs. Wrong primitive.

**Library:** libsecp256k1 + libsecp256k1-zkp, vendored as `src/crypto/secp256k1/` + `src/crypto/secp256k1_zkp/` per CRYPTO-C99-SPEC.md §2.Q3. Pinned upstream commit. C99 native; no external runtime dependency. ~10 days integration (vs ~2-3 weeks for ristretto255 alternative via libsodium vendoring).

### Q3: Sender-recipient amount handshake

**Decision: ephemeral ECDH on the profile's native curve against recipient's published `view_master_pk`, then HKDF derivation with epoch label. Curve follows profile across all v2.22 DH: MODERN → secp256k1 ECDH, FIPS → NIST P-256 ECDH. X25519 is removed from v2.22 (stays in the stack for v2.10 FROST-Ed25519 only).**

Note: v2.22 confidential transactions ship in MODERN only (per §1, gated by C99-11). The amount handshake therefore runs on secp256k1 ECDH in practice. The "curve follows profile" principle is recorded here for v2.22-adjacent DH (e.g. audit-key exchange in v2.24 when extended to FIPS clear-amount deployments).

**Protocol (MODERN profile).**
1. Recipient publishes `view_master_pk` (secp256k1 public key) on-chain at account creation (in REGISTER tx payload extension, or via a one-time `PUBLISH_VIEW_KEY` tx for existing accounts).
2. Sender generates ephemeral `eph_sk` (secp256k1 secret); computes `eph_pk = eph_sk · G_secp256k1`.
3. Sender computes shared secret: `ss = ECDH_secp256k1(eph_sk, view_master_pk)` (x-coordinate of the shared point, per SEC 1).
4. Sender derives per-tx amount-encryption key: `aek = HKDF-SHA-256(ss, "AMT" || epoch_n || tx_hash)`.
5. Sender encrypts the amount: `amount_ct = XChaCha20-Poly1305(aek, amount_bytes, AAD = "TX-AMT" || tx_hash)`.
6. Tx carries: `commitment = aG + bH` (Pedersen on secp256k1 per Q2), `range_proof`, `eph_pk` (secp256k1, 33 bytes compressed), `amount_ct`.
7. Recipient (or auditor with `view_master_sk`) recomputes `ss = ECDH_secp256k1(view_master_sk, eph_pk)`, derives `aek`, decrypts `amount_ct`, verifies it matches `commitment` via blinding factor check.

**Curve choice rationale.**
- **Single-curve protocol.** Commitment, range proof, and DH all on secp256k1 — eliminates the prior two-curve mismatch. No curve translation in the apply path.
- **No new primitive.** secp256k1 ECDH uses the same field/group operations already vendored for Bulletproofs (Q2) and ECDSA. Zero additional vendoring cost.
- **Profile alignment.** Per `CRYPTO-C99-SPEC.md` C99-11, MODERN profile = secp256k1 family; FIPS profile = NIST P-256 family. v2.22 DH inherits this directly — no profile-specific carve-outs.

**Rationale.**
- **No bidirectional handshake.** Sender doesn't need recipient interaction; one-shot tx submission.
- **Forward secrecy via ephemeral.** Compromised `eph_sk` (sender-side) doesn't affect other txs.
- **Recipient anonymity preserved.** Only `view_master_pk` is public; tx-level decryption requires the sender's `eph_pk` (which is per-tx).
- **Audit compatibility.** Auditor with `view_master_sk` can decrypt any tx to/from the account using the same handshake (auditor doesn't need to be in the loop at tx-time).

**Library:** libsecp256k1 (already vendored for Q2 Bulletproofs and for ECDSA signing). XChaCha20-Poly1305 for AEAD via libsodium (already in tree). No new dependency.

### Q4: Audit integration (composes with v2.24)

**Decision: dual-mode disclosure — `view_master_sk` (full access) or per-epoch `vk_epoch_n` keys (scoped access).**

**Disclosure modes.**

| Mode | Auditor receives | Scope | Use case |
|---|---|---|---|
| Master | `view_master_sk` | All amounts to/from this account, ever | In-house compliance officer; permanent audit relationship |
| Per-epoch | `vk_epoch_n` for specific n | Amounts in epoch n only | External regulator; bounded audit period (quarterly, annual) |

Account holder discloses via off-chain channel (encrypted email, secure messaging, paper). No on-chain disclosure required — keeps audit relationships private.

**v2.24 integration.** `v2.24` adds an optional `audit_view_master_pk` field to the account state. If set, the named auditor has standing pre-authorization to derive view keys (via DH against the auditor's published pubkey). Rotation via a `ROTATE_AUDIT_KEY` tx (analogous to v2.26 `ROTATE_KEY`). Used by deployments where regulatory access is permanent (banks, licensed gambling operators) and the operator wants to bake the audit relationship into the account.

**On-chain audit trail.** When an auditor disclosure happens off-chain, there's no on-chain record by default. For deployments requiring auditable audit access (regulators wanting to know "who audited what when"), a `LOG_AUDIT_ACCESS` tx records the disclosure event. Optional — operator policy.

**Rationale.**
- **Map to regulatory reality.** Real regulator workflows: "Provide all 2026 Q3 transactions for account X." Per-epoch keys deliver exactly this. Real in-house compliance: "Our CFO needs ongoing access." Master key delivers that.
- **No new cryptographic primitive.** Both modes use the Q1 HKDF derivation.
- **Composes with v2.26.** ROTATE_AUDIT_KEY mirrors ROTATE_KEY; rotation cooldown applies; no new mechanism.

### Q5: Per-tx PFS selection (PRIV-6 amendment)

**Decision: opt-in Perfect Forward Secrecy per transaction via a one-time pubkey stream (OTPK). Account-level policy locks the mode at creation: `AUDITABLE_ONLY`, `PFS_ONLY`, or `MIXED`. In `MIXED`, sender picks per tx.**

**Problem this solves.** Q3's amount handshake binds every confidential tx to the recipient's long-term `view_master_pk`. Compromise of `view_master_sk` decrypts every past confidential tx for the account. Some accounts need per-tx PFS — amounts decryptable only by sender + recipient at tx-time, irrecoverable after — independent of the account's general audit posture.

**Cryptographic exclusivity (per ciphertext, not per account).** For any single ciphertext, PFS and key-disclosure-audit are mutually exclusive: PFS requires that no current secret can decrypt past traffic; audit-by-disclosure requires that some current secret can. The exclusivity is bound to the ciphertext. PRIV-6 honors this by tagging each tx with its mode and enforcing the implications at apply-time.

**Mode flag on TRANSFER.**

```
Transaction {
    ...existing fields...,
    amount_mode: enum { AMT_AUDITABLE, AMT_PFS },
    otpk_id:     Option<u64>,   // required iff mode = AMT_PFS
    ...
}
```

`AMT_AUDITABLE` runs the Q3 protocol verbatim (ECDH against `view_master_pk`). `AMT_PFS` substitutes the recipient's next unused OTPK for `view_master_pk` in the ECDH step; recipient deletes the corresponding privkey after applying.

**Account policy (immutable, set at account creation).**

```
Account {
    ...existing fields...,
    confidential_policy: enum {
        AUDITABLE_ONLY,   // validator rejects inbound AMT_PFS; PUBLISH_OTPK_BATCH disallowed
        PFS_ONLY,         // validator rejects inbound AMT_AUDITABLE; initial OTPK batch required at creation
        MIXED,            // both modes allowed; sender picks
    },
    otpk_stream: Vec<OtpkEntry>,   // see below; populated only if policy != AUDITABLE_ONLY
}
```

**OTPK lifecycle.**

```
OtpkEntry = Unused { otpk_id: u64, otpk_pk: secp256k1_point }   // 43 bytes
          | UsedMarker { hash: bytes32 }                        // 32 bytes
```

- Recipient publishes a batch of 32 OTPKs via `PUBLISH_OTPK_BATCH` (~1KB per batch). Recipient wallet retains the corresponding privkeys locally.
- Replenishment trigger: when unused count drops below 8 (default `OTPK_REPLENISH_THRESHOLD`).
- On apply of an `AMT_PFS` tx: validator atomically replaces `Unused{id, pk}` → `UsedMarker{SHA256(id || pk)}`. No grace window — the full pubkey entry is gone after apply.
- Recipient wallet: on observing the applied tx, recompute `ss = ECDH(otpk_sk, sender_eph_pk)`, decrypt amount, then **irrevocably delete `otpk_sk`** from local storage. After this point, no key in the system can decrypt the tx.

**Apply-path validation.**

1. `amount_mode == AMT_PFS` ∧ `otpk_id` absent → reject (`MALFORMED`).
2. `otpk_id` not in `recipient.otpk_stream` → reject (`OTPK_UNKNOWN`).
3. Referenced entry is `UsedMarker` → reject (`OTPK_ALREADY_USED`) — replay/race-loss case.
4. `recipient.confidential_policy == AUDITABLE_ONLY` ∧ `amount_mode == AMT_PFS` → reject (`POLICY_VIOLATION`).
5. `recipient.confidential_policy == PFS_ONLY` ∧ `amount_mode == AMT_AUDITABLE` → reject (`POLICY_VIOLATION`).
6. Validation otherwise identical to Q3.

**Race-loss handling.** If two senders concurrently submit txs referencing the same `otpk_id`, the first to apply wins; the second is rejected with `OTPK_ALREADY_USED`. Sender wallet retries with the next available OTPK.

**Empty-stream behavior (sender-side fallback).**
- Recipient = `MIXED`, OTPK stream empty → sender wallet silently falls back to `AMT_AUDITABLE`.
- Recipient = `PFS_ONLY`, OTPK stream empty → sender errors (`NO_OTPK_AVAILABLE`); recipient must publish a batch before further inbound is possible.
- Recipient = `AUDITABLE_ONLY` → sender always uses `AMT_AUDITABLE` (no OTPK stream exists).

**Outbound enforcement is wallet-policy, not chain-policy.** A regulated sender (bank, licensed operator) required to send only `AMT_AUDITABLE` configures its wallet to refuse `AMT_PFS` submission regardless of recipient policy. The chain does not enforce sender-side mode restrictions.

**Composition with other v2.22 decisions.**

| Existing | Impact |
|---|---|
| Q1 per-epoch HKDF | Unaffected. `view_master_sk` still derives `vk_epoch_n` for `AMT_AUDITABLE`. |
| Q2 Bulletproofs / Pedersen | Unaffected. Commitment + range proof construction identical for both modes. |
| Q3 ECDH amount handshake | Reused — PFS mode substitutes OTPK for `view_master_pk` as recipient pubkey. |
| Q4 dual-mode audit | Composes — `AMT_PFS` txs appear in audit reports as `amount: UNDISCLOSED-PFS`; report includes `N_auditable / N_pfs_undisclosed` ratio. |
| C99-11 MODERN-only | Inherits — PFS mode also MODERN-only. |

**No new cryptographic primitive.** Reuses secp256k1 ECDH already vendored for Q2/Q3.

**Rationale.**
- **Per-tx selection honors the per-ciphertext crypto constraint.** Account-level lock at creation prevents per-tx mode hiding within a `MIXED`-claimed regulated account (the account commits to its posture upfront).
- **Permanent hash-only marker (Option D) minimizes state growth.** ~32 bytes per used OTPK forever; ~3.2GB/year at 1M accounts × 100 PFS receives/year.
- **Atomic replacement at apply step.** No background pruning scheduler; no grace-window state machine.

**Alternatives rejected.**
- *Per-tx mode without account-level lock:* allows MIXED-claimed accounts to selectively hide individual txs from audit. Regulatory evasion concern.
- *Bidirectional ephemeral handshake (sender + recipient ephemeral, no OTPK stream):* requires recipient online before sender can submit. Breaks async tx UX.
- *Forward-secure encryption (puncturable encryption / forward-secure HIBE):* research-grade primitive; no production-tested C99 implementation. Also breaks audit recovery in `MIXED` mode.
- *Grace-window OTPK retention (Option A):* buys nothing operational; used pubkeys have no retrieval purpose post-apply.

#### Q5.1: Wallet-loss reconciliation (sender-side records)

**Problem.** PRIV-6 delivers PFS by having the recipient delete `otpk_sk` after applying an inbound tx. If the recipient's wallet state is lost before that apply (or before observing it), the recipient can no longer decrypt the amount. Funds remain in the account (commitment math already updated the balance), but bookkeeping breaks. Pure PFS forbids OTPK-privkey recovery — by definition, any recoverable secret defeats forward secrecy.

**The only remaining recovery channel is the sender**, who knows what they sent. PRIV-6.1 standardizes this off-chain channel without weakening the on-chain PFS guarantee.

**Key insight.** The recipient can verify a sender's claim of "I sent you X to otpk_id N" without ever holding `otpk_sk` — because the Pedersen commitment is already on-chain and is openable with `(amount, blinding_factor)`. The recipient just needs the sender to disclose those two values + tx_hash.

**Recipient verification math (no new crypto).**

Sender provides reconciliation tuple: `(recipient_addr, otpk_id, amount, blinding_factor, tx_hash)`.

1. Fetch `tx_hash` from chain → confirm `tx.to == recipient_addr` ∧ `tx.amount_mode == AMT_PFS` ∧ `tx.otpk_id == claimed_id`.
2. Fetch original `PUBLISH_OTPK_BATCH` tx from chain (queryable via recipient's own outbound tx history — chain history is immutable even after state compression to `UsedMarker`).
3. Recover `otpk_pk` for `otpk_id` from that batch.
4. Check Pedersen commitment opens: `tx.amount_commitment == amount * G + blinding_factor * H` (on secp256k1).
5. If step 4 passes → sender's claim is cryptographically verified. If step 4 fails → sender is lying; recipient knows the claim is wrong but does not learn the true amount.

**Sender-side persistence requirement.**

Sender wallet must persist for every outbound `AMT_PFS` tx:
```
PfsOutboundRecord {
    recipient_addr:  Address
    otpk_id:         u64
    amount:          u64
    blinding_factor: scalar       // sender already has this — used to build the commitment
    tx_hash:         Hash
    submitted_at:    timestamp
}
```

Most fields the sender wallet keeps anyway (blinding factor is needed to spend the matching commitment-input in any subsequent confidential tx). The additional discipline is just keeping the records keyed for later retrieval. Recommended retention: 7+ years (matches typical tax-record requirements; operator-tunable).

**Reconciliation protocol (off-chain).**

Recipient → sender, arbitrary secure channel (encrypted email, Signal, etc.). Format suggestion (JSON):
```
{
  "type": "DTM-PFS-RECONCILE-RESPONSE",
  "tx_hash": "...",
  "recipient_addr": "...",
  "otpk_id": 5,
  "amount": "1234567",
  "blinding_factor": "..."
}
```

No on-chain registration by default. Operators wanting auditability of reconciliation flows can extend the v2.24 `LOG_AUDIT_ACCESS` tx with a tx-type tag for reconciliation events.

**Trust model (honest documentation).**

- **Sender refuses to provide records** → recipient has no recourse. PFS-mode receipts depend on sender's good-faith record-keeping.
- **Both wallets lost** → amount truly unrecoverable; only the on-chain commitment remains. Equivalent to losing a paper receipt for a cash payment.
- **Sender lies** → commitment verification (step 4) catches it. Recipient learns "sender is dishonest" but not the true amount.
- **Privacy of the request** → recipient asking sender about a specific tx_hash leaks bookkeeping state to the sender. Mitigation: out-of-band channels the sender already knows about; batched reconciliation requests.

**Wallet UX requirements (recipient side).**
- Aggressive backup-discipline warnings at any account creation with `confidential_policy ∈ {MIXED, PFS_ONLY}`.
- New incoming-tx state: `AMT_PFS_UNDECRYPTABLE` distinct from `AMT_PFS_PENDING`. UI surfaces "Request reconciliation from sender" action when an inbound `AMT_PFS` tx exists for which the local `otpk_sk` is missing.
- Reconciliation-tuple importer: parses JSON, runs steps 1-4 above, updates local record with decrypted amount + a flag indicating "reconciled, not locally decrypted."

**Wallet UX requirements (sender side).**
- Outbound `AMT_PFS` persistence: keeps `PfsOutboundRecord` per the retention policy.
- "Generate reconciliation receipt" feature: takes a tx_hash, produces the JSON tuple.

**Auditor-tool extension (composes with PRIV-6 swiss-cheese semantics).**

Two distinct undecryptable cases on a single audit run:
- **Outside observer / auditor with `view_master_sk`**: every `AMT_PFS` tx → `amount: UNDISCLOSED-PFS` (by design).
- **Recipient's own audit run** (auditor running with recipient's keys post-wallet-loss): distinguish `amount: UNDECRYPTABLE-OWN-PFS (request reconciliation)` from `UNDISCLOSED-PFS`. Recipient knows the tx was directed at them (it's in `tx.to`); the chain just denies decryption due to lost `otpk_sk`.

**On-chain footprint: zero.** PRIV-6.1 is entirely operational + wallet-side. No new tx types, no new fields, no state changes. The chain is unchanged from PRIV-6.

#### Q5.2: Publish-cadence padding

**Problem.** PRIV-6 protects payment *amounts* via PFS but leaves payment *cadence* observable. `PUBLISH_OTPK_BATCH` events are on-chain by necessity (sender needs to look up the OTPK; validator enforces uniqueness). An observer sees publish timing and frequency → infers receive rate, correlates with external events. For the high-stakes threat models PRIV-6 implicitly serves (journalism, dissent, regulator-resistant payments), cadence is often as sensitive as amount.

**Mechanism.** Each account with `confidential_policy ∈ {MIXED, PFS_ONLY}` carries an immutable `pfs_padding_cadence` field set at account creation:

```
Account.pfs_padding_cadence: enum {
    NONE,    // no padding — observable cadence
    DAILY,   // 1 batch / day unconditionally
    HOURLY,  // 1 batch / hour unconditionally — high-stakes
}
```

Defaults by `confidential_policy`:
- `AUDITABLE_ONLY` → not applicable (no OTPK stream)
- `MIXED` → `NONE` (cost-sensitive; operator opts in if needed)
- `PFS_ONLY` → `DAILY` (account self-selected high-stakes; default to padding)

Wallet schedules `PUBLISH_OTPK_BATCH` unconditionally at the configured cadence, regardless of actual usage. Used + unused OTPK counts converge to a moving steady-state determined by cadence × batch_size vs. actual receive rate.

**Cost analysis (under chain pruning).**

| Cadence | Per-account chain bandwidth | Per-account annual tx fees | Active-state footprint |
|---|---|---|---|
| NONE | actual usage only | actual usage only | bounded by OTPK stream depth |
| DAILY | ~365KB/year | 365 × per-tx fee | unchanged (old batches prune) |
| HOURLY | ~8.8MB/year | 8760 × per-tx fee | unchanged (old batches prune) |

Padding cost is dominated by recipient-paid tx fees, not network storage — chain pruning amortizes the historical footprint. Recipient explicitly pays for cadence privacy.

**Reconciliation compatibility under pruning.** PRIV-6.1 step 2 (fetch original `PUBLISH_OTPK_BATCH` from chain) requires that historical batches remain queryable. Under chain pruning, recipient wallet must locally cache its own `(otpk_id → otpk_pk)` mappings for batches it published (~130KB/year at 100 receives/year — trivial). Document as wallet-side requirement.

**Anonymity-set thinness (residual).** If only `PFS_ONLY` accounts pad by default, padding cadence itself becomes a signal ("this account chose `PFS_ONLY` → high-stakes"). Privacy gain depends on enough `MIXED` accounts also opting into padding to dilute the signal. Operator guidance for journalism/dissent deployments: encourage `MIXED` accounts to opt into `DAILY` even when not strictly needed.

**Rationale.**
- **No new on-chain mechanism.** Padding is just additional `PUBLISH_OTPK_BATCH` txs scheduled by the wallet. Validator path unchanged.
- **Operator-tunable per account.** Cost-sensitive commercial use opts out; high-stakes use opts in. No global policy decision forced.
- **Default per `confidential_policy` matches threat-model self-selection.** `PFS_ONLY` accounts have already declared high-stakes posture; padding default reflects that.

**Alternatives rejected for cadence protection.**
- *Cohort batching* (multiple recipients mix into shared batches): requires recipient coordination protocol; reduces to mixnet design. Deferred.
- *Decoy batches* (publish real + decoy mixed): adds bandwidth like padding; decoys must be unforgeable to defeat sophisticated analysis. Marginal gain over straight padding.
- *Anonymous bulletin board* (OTPKs posted without recipient binding): requires lookup mechanism; non-trivial design. Deferred.

---

**Alternatives rejected for the PFS mechanism itself (Q5 PRIV-6 broader):**

The OTPK-stream design (PRIV-6) was chosen over four broader alternatives, each rejected for the stated reasons:

- *Forward-secure encryption (FSE / forward-secure HIBE):* Eliminates publish-cadence leak entirely (single PUBLISH at account creation; recipient evolves master per epoch). Rejected for v1.0: canonical Canetti-Halevi-Katz construction requires bilinear pairings (BLS12-381) → conflicts with two-curve-family discipline. Pairing-free lattice-based constructions exist but are bleeding-edge with no production deployments. Revisit when production-grade C99 FSE is available.
- *Puncturable encryption (Green-Miers / Bloom-filter):* Per-tx PFS without on-chain publish events. Rejected for v1.0: research-grade primitive; no production deployments; secret-key state grows with each puncture; false-positive risk in Bloom-filter variants.
- *Stealth addresses (Monero-style):* Combines amount-PFS with recipient-graph-privacy. Mature production deployment (Monero since 2014). Rejected for v2.22: requires full chain scan per recipient per receive (light-client problem is unsolved at scale); architectural change far larger than v2.22's amount-PFS scope; conflicts with v2.24 audit model. Appropriate as v3+ alongside broader graph-privacy work.
- *Out-of-band ephemeral exchange:* Zero on-chain metadata; sender and recipient exchange ephemerals via Signal/encrypted email. Rejected as general solution: requires bidirectional OOB channel per sender-recipient pair; useless for ad-hoc send. May ship as opt-in "established-pair PFS" mode in a future v2.22.x.

## 3. Wire-format extensions

### 3.1 TRANSFER tx changes

Pre-v2.22:
```
Transaction { from, to, amount: u64, fee, nonce, payload, sig }
```

Post-v2.22 (confidential mode):
```
Transaction {
    from, to, fee, nonce, payload, sig,
    amount_mode:          enum { AMT_AUDITABLE, AMT_PFS }  // Q5 PRIV-6
    otpk_id:              Option<u64>      // required iff amount_mode == AMT_PFS
    amount_commitment:    secp256k1_point  // Pedersen C = aG + bH on secp256k1 (33 bytes compressed)
    amount_range_proof:   bytes            // Bulletproof via libsecp256k1-zkp, ~600-1500B
    sender_eph_pk:        secp256k1_point  // secp256k1 ephemeral ECDH pubkey (33 bytes compressed)
    amount_ct:            bytes            // XChaCha20-Poly1305 ciphertext over amount_bytes (8 bytes plaintext + 16 byte tag)
    balance_proof:        bytes            // sum(inputs) = sum(outputs) proof on secp256k1
}
```

Backward-compat: A `tx_type: u8` field distinguishes clear-amount (legacy `amount: u64`) from confidential-amount (commitment + proofs). Pre-flag-day: both types valid; post-flag-day: confidential-amount required for accounts that have published a `view_master_pk`.

### 3.2 New on-chain fields

```
Account {
    ...existing fields...,
    view_master_pk:       secp256k1_point        // optional secp256k1 pubkey (33 bytes compressed); absent = clear-amount only
    audit_view_master_pk: secp256k1_point        // optional v2.24 field (MODERN); FIPS deployments use P-256 point
    confidential_policy:  enum {                 // Q5 PRIV-6; immutable, set at account creation
        AUDITABLE_ONLY,                          //   reject inbound AMT_PFS; PUBLISH_OTPK_BATCH disallowed
        PFS_ONLY,                                //   reject inbound AMT_AUDITABLE; initial OTPK batch required
        MIXED,                                   //   both modes; sender picks per tx
    }
    otpk_stream:          Vec<OtpkEntry>         // Q5 PRIV-6; populated only if policy != AUDITABLE_ONLY
                                                 //   Unused{otpk_id, otpk_pk} (43B) | UsedMarker{hash_32} (32B)
    pfs_padding_cadence:  enum {                 // Q5.2 PRIV-6.2; immutable; N/A if policy == AUDITABLE_ONLY
        NONE,                                    //   no padding (default for MIXED)
        DAILY,                                   //   1 PUBLISH_OTPK_BATCH/day unconditionally (default for PFS_ONLY)
        HOURLY,                                  //   1 PUBLISH_OTPK_BATCH/hour unconditionally
    }
}
```

### 3.3 New tx types

- `PUBLISH_VIEW_KEY` — sets `view_master_pk` for an account that doesn't have one. One-time per account; subsequent rotations via `ROTATE_VIEW_MASTER`.
- `ROTATE_VIEW_MASTER` — emergency rotation of `view_master_sk` (compromise recovery). Carries new pubkey + old-key signature; analogous to v2.26 `ROTATE_KEY`.
- `ROTATE_AUDIT_KEY` (v2.24) — rotation of `audit_view_master_pk`.
- `LOG_AUDIT_ACCESS` (v2.24, optional) — on-chain record of disclosure events.
- `PUBLISH_OTPK_BATCH` (Q5 PRIV-6) — recipient publishes a batch of 32 fresh OTPKs. Allowed only when `confidential_policy != AUDITABLE_ONLY`. Sender wallet auto-replenishes when recipient's unused OTPK count drops below 8.

### 3.4 Block format

No block-format changes. All v2.22 additions ride in the existing tx envelope.

## 4. Implementation work units

### 4.1 Bulletproofs over secp256k1 (~10 days)

- Vendor `libsecp256k1` + `libsecp256k1-zkp` (Blockstream / Mimblewimble-Grin) as `src/crypto/secp256k1/` + `src/crypto/secp256k1_zkp/` per CRYPTO-C99-SPEC.md §2.Q3. Pinned upstream commit; constant-time build flags.
- Enable Bulletproofs module + ECDH at vendor-build time (per CRYPTO-C99-SPEC.md §3.x).
- Aggregation support: single proof covers N range claims (native to upstream).
- Verification side: batch-verify multiple proofs in one pass.
- C++ wrapper for integration with `Transaction` apply path.
- Test vectors: upstream reference vectors from libsecp256k1-zkp (already shipped by Liquid / Grin).

### 4.2 Pedersen commitment integration (~2-3 weeks)

- New `Amount` type wrapping `Commitment | u64` (confidential vs clear).
- `Transaction` extended with commitment + proof fields.
- Apply path: balance-conservation check via commitment algebra (sum(input C) == sum(output C) iff blinding factors cancel).
- Validator: range-proof check + balance-proof check per confidential tx.
- Mempool: drop txs with invalid proofs at admission (analogous to S-002 sig-verify gate).

### 4.3 View-master + per-epoch derivation (~2 weeks)

- Account state extended with `view_master_pk`.
- Wallet-side `view_master_sk` storage (encrypted via existing envelope).
- HKDF derivation library (libsodium primitives, no new crypto).
- `PUBLISH_VIEW_KEY` tx type + apply path.

### 4.4 Sender-recipient ECDH handshake (~1 week)

- secp256k1 ECDH primitive (already in libsecp256k1 — `secp256k1_ecdh`).
- XChaCha20-Poly1305 AEAD over amount bytes (libsodium, already in tree).
- Sender-side: ephemeral secp256k1 key gen + ECDH + HKDF + AEAD encrypt.
- Recipient-side: ECDH + HKDF + AEAD decrypt + commitment-blinding verify.
- Wallet CLI: `determ transfer --confidential <addr> <amount>`.

### 4.5 `ROTATE_VIEW_MASTER` tx (~1 week)

- New tx type analogous to v2.26 `ROTATE_KEY`.
- Validator: verify old-key signature over `(account_addr, old_pubkey, new_pubkey, effective_height)`.
- Apply: update account's `view_master_pk` at `effective_height`.
- Rotation cooldown (one rotation per N blocks per account).

### 4.6a Per-tx PFS infrastructure (Q5 PRIV-6 + PRIV-6.1, ~4.5-6 weeks)

**PRIV-6 chain + wallet core (~3-4 weeks):**
- `confidential_policy` enum on Account (set at creation; validator enforces immutability).
- `otpk_stream` Vec state + `PUBLISH_OTPK_BATCH` tx (batch of 32; ~1KB payload).
- Sender wallet: OTPK discovery via chain query; next-unused selection; race-loss retry.
- Recipient wallet: OTPK batch generation; privkey storage; delete-after-use lifecycle; replenishment trigger at threshold 8.
- Apply path: atomic `Unused → UsedMarker{SHA256(id || pk)}` replacement; policy enforcement (rules 4-5).
- Auditor tool extension: swiss-cheese report rendering (AMT_PFS txs as `amount: UNDISCLOSED-PFS`; report includes auditable/PFS ratio).

**PRIV-6.1 wallet-loss reconciliation (~1.5-2 weeks):**
- Sender wallet: `PfsOutboundRecord` persistence (blinding factor + amount + tx_hash); "Generate reconciliation receipt" UI feature (~3-4 days; mostly UI since wallet already keeps blinding factors).
- Recipient wallet: `AMT_PFS_UNDECRYPTABLE` incoming-tx state; reconciliation-tuple importer; verification logic (steps 1-4); reconciled-record marking (~1 week).
- Auditor tool: distinguish `UNDECRYPTABLE-OWN-PFS` vs `UNDISCLOSED-PFS` (~2-3 days).
- Tests + docs (~3-4 days).
- **No chain changes** — PRIV-6.1 is operational + wallet-side only.

**PRIV-6.2 cadence padding (~3-5 days):**
- `pfs_padding_cadence` enum field on Account (immutable; set at creation alongside `confidential_policy`).
- Wallet padding scheduler: schedules unconditional `PUBLISH_OTPK_BATCH` at configured cadence (DAILY/HOURLY); skips if `NONE`.
- Recipient wallet (PRIV-6.1 cache requirement under pruning): local `(otpk_id → otpk_pk)` cache for own published batches (~130KB/year per account).
- Tests covering steady-state stream depth under padding + actual usage.
- **No chain validator changes** — padding is just additional ordinary `PUBLISH_OTPK_BATCH` txs.

### 4.6 v2.24 audit-mode tooling + integration (~2-3 weeks)

- `audit_view_master_pk` field on Account.
- `ROTATE_AUDIT_KEY` tx (analogous to v2.26).
- Audit-mode RPC: `audit_decrypt_tx(tx_hash, vk_epoch_n) → amount`.
- `LOG_AUDIT_ACCESS` tx (optional).
- Reference auditor tool: takes view-master-sk or per-epoch keys, decrypts all in-scope txs, produces compliance report (CSV / JSON).

### 4.7 Tests + docs (~2 weeks)

- Range-proof unit tests (boundary, max, zero, malformed).
- End-to-end confidential transfer test (sender + recipient + auditor).
- Audit-mode integration test (operator deploys with `audit_view_master_pk` set; auditor reads via standing key).
- Migration test: clear-amount → confidential-amount account transition.
- Documentation: `PROTOCOL.md` confidential-amount section, wallet UX docs.

### 4.8 Wire-format stamp + genesis rule (~1-2 days) — REVISED 2026-06-05

**Originally (~1 week):** flag-day migration tooling — bump tx-format wire version + flag-day genesis field + dual-decode validator path (pre-flag-day accepted clear-amount; post-flag-day required confidential for accounts with `view_master_pk`).

**Revised under no-migrations:** the flag-day machinery is dead code per memory `dlt-no-migrations-constraint` (v1.0 ships with confidential-amount rule active from genesis; no pre/post-flag-day boundary exists). What remains:

- Wire-format version stamp on tx envelope for forward-compat (~1 day) — does NOT enable migration; just identifies the wire version for future additive-only changes
- Genesis-pinned validation rule: accounts with `view_master_pk` set MUST use confidential amounts (~few hours; trivial validator gate)

**Removed:** dual-decode path (~3-4 days dead), flag-day activation logic (~1-2 days dead), migration tooling for operator coordination (~1-2 days dead).

**Net effort reduction:** ~5-7 days deleted from Bundle 3 critical path.

## 6. Risks and rollback plan

**Risk: Bulletproofs integration bugs.** libsecp256k1-zkp has been running Bulletproofs in production since 2018-2019 (Liquid sidechain, Grin) over the same secp256k1 curve, so the upstream code is well-exercised — but integration into Determ's `Transaction` apply path introduces wrapper-surface risk.

*Mitigation.* Pin libsecp256k1-zkp to a specific upstream commit. Cross-check against upstream's reference test vectors directly. Bug-bounty period before flag-day. Conservative fallback: ship v2.22 with clear-amount default and confidential-amount opt-in until audit clears.

**Risk: Master view-key compromise exposes account history.** Compromise of `view_master_sk` reveals all past and future amounts for the account until `ROTATE_VIEW_MASTER` fires.

*Mitigation.* Wallet-side hardware-isolation guidance for `view_master_sk` (HSM, hardware wallet, paper backup with offline storage). Recommend per-purpose accounts (one account per relationship — limits blast radius). Rotation cooldown prevents griefing but doesn't prevent legitimate emergency rotation.

**Risk: Range-proof verification cost on validator.** ~600-1500 bytes per proof + O(N log N) verification per N outputs. At high tx rate, validator CPU could become the bottleneck.

*Mitigation.* Aggregation: batch-verify multiple range proofs in one pass. Profile validator under realistic load. If verification cost is prohibitive, ship per-account opt-in rather than mandatory (operators choose).

**Risk: v2.24 audit-mode disclosure leaks beyond intended scope.** Auditor with master key has permanent unrevocable access; per-epoch keys only revocable by waiting out the epoch.

*Mitigation.* `ROTATE_AUDIT_KEY` tx for permanent-access revocation. Operator-side policy guidance: prefer per-epoch keys for external auditors; master keys only for trusted internal compliance.

**Risk: Hybrid clear+confidential accounts create privacy leakage via traffic analysis.** Accounts using confidential mode are observable on-chain (they have `view_master_pk` set). An observer can correlate confidential-mode adoption with account behavior.

*Mitigation.* Accepted residual. Full traffic-analysis resistance requires graph-privacy (stealth addresses, ring signatures) which is out of v2.22's scope. Document for operators.

**Risk: PFS recipient wallet loss leaves amounts undecryptable locally (Q5 PRIV-6).** Lost `otpk_sk` means inbound `AMT_PFS` amount is unrecoverable from chain alone. Funds remain (balance commitment is up-to-date), but recipient cannot reconcile the amount.

*Mitigation.* PRIV-6.1 reconciliation: sender wallet retains `PfsOutboundRecord` (blinding factor + amount + tx_hash) per operator-policy retention; recipient requests off-chain reconciliation tuple; cryptographic verification via Pedersen commitment opening (no new crypto). Residual: sender refusal or dual wallet loss leaves amount unrecoverable — equivalent to losing a paper receipt for cash. Document for operators choosing `MIXED` or `PFS_ONLY` accounts; require aggressive backup-discipline UX at account creation.

**Risk: Publish-cadence leak signals receive-rate to observers (Q5 PRIV-6).** `PUBLISH_OTPK_BATCH` events are on-chain; their timing/frequency leaks recipient's PFS-receive cadence — material for high-stakes threat models (journalism, dissent, regulator-resistant payments).

*Mitigation.* PRIV-6.2 cadence padding: per-account `pfs_padding_cadence` field (NONE / DAILY / HOURLY); wallet schedules unconditional `PUBLISH_OTPK_BATCH` at configured rate. Default DAILY for `PFS_ONLY` accounts; default NONE for `MIXED` (operator opts in). Recipient pays tx-fee cost for padding privacy. **Residual: padding adoption thinness** — if only `PFS_ONLY` accounts pad, padding itself becomes a high-stakes signal; privacy depends on enough `MIXED` accounts also opting into DAILY to dilute the signal. Document for operators serving high-stakes deployments.

**Rollback plan.** If v2.22 ships with a bug discovered after flag-day:
1. Governance pause (v2 mechanism) freezes new block production.
2. Disable confidential-amount tx type via flag-day reversal.
3. Existing confidential-amount commitments remain in chain history but are unrecoverable without view keys (matches Zcash/Monero shielded-pool semantics).
4. Re-attempt v2.22 in a future release after bug fix.

## 7. What this enables downstream

v2.22 with Option C view-key infrastructure enables:

- **v2.24 audit hooks** — concrete disclosure mechanism (master or per-epoch keys) instead of an abstract "auditor decrypts" claim.
- **v2.25 DSSO** — DSSO assertions can carry encrypted account-history summaries scoped to the relying party (using per-tx ephemeral DH against the RP's view key). Not v2.22's scope, but the infrastructure is reusable.
- **Regulated gambling deployments** — operators ship with `audit_view_master_pk` set to their licensing-jurisdiction regulator's pubkey; regulator gets standing audit access; players retain confidentiality of amounts from each other.
- **B2B settlement** — invoices, payroll, vendor payments where amounts are commercially sensitive but compliance disclosure is required.

The view-key infrastructure described here is **shared foundation** for Determ's privacy + compliance layer, not a v2.22-only investment.
