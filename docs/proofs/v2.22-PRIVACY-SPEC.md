# v2.22 confidential transactions — design specification

**Status:** specification only. No code. Resolves the 4 interlinked design questions for v2.22 (view-key rotation cadence, range-proof choice, sender-recipient amount handshake, audit integration with v2.24). Implementation should not begin until this document's design choices are reviewed and committed.

**Companion documents:**
- `V2-DESIGN.md` v2.22 — v2 design space entry; motivation
- `V2-DESIGN.md` v2.24 — audit / compliance hooks (composes with this spec)
- `V2-DESIGN.md` v2.26 — key rotation (analogous `ROTATE_*` pattern)
- `v2.10-DKG-SPEC.md` — curve25519-family DKG infrastructure (FROST-Ed25519) this spec builds on
- `F2-SPEC.md` — analogous structure for v2.7

## 1. Scope

**Profile applicability.** Confidential transactions are available only in deployments using the **MODERN** cryptographic profile (per `CRYPTO-C99-SPEC.md` §2.Q10). The **FIPS** profile (bundled with both the `tactical` and `cluster` timing profiles) **does NOT support confidential transactions** because no FIPS-validated zero-knowledge range proof construction exists.

| Profile | Confidential tx availability |
|---|---|
| `cluster` (FIPS — in-house enterprise / financial services / regulated) | ❌ Not available; use clear amounts + v2.24 audit hooks |
| `tactical` (FIPS — military / defense / embedded) | ❌ Not available; use clear amounts |
| `web` (MODERN — web-scale, commercial single-cluster) | ✅ Available |
| `regional` (MODERN — regional sharded) | ✅ Available |
| `global` (MODERN — global-scale) | ✅ Available |

FIPS-profile deployments use clear-amount TRANSFER txs exclusively + v2.24 audit hooks for regulator access (audit access controlled at the chain-read layer rather than via view-key cryptographic disclosure). This is a structural trade-off documented as accepted for FIPS compliance.

For MODERN-profile deployments (web, regional, global), v2.22 confidential transactions ship as documented below. Financial services / banking deployments that NEED confidential amounts and accept non-FIPS cryptography should use `web` or `regional` profiles instead of `cluster`.

This spec covers ONLY confidential-amount transactions and their view-key infrastructure. It does NOT cover:
- Transaction-graph privacy (sender / recipient unlinkability) — out of scope for v2.22; the chain still records `tx.from` and `tx.to` publicly. Adding graph privacy is a v2.x follow-on that requires deeper architectural changes (ring signatures, stealth addresses, or zk-SNARK shielded pools).
- Smart-contract privacy — Determ has no contract VM; computation privacy is delivered via the God Stack pattern (zk-VM L2), out of v2.22's scope.
- Bridge interoperability with other confidential-amount chains (Zcash, Monero) — out of scope; v2.23's bridge ships clear-amount cross-chain transfers.

The artifacts produced by v2.22:
- **Pedersen-commitment amounts** in TRANSFER (and any future confidential-amount tx type)
- **Bulletproof range proofs** ensuring committed amounts are in `[0, 2^64)` (no underflow)
- **Balance-conservation proofs** ensuring sum(inputs) = sum(outputs) without revealing values
- **Per-epoch view-key derivation** for amount recovery by recipient, sender, and designated auditors
- **Sender-recipient ephemeral-DH handshake** for per-tx amount-decryption key derivation
- **Audit-mode disclosure tooling** (composes with v2.24)

## 2. Design decisions

### Q1: View-key rotation cadence

**Decision: per-epoch automatic rotation via HKDF hierarchical derivation (Option C).**

Each account has a long-term `view_master` keypair. Per-epoch view keys derive deterministically:

```
vk_epoch_n = HKDF(view_master_sk, "VK" || chain_id || account_addr || epoch_n)
pk_epoch_n = vk_epoch_n · G   (point on ristretto255)
```

Where `epoch_n` is the epoch number at tx-apply time. The recipient (or auditor with view_master) recomputes the same `vk_epoch_n` to decrypt amounts within epoch n.

**Rationale.**
- **Bounded exposure per epoch.** A compromised `vk_epoch_n` exposes only epoch n's amounts; subsequent epochs unaffected by HKDF forward secrecy.
- **Zero on-chain cost for rotation.** Derivation is purely client-side. No `ROTATE_VIEW_KEY` tx, no gas burn.
- **Maps to regulator cadence.** Quarterly/annual audits → grant per-epoch keys for the audit window only. No leakage of out-of-scope periods.
- **No rotation discipline required.** Happens automatically at every epoch boundary; eliminates "forgot to rotate" OPSEC failure.
- **Composes with v2.10's epoch infrastructure.** Epoch boundaries are already a protocol sync point.

**Alternatives rejected:**
- *Genesis-pinned (Option A):* Catastrophic forward-secrecy failure — compromise = permanent exposure for account lifetime.
- *Per-account mutable via tx (Option B):* Burden-heavy (user rotation discipline; gas cost); audit re-disclosure friction.
- *Per-tx ephemeral (Option D):* Auditor disclosure becomes expensive (per-tx key derivation for account-level audits typical in regulation).
- *Hybrid master + per-tx (Option E):* Over-engineered for v2.22's scope; could be a v2.x follow-on if specific deployments demand it.

**Master compromise mitigation.** A `ROTATE_VIEW_MASTER` tx (analogous to v2.26 `ROTATE_KEY`) allows one-time recovery from catastrophic master compromise. Not for routine rotation (which Option C handles via HKDF) — only for emergency.

**Optional hierarchical HKDF for stronger forward secrecy (recommended).** The flat per-epoch derivation `vk_epoch_n = HKDF(view_master, "VK" || chain_id || account_addr || epoch_n)` gives epoch-level PFS — a compromised epoch key reveals only that epoch's amounts. But compromise of `view_master` itself exposes ALL past and future per-epoch keys. To shrink the master-compromise blast radius from "all history" to "one period of history," derive via a hierarchical tree:

```
view_master_sk   (long-term, hardware-isolated; never network-touched)
    │
    └─ period_sk_p = HKDF(view_master_sk, "PERIOD" || chain_id || account_addr || period_p)
              │
              └─ vk_epoch_n = HKDF(period_sk_p, "VK" || epoch_n)
                              for epoch_n in [p * period_blocks, (p+1) * period_blocks)
```

Where `period_blocks` is a genesis-pinned parameter (default: `view_master_period_blocks = 10000` ≈ 100 epochs at `epoch_blocks=100`). Each period derives a `period_sk_p` from the master; per-epoch keys then derive from `period_sk_p`. The user (or auditor with the master) recomputes any `vk_epoch_n` via the two-level derivation.

**What this delivers:**
- **Period-level PFS against master compromise** — a compromised `view_master_sk` at time T exposes only the *current and future* periods' keys. *Past* periods' derived keys are not recoverable from a stolen master alone — the user must have rotated to a new `view_master_sk` between periods via `ROTATE_VIEW_MASTER`.

Wait — that last point isn't quite right. Hierarchical HKDF alone doesn't give past-period PFS against master compromise (HKDF is one-way derivation, not one-way evolution). The master can derive any past period's key just as easily as future ones.

**Honest re-statement of what hierarchical HKDF actually delivers:**

The hierarchical structure does NOT close the master-compromise gap on its own. It enables a separate operational protection: **the user can keep `view_master_sk` in cold/hardware-isolated storage and only derive `period_sk_p` for the current period for hot wallet use.** A compromise of the hot-wallet `period_sk_p` then exposes only that period — not all history. The master stays cold; an attacker compromising the hot wallet doesn't get the master.

This is a **cold/hot key separation** pattern, made structurally possible by the hierarchical derivation. Without the hierarchy, the user must keep `view_master_sk` available for every per-epoch derivation; with the hierarchy, the user can derive one `period_sk_p` per period from cold storage, then use it from hot storage for `period_blocks` epochs before refreshing.

**Recommended for high-stakes deployments** (regulated gambling, B2B payment, healthcare) where hardware-isolation of the master is operationally feasible. Not necessary for low-stakes deployments where the user must keep the master online anyway.

**Genesis fields (when adopted):**
- `view_master_period_blocks: u64` — 0 = flat derivation (Option C baseline); >0 = hierarchical derivation with this period length

**Cost:** ~50 LOC. Adds one HKDF level to the derivation; one new genesis field; documentation update to v2.22 + v2.24 (auditor disclosure can now be at master, period, or epoch granularity). Compatible with the rest of v2.22's design.

### Q2: Range-proof construction

**Decision: Bulletproofs over secp256k1 via libsecp256k1-zkp.**

**Rationale.**
- **Most-production-tested C99 Bulletproof implementation available.** libsecp256k1-zkp (maintained by Blockstream / Mimblewimble-Grin) has been running Bulletproofs in production since 2018-2019 in Liquid (Blockstream's Bitcoin sidechain) and Grin. ~5+ years of production exposure of Bulletproofs-over-secp256k1 specifically.
- **C99 native, libsodium-independent.** Aligns with the CRYPTO-C99-SPEC.md strategy of full libsodium removal. libsecp256k1 + libsecp256k1-zkp ship as vendored C99 modules. No external dependency.
- **Bitcoin-grade audit pedigree.** libsecp256k1 underlies Bitcoin Core (~$1T+ secured); audited multiple times; constant-time discipline reviewed by cryptographers including Pieter Wuille, Greg Maxwell, Jonas Nick.
- **Prime-order group natively.** secp256k1 has cofactor 1 — no cofactor-clearing complications. Same property as ristretto255.
- **No trusted setup.** Bulletproofs require no trusted setup ceremony. Matches Determ's mutual-distrust posture.
- **Aggregation friendly.** Bulletproofs aggregate naturally — a single proof can cover N range claims with O(log N) overhead. Critical for TRANSFER txs with multiple outputs.

**Trade-offs acknowledged.**
- **Adds secp256k1 as a second curve family.** Determ's cryptographic stack becomes: SHA-256 + curve25519 family (Ed25519 sigs, X25519 KX, FROST-Ed25519 threshold) + secp256k1 family (Bulletproofs, OPRF, prime-order needs). Three curves total; two underlying mathematical families. Documented as deliberate trade-off in `CRYPTO-C99-SPEC.md` §2.Q1.
- Proof size: ~600-1500 bytes per output (acceptable; <0.5% of typical block bandwidth at tactical profile).
- Verification cost: O(N log N) for N outputs. Aggregation helps; cost is bounded.
- Larger than Plonk/SNARK proofs but no trusted setup — the right trade for Determ.

**Alternatives considered:**
- *Bulletproofs over curve25519/ristretto255 (dalek-cryptography reference):* Earlier draft chose this. Reverted because the only mature C99 ristretto255 implementation is in libsodium; vendoring libsodium-derived ristretto255 source conflicts with the libsodium-removal goal in CRYPTO-C99-SPEC.md. libsecp256k1-zkp delivers Bulletproofs without any libsodium dependence.
- *Bulletproofs over BLS12-381 (blst):* Adds pairing-friendly curve to the stack; previously rejected for "two primitives" reasons. Pairings not needed (God Stack pattern offloads pairing-based ZK to external L2).
- *BBS+ signatures over BLS12-381:* Designed for selective-disclosure signatures, not range proofs. Wrong primitive.

**Library:** libsecp256k1 + libsecp256k1-zkp, vendored as `src/crypto/secp256k1/` + `src/crypto/secp256k1_zkp/` per CRYPTO-C99-SPEC.md §2.Q3. Pinned upstream commit. C99 native; no external runtime dependency. ~10 days integration (vs ~2-3 weeks for ristretto255 alternative via libsodium vendoring).

### Q3: Sender-recipient amount handshake

**Decision: ephemeral DH on X25519 (curve25519 family) against recipient's published `view_master_pk`, then HKDF derivation with epoch label.**

**Protocol.**
1. Recipient publishes `view_master_pk` (X25519 public key) on-chain at account creation (in REGISTER tx payload extension, or via a one-time `PUBLISH_VIEW_KEY` tx for existing accounts).
2. Sender generates ephemeral `eph_sk` (X25519 secret); computes `eph_pk = X25519_scalarmult_base(eph_sk)`.
3. Sender computes shared secret: `ss = X25519_dh(eph_sk, view_master_pk)`.
4. Sender derives per-tx amount-encryption key: `aek = HKDF-SHA-256(ss, "AMT" || epoch_n || tx_hash)`.
5. Sender encrypts the amount: `amount_ct = XChaCha20-Poly1305(aek, amount_bytes, AAD = "TX-AMT" || tx_hash)`.
6. Tx carries: `commitment = aG + bH` (Pedersen on **secp256k1** for compatibility with Bulletproofs per Q2), `range_proof`, `eph_pk` (X25519, 32 bytes), `amount_ct`.
7. Recipient (or auditor with `view_master_sk`) recomputes `ss = X25519_dh(view_master_sk, eph_pk)`, derives `aek`, decrypts `amount_ct`, verifies it matches `commitment` via blinding factor check.

**Curve choice rationale.**
- **X25519 for DH (curve25519 family)** — standard DH primitive; mature C99 implementations (curve25519-donna); aligns with gossip-handshake KX which also uses X25519. Per `CRYPTO-C99-SPEC.md` §2.Q1.
- **secp256k1 for Pedersen commitment (per Q2 above)** — Bulletproofs requires the commitment to be on the same curve as the range proof. Two-curve protocol: DH on X25519 (for amount-encryption-key derivation), commitment on secp256k1 (for range-proof compatibility). DH curve choice is independent of commitment curve choice — DH only needs a shared secret; AEAD encryption of the amount is curve-independent.

**Rationale.**
- **No bidirectional handshake.** Sender doesn't need recipient interaction; one-shot tx submission.
- **Forward secrecy via ephemeral.** Compromised `eph_sk` (sender-side) doesn't affect other txs.
- **Recipient anonymity preserved.** Only `view_master_pk` is public; tx-level decryption requires the sender's `eph_pk` (which is per-tx).
- **Audit compatibility.** Auditor with `view_master_sk` can decrypt any tx to/from the account using the same handshake (auditor doesn't need to be in the loop at tx-time).

**Library:** libsodium already in tree (used by wallet `envelope.cpp` and Theme-7 direct-to-DApp pattern). XChaCha20-Poly1305 for AEAD. ristretto255 DH via libsodium's `crypto_scalarmult_ristretto255` primitives — no new dependency.

### Q4: Audit integration (composes with v2.24)

**Decision: dual-mode disclosure — `view_master_sk` (full access) or per-epoch `vk_epoch_n` keys (scoped access).**

**Disclosure modes.**

| Mode | Auditor receives | Scope | Use case |
|---|---|---|---|
| Master | `view_master_sk` | All amounts to/from this account, ever | In-house compliance officer; permanent audit relationship |
| Per-epoch | `vk_epoch_n` for specific n | Amounts in epoch n only | External regulator; bounded audit period (quarterly, annual) |

Account holder discloses via off-chain channel (encrypted email, secure messaging, paper). No on-chain disclosure required — keeps audit relationships private.

**v2.24 integration.** `v2.24` adds an optional `audit_view_master_pk` field to the account state. If set, the named auditor has standing pre-authorization to derive view keys (via DH against the auditor's published pubkey). Rotation via a `ROTATE_AUDIT_KEY` tx (analogous to v2.26 `ROTATE_KEY`). Used by deployments where regulatory access is permanent (banks, licensed gambling operators) and the operator wants to bake the audit relationship into the account.

**On-chain audit trail.** When an auditor disclosure happens off-chain, there's no on-chain record by default. For deployments requiring auditable audit access (regulators wanting to know "who audited what when"), a `LOG_AUDIT_ACCESS` tx records the disclosure event. Optional — operator policy.

**Rationale.**
- **Map to regulatory reality.** Real regulator workflows: "Provide all 2026 Q3 transactions for account X." Per-epoch keys deliver exactly this. Real in-house compliance: "Our CFO needs ongoing access." Master key delivers that.
- **No new cryptographic primitive.** Both modes use the Q1 HKDF derivation.
- **Composes with v2.26.** ROTATE_AUDIT_KEY mirrors ROTATE_KEY; rotation cooldown applies; no new mechanism.

## 3. Wire-format extensions

### 3.1 TRANSFER tx changes

Pre-v2.22:
```
Transaction { from, to, amount: u64, fee, nonce, payload, sig }
```

Post-v2.22 (confidential mode):
```
Transaction {
    from, to, fee, nonce, payload, sig,
    amount_commitment:    secp256k1_point  // Pedersen C = aG + bH on secp256k1 (33 bytes compressed)
    amount_range_proof:   bytes            // Bulletproof via libsecp256k1-zkp, ~600-1500B
    sender_eph_pk:        x25519_point     // X25519 ephemeral DH pubkey (32 bytes)
    amount_ct:            bytes            // XChaCha20-Poly1305 ciphertext over amount_bytes (8 bytes plaintext + 16 byte tag)
    balance_proof:        bytes            // sum(inputs) = sum(outputs) proof on secp256k1
}
```

Backward-compat: A `tx_type: u8` field distinguishes clear-amount (legacy `amount: u64`) from confidential-amount (commitment + proofs). Pre-flag-day: both types valid; post-flag-day: confidential-amount required for accounts that have published a `view_master_pk`.

### 3.2 New on-chain fields

```
Account {
    ...existing fields...,
    view_master_pk:       x25519_point    // optional X25519 pubkey (32 bytes); absent = clear-amount only
    audit_view_master_pk: x25519_point    // optional v2.24 field; absent = no standing auditor
}
```

### 3.3 New tx types

- `PUBLISH_VIEW_KEY` — sets `view_master_pk` for an account that doesn't have one. One-time per account; subsequent rotations via `ROTATE_VIEW_MASTER`.
- `ROTATE_VIEW_MASTER` — emergency rotation of `view_master_sk` (compromise recovery). Carries new pubkey + old-key signature; analogous to v2.26 `ROTATE_KEY`.
- `ROTATE_AUDIT_KEY` (v2.24) — rotation of `audit_view_master_pk`.
- `LOG_AUDIT_ACCESS` (v2.24, optional) — on-chain record of disclosure events.

### 3.4 Block format

No block-format changes. All v2.22 additions ride in the existing tx envelope.

## 4. Implementation work units

### 4.1 Bulletproofs over curve25519 (~2-3 weeks)

- Vendor `dalek-cryptography/bulletproofs` directly (Rust) or use a C/C++ port. Pinned version, constant-time build flags.
- Aggregation support: single proof covers N range claims (native to upstream).
- Verification side: batch-verify multiple proofs in one pass.
- C++ wrapper for integration with `Transaction` apply path.
- Test vectors: use the upstream reference vectors directly — no curve translation needed.

### 4.2 Pedersen commitment integration (~2-3 weeks)

- New `Amount` type wrapping `Commitment | u64` (confidential vs clear).
- `Transaction` extended with commitment + proof fields.
- Apply path: balance-conservation check via commitment algebra (sum(input C) == sum(output C) iff blinding factors cancel).
- Validator: range-proof check + balance-proof check per confidential tx.
- Mempool: drop txs with invalid proofs at admission (analogous to S-002 sig-verify gate).

### 4.3 View-master + per-epoch derivation (~2 weeks)

- Account state extended with `view_master_pk`.
- Wallet-side `view_master_sk` storage (encrypted via existing envelope).
- HKDF derivation library (libsodium primitives, no new crypto).
- `PUBLISH_VIEW_KEY` tx type + apply path.

### 4.4 Sender-recipient DH handshake (~1 week)

- libsodium sealed-box integration for AEAD over amount bytes.
- Sender-side: ephemeral key gen + DH + HKDF + AEAD encrypt.
- Recipient-side: DH + HKDF + AEAD decrypt + commitment-blinding verify.
- Wallet CLI: `determ transfer --confidential <addr> <amount>`.

### 4.5 `ROTATE_VIEW_MASTER` tx (~1 week)

- New tx type analogous to v2.26 `ROTATE_KEY`.
- Validator: verify old-key signature over `(account_addr, old_pubkey, new_pubkey, effective_height)`.
- Apply: update account's `view_master_pk` at `effective_height`.
- Rotation cooldown (one rotation per N blocks per account).

### 4.6 v2.24 audit-mode tooling + integration (~2-3 weeks)

- `audit_view_master_pk` field on Account.
- `ROTATE_AUDIT_KEY` tx (analogous to v2.26).
- Audit-mode RPC: `audit_decrypt_tx(tx_hash, vk_epoch_n) → amount`.
- `LOG_AUDIT_ACCESS` tx (optional).
- Reference auditor tool: takes view-master-sk or per-epoch keys, decrypts all in-scope txs, produces compliance report (CSV / JSON).

### 4.7 Tests + docs (~2 weeks)

- Range-proof unit tests (boundary, max, zero, malformed).
- End-to-end confidential transfer test (sender + recipient + auditor).
- Audit-mode integration test (operator deploys with `audit_view_master_pk` set; auditor reads via standing key).
- Migration test: clear-amount → confidential-amount account transition.
- Documentation: `PROTOCOL.md` confidential-amount section, wallet UX docs.

### 4.8 Migration / wire-version bump (~1 week)

- Bump tx-format wire version.
- Flag-day genesis field: `confidential_amounts_required_for_accounts_with_view_master_pk: bool`.
- Pre-flag-day: clear-amount accepted for all accounts.
- Post-flag-day: accounts with `view_master_pk` set must use confidential amounts.

## 6. Risks and rollback plan

**Risk: Bulletproofs integration bugs.** dalek-cryptography/bulletproofs is the most-deployed reference impl (Monero, Mimblewimble/Grin) but C++ integration via FFI or port introduces integration-surface risk.

*Mitigation.* Use upstream Rust crate via FFI or audited C/C++ port. Pin to specific version. Cross-check against upstream's reference test vectors directly. Bug-bounty period before flag-day. Conservative fallback: ship v2.22 with clear-amount default and confidential-amount opt-in until audit clears.

**Risk: Master view-key compromise exposes account history.** Compromise of `view_master_sk` reveals all past and future amounts for the account until `ROTATE_VIEW_MASTER` fires.

*Mitigation.* Wallet-side hardware-isolation guidance for `view_master_sk` (HSM, hardware wallet, paper backup with offline storage). Recommend per-purpose accounts (one account per relationship — limits blast radius). Rotation cooldown prevents griefing but doesn't prevent legitimate emergency rotation.

**Risk: Range-proof verification cost on validator.** ~600-1500 bytes per proof + O(N log N) verification per N outputs. At high tx rate, validator CPU could become the bottleneck.

*Mitigation.* Aggregation: batch-verify multiple range proofs in one pass. Profile validator under realistic load. If verification cost is prohibitive, ship per-account opt-in rather than mandatory (operators choose).

**Risk: v2.24 audit-mode disclosure leaks beyond intended scope.** Auditor with master key has permanent unrevocable access; per-epoch keys only revocable by waiting out the epoch.

*Mitigation.* `ROTATE_AUDIT_KEY` tx for permanent-access revocation. Operator-side policy guidance: prefer per-epoch keys for external auditors; master keys only for trusted internal compliance.

**Risk: Hybrid clear+confidential accounts create privacy leakage via traffic analysis.** Accounts using confidential mode are observable on-chain (they have `view_master_pk` set). An observer can correlate confidential-mode adoption with account behavior.

*Mitigation.* Accepted residual. Full traffic-analysis resistance requires graph-privacy (stealth addresses, ring signatures) which is out of v2.22's scope. Document for operators.

**Rollback plan.** If v2.22 ships with a bug discovered after flag-day:
1. Governance pause (v2 mechanism) freezes new block production.
2. Disable confidential-amount tx type via flag-day reversal.
3. Existing confidential-amount commitments remain in chain history but are unrecoverable without view keys (matches Zcash/Monero shielded-pool semantics).
4. Re-attempt v2.22 in a future release after bug fix.

## 7. What this enables downstream

v2.22 with Option C view-key infrastructure enables:

- **v2.24 audit hooks** — concrete disclosure mechanism (master or per-epoch keys) instead of an abstract "auditor decrypts" claim.
- **v2.25 DSSO** — DSSO assertions can carry encrypted account-history summaries scoped to the relying party (using per-tx ephemeral DH against the RP's view key). Not v2.22's scope, but the infrastructure is reusable.
- **Regulated gambling deployments** — operators ship with `audit_view_master_pk` set to their licensing-jurisdiction regulator's pubkey; regulator gets standing audit access; players retain confidentiality of amounts from each other.
- **B2B settlement** — invoices, payroll, vendor payments where amounts are commercially sensitive but compliance disclosure is required.

The view-key infrastructure described here is **shared foundation** for Determ's privacy + compliance layer, not a v2.22-only investment.

## 8. Decision review

This spec is recommended to be reviewed before implementation. Reviewers should confirm:

1. **Q1 per-epoch HKDF cadence.** Acceptable wallet UX complexity for per-epoch derivation? Acceptable forward-secrecy posture (epoch-bounded compromise)?
2. **Q2 Bulletproofs over curve25519 (dalek-cryptography reference).** Accepted as range-proof construction? Alternative: BLS12-381 port (~2-3 weeks additional cost; only justified if future pairing-based features become in-scope).
3. **Q3 ephemeral DH handshake.** Accepted protocol shape? Alternative: bidirectional handshake (sender+recipient interaction before tx submission) — more secure, worse UX.
4. **Q4 dual-mode audit disclosure (master + per-epoch).** Accepted as the v2.24 integration? Alternative: per-epoch only (more conservative; loses in-house compliance use case).
5. **Wire-format break + flag-day migration.** Accepted vs. hard fork with re-genesis.

Once confirmed, implementation proceeds against §4 work units. Estimated 3 months from spec-review acceptance.
