**Status:** specification only. Resolves the 12 design questions for v2.26 ROTATE_KEY tx + composition with v2.15 multi-sig + v2.10 FROST + post-DSSO-as-DApp KR-10 unification. Implementation should not begin until this document's design choices are reviewed and committed.

## 1. Scope

This spec covers ONLY on-chain key rotation. It does NOT cover:
- Off-chain key generation (wallet ergonomics)
- Hardware-wallet UX certification (deferred to v3 per KR-11)
- Recovery from total key loss (handled by v2.14 OPAQUE, v2.15 multi-sig, or DSSO-DApp recovery flow per KR-12)

Per `dlt-no-migrations-constraint`, v2.26 ships in v1.0 mainnet genesis. Post-v1.0 changes constrained to additive-only or security-critical hard fork.

The artifacts produced by v2.26:
- **`ROTATE_KEY` tx type** (tx_type = 11) with `key_target` enum byte unifying REGISTER pubkey + DAPP_REGISTER service_pubkey rotation
- **`PendingRotation` state** queued at tx inclusion, applied at `effective_height` via pointer flip
- **Per-account configurable cooldown** field on RegistryEntry (clamped per profile)
- **`last_rotation_height` field** on RegistryEntry for RP staleness checks (DSSO use case)
- **`rotation_history` RPC** returning the last-rotation-height field
- **Dual-validity window** during `[effective_height, grace_until]` for in-flight tx tolerance
- **Cross-epoch DKG guard** (composes with v2.10 FROST committee continuity)
- **Multi-sig (v2.15) gating from day one** — no transient un-gated window

## 2. Design decisions

### KR-1: Wire format with dual-key Proof-of-Possession

**Decision: ROTATE_KEY = 11 tx with outer Transaction.sig signed by NEW key (PoP) + embedded `old_key_sig` field signed by OLD key. Payload includes `key_target` enum byte (per KR-10), `op` enum (rotate vs revoke), `new_pubkey` (variable-length per 7.5.7), `effective_height_delta`, `grace_blocks`, `reason_code`.**

Payload (binary codec):

```
[version: u8 = 1]                  # forward-compat byte, reserved 0xFF
[key_target: u8]                   # 0 = REGISTER pubkey, 1 = DAPP_REGISTER service_pubkey, 2..255 reserved
[op: u8]                           # 0 = rotate, 1 = revoke-only (no replacement)
[new_pubkey: PubKey]               # variable-length per §7.5.7 (pubkey_form + body); zero-length for op=1
[effective_height_delta: u16]      # blocks from inclusion to activation; clamped [ROTATE_MIN_DELAY, ROTATE_MAX_DELAY]
[grace_blocks: u16]                # overlap window during which old key remains accepted; clamped [0, ROTATE_GRACE_MAX]
[reason_code: u8]                  # 0=routine, 1=suspected-compromise, 2=scheduled, 3=lost-device, 4..=reserved
[old_key_sig: Signature]           # variable-length per §7.5.6 sig_form; signed by old key over canonical message
```

**Canonical old_key_sig message:** `chain_id || "ROTATE_KEY" || key_target || domain || old_pubkey || new_pubkey || effective_height || grace_blocks || reason_code || nonce`

Both signatures must verify. Validator checks `old_key_sig` against registry-resolved old pubkey BEFORE outer sig (rejects stale `old_pubkey` before crypto-on-attacker-supplied-data work).

**Rationale.** Dual-key PoP defeats unwitting-key-substitution: attacker without `new_privkey` cannot produce the outer Transaction.sig. Standard PoP pattern from PKI literature.

**Composition with §7.5.6 / §7.5.7:** Outer Transaction.sig and embedded old_key_sig follow tx-level `Transaction.sig_form` (homogeneous within tx, per §7.6.6). `new_pubkey` uses variable-length pubkey encoding (`pubkey_form` discriminator). PQ migration of rotation tx (post-v1.0, when v2.8 ships) requires no v2.26 spec change — discriminator dispatch handles it.

### KR-2: Five genesis-pinned constants

**Decision: 5 constants in `params.hpp` mirrored from `GenesisConfig`.**

| Constant | Default | Purpose |
|---|---|---|
| `ROTATE_MIN_DELAY` | 5 blocks | Smallest activation delay; gives observers a finalization window |
| `ROTATE_MAX_DELAY` | 1024 blocks | Bounds duration of "rotation is queued" state per domain |
| `ROTATE_GRACE_MAX` | 32 blocks | Caps overlap window during which both keys validate |
| `ROTATE_COOLDOWN` (single-sig default) | 256 blocks | Minimum gap between successive rotations on single-sig domain (multi-sig per KR-9) |
| `ROTATE_PENDING_CAP` | 1 per domain | At most one rotation queued per domain at any time |

All five live in `include/determ/chain/params.hpp` mirrored from `GenesisConfig` (same pattern as `merge_threshold_blocks`, `REGISTRATION_DELAY_WINDOW`). Genesis-pinned so all nodes agree on rotation policy.

### KR-3: Revoke-only escape valve (op=1) bypasses cooldown

**Decision: ROTATE_KEY with `op=1` (revoke-only, no replacement) is exempt from cooldown.**

Allows emergency kill-switch after operator detects a freshly-rotated-to key is also compromised. After revoke applies, `inactive_from = current_height + 1`; operator re-REGISTERs cleanly.

**Rationale.** Strictly worse than rotation (loses continuity) but better than running with known-compromised key. Cooldown otherwise blocks re-rotation within 256 blocks.

### KR-4: Dual-validity window with grace=0 for compromise

**Decision: during `[effective_height, grace_until]` both old and new keys validate outgoing txs from this domain. `grace_until = effective_height + grace_blocks`. For `reason_code=1` (suspected compromise), operator sets `grace_blocks=0` collapsing the window — old key is dead immediately at `effective_height`.**

Prevents in-flight tx rejection: tx signed by soon-to-be-retired key at block H lands at H+3, with effective_height=H+5, validates because dual window is still open. Operator trade-off for compromise: in-flight legitimate txs die with attacker's potential txs.

### KR-5: Cross-epoch DKG guard

**Decision: rotation rejected at validate-time if `effective_height` straddles an epoch boundary (`effective_height < next_epoch_start || effective_height > next_epoch_start + EPOCH_LENGTH/4`).**

Composes with v2.10 FROST committee continuity. Active-epoch share remains bound to old key (share-distribution ciphertext encrypted to old_pubkey). Next-epoch DKG ceremony targets new_pubkey. The guard widens the no-rotate window around DKG ceremonies, preventing mid-DKG rotation races.

**Composition with v2.10:** validator no-op pre-v2.10; activates automatically once v2.10 DKG ceremonies begin. No flag-day required.

### KR-6: Multi-sig (v2.15) gating from day one

**Decision: when `multisig_policies_[domain]` exists for the rotating domain, ROTATE_KEY requires `aux_sigs` meeting M-of-N threshold. Validator checks multi-sig BEFORE old_key_sig.**

The embedded `old_key_sig` counts as one of the M required signatures (the policy's primary slot). v2.26 ships v2.15-aware; no transient un-gated window.

### KR-7: Genesis validators subject to cooldown from block 0

**Decision: genesis validators cannot rotate until block 256 (ROTATE_COOLDOWN). Operators wanting immediate genesis-validator rotation must update the genesis file pre-launch.**

Intentional: initial network reaches steady state before any genesis validator can rotate. All-genesis-validators-rotate scenario is safe (K-of-K preserved at each block via independent gating).

### KR-8: Domain history RPC + on-chain `last_rotation_height` field

**Decision: lightweight `rotation_history domain=X` RPC backed by `RegistryEntry.last_rotation_height: u64` field (8 bytes per RegistryEntry).**

Field updated by `on_block_apply_end` whenever a pending rotation activates. RPC returns the field directly. For forensic history beyond last rotation, RPs scan block index back from `last_rotation_height`.

**DSSO RP use case:** assertion staleness check is `assertion.iat < domain.last_rotation_height` — single field lookup; no block-scan in the hot path. Per V2-DESIGN.md threat-model entry "DSSO identity-takeover."

**Rationale.** 8 bytes per registry entry is trivial under §7.5.7 variable-length encoding (RegistryEntry already variable-sized). Forensic history is rare; common case is "when was last rotation" which the field directly answers.

### KR-9: Per-account configurable cooldown (clamped)

**Decision: `RegistryEntry.cooldown_blocks: u16` per-account, clamped `[ROTATE_COOLDOWN_MIN=8, ROTATE_COOLDOWN_MAX=1024]`. Default 256 (single-sig pattern) or 32 (multi-sig pattern) at account creation; operator-tunable per-account thereafter via a new tx field on REGISTER.**

Composition pattern matches §7.5 discriminator-philosophy: preserve operator optionality at trivial state cost (2 bytes per RegistryEntry). Single-sig and multi-sig accounts can choose appropriate cooldown for their threat model.

**Multi-sig recovery:** multi-sig accounts can pick `cooldown_blocks=8` for rapid emergency recovery (since multi-sig threshold itself prevents single-signer griefing). KR-12's "v2.15 multi-sig as recovery path" is not rate-limited by global cooldown.

**Threshold-collusion griefing bounded** by the operator's chosen `cooldown_blocks` value (min 8). Operator picks per threat model.

### KR-10: ROTATE_KEY unifies REGISTER and DAPP_REGISTER rotation

**Decision: `ROTATE_KEY.key_target` enum byte (0 = REGISTER pubkey, 1 = DAPP_REGISTER service_pubkey, 2..255 reserved) lets a single tx type rotate either registry's pubkey. DApp service_pubkey rotation inherits all v2.26 protections (dual-validity, PoP, cooldown, audit trail, multi-sig gating).**

Validator dispatches old-key lookup by `key_target`:
- `key_target=0`: look up old pubkey in `Chain::registrants_[domain].ed_pub`
- `key_target=1`: look up old pubkey in `Chain::dapp_registry_[domain].service_pubkey`

Apply path also dispatches: `PendingRotation` stored in unified pending-rotations map; on apply, the appropriate registry pointer is flipped.

**Pre-mainnet migration of DAPP_REGISTER op=0 rotation users:** since v2.26 ships in v1.0 mainnet genesis, no live chain has DAPP_REGISTER op=0 rotations to migrate. DAPP_REGISTER op=0 is deprecated for rotation use from v1.0 day 1; DApp owners use ROTATE_KEY with key_target=1.

**Future extensibility:** new key types (e.g., v2.22 `audit_view_master_pk` via `ROTATE_AUDIT_KEY` → `key_target=2`) fold into the same enum. Avoids tx-type proliferation.

**Rationale (per DSSO-as-DApp implication).** DSSO is a DApp post-DSSO-as-DApp reclassification (memory `dlt-dsso-as-dapp`); DSSO DApp instances each have service_pubkeys. Unification ensures DSSO instance rotations get v2.26's security properties (PoP, dual-validity window for in-flight assertion safety, cooldown, audit trail).

### KR-11: Hardware-wallet support deferred to v3

**Decision: v1.0 chain provides no specific hardware-wallet support commitments. Standard Ed25519 signing over hashed envelope is wire-format-compatible with existing Ledger/Trezor firmware; explicit certification + UX support is post-v1.0 ecosystem work.**

V2-DESIGN.md originally said "wallet UX shows 'Rotate operator key for domain X to new key Y' before signing." That UX commitment is operationally-facing, not chain-level; reclassified to post-v1.0 wallet-ecosystem concern.

**Reference:** `Improvements.md §5.5` (post-v1.0 process item: hardware-wallet UX certification).

### KR-12: Recovery from total key loss

**Decision: v2.26 cannot help with total key loss (embedded `old_key_sig` is unforgeable by design). Recovery requires:**
1. **v2.14 OPAQUE wallet recovery** — per-account recovery envelope
2. **v2.15 multi-sig** — other signers can rotate via M-of-N threshold (note: KR-9 per-account cooldown allows multi-sig accounts to pick rapid-response cooldown)
3. **DSSO-DApp recovery flow** — committee-assisted re-bind via DSSO DApp post-v1.0

v2.26 makes recovery *possible* via these paths; it does not itself recover from total loss.

## 3. Wire-format extensions

### 3.1 New tx type

```
TxType::ROTATE_KEY = 11   // next free slot after DAPP_CALL = 10
```

### 3.2 ROTATE_KEY payload schema (binary codec)

Per KR-1 above. ~120 bytes typical (with Ed25519 sigs); larger when PQ migration (post-v1.0) drops `pubkey_form=PUBKEY_DILITHIUM`.

### 3.3 New RegistryEntry fields

```
RegistryEntry {
    ed_pub: PubKey,              // variable-length per §7.5.7 (was fixed PubKey32 pre-§7.5.7)
    ...existing fields...,
    last_rotation_height: u64,   // KR-8; 0 if never rotated
    cooldown_blocks: u16,        // KR-9; clamped [8, 1024]; default 256 single-sig / 32 multi-sig
}
```

### 3.4 New chain state

```
struct PendingRotation {
    key_target: u8,                  // 0=REGISTER, 1=DAPP_REGISTER service_pubkey
    new_pubkey: PubKey,              // variable-length per §7.5.7
    effective_height: u64,
    grace_until: u64,                // effective_height + grace_blocks
    op: u8,                          // 0=rotate, 1=revoke
    reason_code: u8,
    enqueued_at: u64,
}

std::map<std::string /*domain*/, PendingRotation> pending_rotations_;
// Bounded by registrants_.size() + dapp_registry_.size() × ROTATE_PENDING_CAP=1

struct RecentRotation {
    old_pubkey: PubKey,              // for dual-window validator check
    grace_until: u64,                // when dual-window closes
}

ring_buffer<RecentRotation> recent_rotations_;
// Bounded; ringbuffer; supports dual-validity window checks
```

### 3.5 New RPC

```
rotation_history domain=X
  → {
      last_rotation_height: u64,    // from RegistryEntry.last_rotation_height
      pending: bool,                 // true if pending_rotations_[domain] exists
      pending_effective_height: u64, // 0 if not pending
      pending_reason_code: u8,       // 0 if not pending
  }
```

For full historical sequence, RPs walk block index back from `last_rotation_height`.

### 3.6 New state-root namespace

```
"o:" → operator-rotation namespace
  Leaves: domain → SerializedPendingRotation
```

Contributes to `state_root` via the existing leaf-builder pattern (per v2.1 Merkle commitment).

## 4. Apply-path integration

### 4.1 Validator gate (`BlockValidator::validate_tx`)

For TxType::ROTATE_KEY rejects if:
1. `key_target` not in {0, 1, 2..255} (where ≥2 is reserved-rejected at v1.0)
2. Per-target registry lookup fails (domain not in registrants_ for key_target=0, not in dapp_registry_ for key_target=1)
3. Target's old pubkey doesn't match registry-resolved value
4. Embedded `old_key_sig` fails verification against old pubkey
5. Pending rotation already exists for this domain (and current tx is not a `op=1` revoke against that pending entry)
6. Cooldown window since last applied rotation has not elapsed (per `RegistryEntry.cooldown_blocks`); EXCEPT op=1 revoke is exempt (KR-3)
7. `effective_height_delta` or `grace_blocks` out of clamp range
8. Cross-epoch DKG guard fails (KR-5; pre-v2.10 = no-op)
9. Multi-sig policy exists and `aux_sigs` doesn't meet threshold (KR-6)

### 4.2 Apply path (`Chain::apply_transactions`)

On accept, queues `PendingRotation` (deferred-apply pattern). Updates `RegistryEntry.last_rotation_height = effective_height` at tx accept time so RPs can verify staleness against the imminent rotation.

### 4.3 Per-block tick (`Chain::on_block_apply_end`)

Walks `pending_rotations_`:
- For every entry with `effective_height == current_height`: flip registry pointer (registrants_[domain].ed_pub or dapp_registry_[domain].service_pubkey, dispatched by key_target). For op=1 revoke, mark entry inactive instead.
- For every entry with `grace_until == current_height`: clear `recent_rotations_[domain]` (dual-window closes).

Walking is amortized O(pending) per block; cap of 1 per domain × (registrants + dapp_registry) keeps this bounded.

### 4.4 Dual-validity window check

Between `effective_height` and `grace_until`, validator accepts txs signed by either old OR new pubkey for the rotating domain. After `grace_until`, only new pubkey validates.

### 4.5 Snapshot serialize/restore

`pending_rotations_` + `recent_rotations_` ringbuffer + RegistryEntry's new fields (`last_rotation_height`, `cooldown_blocks`) added to snapshot. New `o:` state-root namespace per §3.6.

### 4.6 Block-digest path (`compute_block_digest`)

No change. ROTATE_KEY txs are ordinary Transaction records and ride through existing union-tx-root + Phase-2 reveal path unmodified.

## 7. Risks and rollback plan

**Risk: Stolen-key drain.** Mitigated by `reason_code=1` + `grace_blocks=0` immediate-rotation path. Operator has at most ROTATE_MIN_DELAY=5 blocks exposure (or per-account configurable cooldown floor of 8 blocks).

**Risk: Unwitting-key-substitution.** Mitigated by outer Transaction.sig PoP requirement (KR-1). Attacker without new_privkey cannot produce the outer sig.

**Risk: Rotation-replay across forks.** Mitigated by `chain_id` binding in old_key_sig canonical message (KR-1).

**Risk: Rotation-griefing storm.** Mitigated by per-account cooldown (KR-9; minimum 8 blocks) + ROTATE_PENDING_CAP=1 + revoke-only escape valve (KR-3).

**Risk: Validator-committee disruption.** Mitigated by deferred-apply (5+ block delay) + cross-epoch DKG guard (KR-5).

**Risk: DSSO identity-takeover.** Mitigated by `last_rotation_height` field (KR-8) enabling RP staleness check; KR-10 unification ensures DSSO DApp service_pubkey rotation gets the same protections.

**Risk: Rotation-as-deplatforming.** Operational only — recommend multi-sig for high-value accounts; chain does not forbid single-sig rotation (would block legitimate solo operators).

**Risk: Compromised-but-still-signing race.** Mitigated by op=1 revoke-only escape valve (KR-3) after detected attacker activity during the 5-block window.

**Rollback plan.** If v2.26 ships with a bug discovered post-launch under no-migrations constraint:
1. Per `dlt-no-migrations-constraint`, breaking fix requires security-critical hard fork.
2. ROTATE_KEY tx type can be operationally disabled via consensus rule update (last-resort hard fork).
3. Existing rotations remain in history; corrupted rotations may need governance-coordinated recovery.

## 8. What this enables downstream

v2.26 with the resolved 12 decisions unlocks:

- **Identity continuity for validators** — rotation without DEREGISTER/re-REGISTER cycle; preserves domain string + stake + committee position
- **Identity continuity for wallet users** — compromised-key rotation without losing on-chain identity
- **DSSO DApp production posture** — DSSO DApp service_pubkey rotation gets full v2.26 protections (KR-10 unification); RP staleness check via `last_rotation_height` (KR-8); composable with v2.15 multi-sig (KR-6) + v2.10 FROST (KR-5)
- **v2.22 audit-key rotation pattern** — ROTATE_AUDIT_KEY can fold into ROTATE_KEY via key_target=2 (future-extensible)
- **v2.14 OPAQUE recovery composition** — total-key-loss recovery via OPAQUE-rebound rotation
- **v2.15 multi-sig recovery composition** — multi-sig signers rotate compromised key per KR-12
