## 0. Three properties locked at v1.1 (immutable for chain lifetime)

v1.1 is not just a feature ship — it is the *commitment surface* the formal-verification track targets. Three properties are load-bearing-locked at launch:

| Property | Mechanism | Bundle | Why locked |
|---|---|---|---|
| **God protocol** (Szabo sense — trusted third party no participant can subvert) | K-of-K mutual-distrust consensus + union_tx_root + commit-reveal randomness + light-client verifiability — **no FROST per `FROST_DEVIATION_NOTICE.md` 2026-06-07** | Substrate (already shipped; v1.x commit-reveal already provides unbiasable randomness) | Default deployment posture. §6.2 Quorum Liveness OPTIONAL is the only documented relaxation, opt-in per-deployment at genesis. |
| **Decentralized identity provider** | DSSO via **DLT-B** composition (`v2.25-DSSO-DAPP-SPEC.md`; `DECISION-LOG.md` 2026-07-15) — t-of-n T-OPRF on the shipped P-256 RFC 9497 VOPRF stack (joint password evaluation, per-share DLEQ-verified; any t of n suffice, no evaluation order) + block-anchored DAPP_CALL assertion (K-of-K block signature attests inclusion; assertion truth = the user's credential signature). Zero new crypto primitives — Ed25519 + P-256 (§3.9b) + DAPP_CALL + light-client verifier, all already shipped. | Bundle A | Identity primitives that DApp-layer per-principal billing (§9.2) depends on; substrate primitives must persist for chain lifetime. |
| **Perfect forward secrecy** | v2.22 PRIV-6 per-tx PFS via OTPK; amount irrecoverable after OTPK consumed | Substrate Bundle 4 (v2.22) | Opt-in per-account confidential_policy; capability preserved for chain lifetime. |

These three are **commitments**, not just features. The cost of breaking any of them post-launch is the full formal-verification track restart (FA1-FA12 + FB1-FB4 + ~100 `docs/proofs/` soundness theorems) against a new protocol — plus trust degradation from a moving target. The no-migrations constraint exists to protect these three (and the chain's other invariants) as a unit.

### Why no migrations — the formal-verifiability rationale

The ~100 `docs/proofs/` soundness theorems + FA1-FA12 analytic proofs + FB1-FB4 TLA+ specs are parameterized over a *specific* protocol. Wire-format break = proof targets invalidated = verification track restart. The decision is: **lock the protocol at v1.1 launch, never touch it, let the formal-verification track stabilize against an immutable target.** Trust property delivered: "the protocol that exists now is the protocol that will exist forever, and the proofs that hold now will hold forever." That is qualitatively different from "the current version is verified."

This is the load-bearing reason for the no-migrations constraint per memory `dlt-no-migrations-constraint`. Reference `DECISION-LOG.md 2026-06-06 (afternoon)` for the full rationale.

## 2. Bundle A — DSSO-DApp (Theme 9 production deployment)

**Goal.** Distributed identity provider running as a chain-aware DApp per `Improvements.md §8.1` reclassification. K-of-K-committee-hosted DApp instances; T-OPAQUE-style authentication (T-OPRF + envelope, no AKE); assertions verifiable via the chain. **Mechanism spec: `v2.25-DSSO-DAPP-SPEC.md` (DLT-B, 2026-07-15) — the single authoritative DSSO doc; this bundle section carries sequencing and effort only.**

**Pre-requisites (must be in v1.0):**
- ✅ Ed25519 (shipped; commit `031be9e`) — credential + assertion signing
- ✅ NIST P-256 + RFC 9497 OPRF/VOPRF (shipped; `CRYPTO-C99-SPEC.md` §3.8c/§3.9b) — the T-OPRF substrate (t-of-n, per-share DLEQ-verified)
- ✅ X25519 (shipped; commit `bc87704`) — share-envelope KX at registration (transport only)
- ✅ v2.18 DAPP_REGISTER (shipped) — DSSO registers as a DApp
- ✅ v2.19 DAPP_CALL (shipped) — user ↔ DSSO + DSSO ↔ RP messaging substrate
- ✅ Light-client suite (shipped) — RP verifies block sig + tx-inclusion proof
- ⏳ v2.26 ROTATE_KEY with KR-10 unification (Bundle 5 of substrate; not yet started)
- ⏳ §7.5.7 pubkey_form variable-length encoding (pre-bundle; not yet started)
- ❌ **NOT FROST** — per `FROST_DEVIATION_NOTICE.md` 2026-06-07/§10, FROST is removed from v1.1 scope; DLT-B introduces no threshold signing.

**DLT-B composition (2026-07-15 — `DECISION-LOG.md`; full protocol in `v2.25-DSSO-DAPP-SPEC.md`):**

Two crypto legs, both on primitives already shipped:

1. **T-OPRF leg.** t-of-n verifiable threshold OPRF on the shipped P-256 RFC 9497 VOPRF stack: per-user OPRF key, user-dealt Shamir shares (no DKG), per-response DLEQ verification against block-anchored share pubkeys, client-side Lagrange recombination (any t responses, any order). Output unseals an AEAD envelope holding the user's credential key. No operator (indeed no < t coalition) learns the password.

2. **Assertion leg (unchanged from DLT-A).** The user signs the RP challenge with the envelope-recovered credential key; a DSSO instance emits the assertion via DAPP_CALL; the RP light-client-verifies (a) K-of-K block signatures, (b) Merkle tx inclusion, (c) the **user's** signature under the block-anchored `cred_pk`. The block signature attests inclusion — assertion truth reduces to the user's Ed25519 signature, so no operator coalition can mint an assertion (`v2.25-DSSO-DAPP-SPEC.md` §5–§6).

**Crypto inventory:** Ed25519 + P-256 (§3.9b OPRF/VOPRF) + X25519 (share transport) + SHA-2/HKDF + profile AEAD + DAPP_CALL + state-root proofs. **All already shipped. Zero new primitives, zero new assumptions beyond §3.9b's.** DSSO correctness reduces to the published T-OPRF argument (2HashDH/TOPPSS lineage) plus existing block-sig + tx-inclusion proofs already in the formal-verification target surface.

**Work units (revised under DLT-B):**

| Sub-component | Effort | Notes |
|---|---|---|
| DSSO-DApp spec doc | ✅ done 2026-07-15 | `v2.25-DSSO-DAPP-SPEC.md` — DLT-B composition, T-OPRF protocol, registration/login, assertion format, RP verification, recovery, proof map, green gates G1–G6 |

**Open design questions:**
- Operator selection: chain's K committee directly (spec default n = K), or DApp-specific opt-in operators?

## 3. Bundle B — zk-VM-DApp (verifiable compute substrate)

**Goal.** A zero-knowledge virtual machine running off-chain that uses Determ for identity (DSSO), state anchoring (state_root proofs per v2.1), and settlement (Determ TRANSFER on exit). Enables private compute, zk-rollups, anonymous credentials, verifiable AI inference, and similar zk-applications — per V2-DESIGN.md §God-protocol "offloads pairing-based ZK to external L2."

**Pre-requisites:**
- v1.0 mainnet shipped
- DSSO-DApp shipped (Bundle A) — used for zk-VM user identity
- v2.1 state Merkle root ✅ (already shipped) — anchor target

**Architectural choice (biggest open question): which zk-VM stack?**

| Candidate | Trade-off |
|---|---|
| **RISC Zero (zk-VM over RISC-V)** | Mature; general-purpose; large proof sizes; production deployments since 2024 |
| **SP1 (Succinct Labs)** | Newer; RISC-V-based; smaller proofs; active development; performant prover |
| **StarkNet-style (Cairo-based)** | Mature; STARK proofs (post-quantum-friendly); proprietary VM (Cairo); ecosystem lock-in |
| **Custom Determ-optimized zk-VM** | Tailored to Determ's primitives; long horizon; high risk |
| **Multi-stack support** | Determ anchors any zk-VM; operators choose; highest flexibility; highest complexity |

**Recommended:** start with RISC Zero or SP1 (most production-ready); evaluate StarkNet for PQ-friendly variant. **Avoid** custom zk-VM (too much scope; would delay v1.1 by years).

**Work units (assuming RISC Zero or SP1 stack):**

| Sub-component | Effort | Notes |
|---|---|---|
| zk-VM-DApp architecture spec | ~2-3 weeks | New doc: `ZK-VM-DAPP-SPEC.md`; covers anchor model, settlement bridge, prover-network design, exit semantics |
| zk-VM stack vendoring + integration | ~2-3 weeks | Vendor chosen stack (RISC Zero / SP1); wrap in Determ-aware service |
| State-anchor mechanism (zk-VM proofs anchored to Determ state_root) | ~2-3 weeks | Periodic anchor txs commit zk-VM state to Determ; light-client verifiable |
| Settlement bridge (zk-VM exits as Determ TRANSFER) | ~3-4 weeks | Withdraw flow: zk-VM proof of exit → Determ TRANSFER tx; deposit flow inverse |
| Prover network (who runs the zk-VM compute) | ~3-4 weeks | Distributed prover marketplace OR designated prover operator; pricing model per `DAPP_SDK_GUIDANCE.md §7` |
| (Reference zk-VM applications previously listed here have been promoted to first-class Bundle D members — see D.6, D.7, D.8) | — | Moved to Bundle D per 2026-06-06 coherence pass — those 3 apps are killer DApps in their own right, not internal Bundle B demos |
| **Bundle B total** | **~3-5 months** | Highly dependent on zk-VM stack choice; lower bound assumes RISC Zero/SP1 turnkey integration; upper bound assumes more substantial customization. Reduced from prior ~3-6 months by lifting the 3 reference apps out (they're now Bundle D items consuming Bundle B's substrate). |

**Open design questions:**
- zk-VM stack choice (see table above)
- Prover network economics (who pays provers? gas-style for zk-VM operations?)
- Anchor frequency (every block? every N blocks? on-demand?)
- Settlement gas — zk-VM exit-tx fees set by zk-VM operator OR by Determ deployment?
- PQ readiness — STARKs are PQ-friendly; SNARKs aren't. Aligns with v2.8 PQ migration timeline?
