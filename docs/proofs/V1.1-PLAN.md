# v1.1 plan — DSSO + zk-VM + killer DApps

**Status:** planning artifact. Covers the post-v1.0 release that delivers the Theme 7 DApp ecosystem on top of v1.0 mainnet's substrate. All work is Additive (no chain protocol changes; rides §7.5 discriminators that ship in v1.0).

**Premise.** v1.0 mainnet ships the substrate (per `IMPLEMENTATION-SEQUENCING.md`). v1.1 is the application layer that turns the substrate into a usable platform — DSSO identity, zk-VM compute, and reference killer-DApps that demonstrate the substrate's value to operators and developers.

**Companion documents:**
- `IMPLEMENTATION-SEQUENCING.md` — v1.0 execution plan (this doc is the v1.1 successor)
- `V210ImplementationRoadmap.md` — v2.10 FROST threshold-randomness implementation tracker (the v1.0 Phase 0 work in active progress; v1.1 Bundle A DSSO-DApp depends on v2.10 FROST being shipped)
- `C99CryptoStackAudit.md` — adversarial audit of the libsodium-free C99 cryptographic primitives shipped during v1.0 Phase 0; remediations complete per commit `2e0058b`. v1.1 DApps inherit the audited substrate.
- `DECISION-LOG.md` — deliberation history (v1.1 decisions append here)
- `Improvements.md` — §8 DApp roadmap (DSSO entry); §10 near-term unblocks; this doc operationalizes those entries
- `DAPP_SDK_GUIDANCE.md` — browser-side crypto strategy; v1.1 DApps follow this guidance
- `MOTIVATION.md` — public-interest mandate; v1.1 killer DApps target the originating threat models

**Convention.** v1.1 is **Additive only** under `dlt-no-migrations-constraint`. Anything requiring chain protocol changes (consensus rules, wire format breaks, new tx types) belongs in v1.0 mainnet genesis, not v1.1. v1.1 deliverables are DApps (running on v2.18 + v2.19 + v2.20 substrate) + Additive chain features that ride existing §7.5 discriminators.

## 3. Bundle B — zk-VM-DApp (verifiable compute substrate)

**Goal.** A zero-knowledge virtual machine running off-chain that uses Determ for identity (DSSO), state anchoring (state_root proofs per v2.1), and settlement (Determ TRANSFER on exit). Enables private compute, zk-rollups, anonymous credentials, verifiable AI inference, and similar zk-applications — per V2-DESIGN.md §God-protocol "offloads pairing-based ZK to external L2."

**Pre-requisites:**
- v1.0 mainnet shipped
- DSSO-DApp shipped (Bundle A) — used for zk-VM user identity
- v2.1 state Merkle root ✅ (already shipped) — anchor target

**Architectural choice (biggest open question): which zk-VM stack?**

| Candidate | Trade-off |
|---|---|
| **RISC Zero (zk-VM over RISC-V)** | Mature; general-purpose; large proof sizes; production deployments since 2024 |
| **SP1 (Succinct Labs)** | Newer; RISC-V-based; smaller proofs; active development; performant prover |
| **StarkNet-style (Cairo-based)** | Mature; STARK proofs (post-quantum-friendly); proprietary VM (Cairo); ecosystem lock-in |
| **Custom Determ-optimized zk-VM** | Tailored to Determ's primitives; long horizon; high risk |
| **Multi-stack support** | Determ anchors any zk-VM; operators choose; highest flexibility; highest complexity |

**Recommended:** start with RISC Zero or SP1 (most production-ready); evaluate StarkNet for PQ-friendly variant. **Avoid** custom zk-VM (too much scope; would delay v1.1 by years).

**Work units (assuming RISC Zero or SP1 stack):**

| Sub-component | Effort | Notes |
|---|---|---|
| zk-VM-DApp architecture spec | ~2-3 weeks | New doc: `ZK-VM-DAPP-SPEC.md`; covers anchor model, settlement bridge, prover-network design, exit semantics |
| zk-VM stack vendoring + integration | ~2-3 weeks | Vendor chosen stack (RISC Zero / SP1); wrap in Determ-aware service |
| State-anchor mechanism (zk-VM proofs anchored to Determ state_root) | ~2-3 weeks | Periodic anchor txs commit zk-VM state to Determ; light-client verifiable |
| Settlement bridge (zk-VM exits as Determ TRANSFER) | ~3-4 weeks | Withdraw flow: zk-VM proof of exit → Determ TRANSFER tx; deposit flow inverse |
| Prover network (who runs the zk-VM compute) | ~3-4 weeks | Distributed prover marketplace OR designated prover operator; pricing model per `DAPP_SDK_GUIDANCE.md §7` |
| (Reference zk-VM applications previously listed here have been promoted to first-class Bundle D members — see D.6, D.7, D.8) | — | Moved to Bundle D per 2026-06-06 coherence pass — those 3 apps are killer DApps in their own right, not internal Bundle B demos |
| **Bundle B total** | **~3-5 months** | Highly dependent on zk-VM stack choice; lower bound assumes RISC Zero/SP1 turnkey integration; upper bound assumes more substantial customization. Reduced from prior ~3-6 months by lifting the 3 reference apps out (they're now Bundle D items consuming Bundle B's substrate). |

**Open design questions:**
- zk-VM stack choice (see table above)
- Prover network economics (who pays provers? gas-style for zk-VM operations?)
- Anchor frequency (every block? every N blocks? on-demand?)
- Settlement gas — zk-VM exit-tx fees set by zk-VM operator OR by Determ deployment?
- PQ readiness — STARKs are PQ-friendly; SNARKs aren't. Aligns with v2.8 PQ migration timeline?

### 5.3 Composition notes

- Each DApp is a Theme 7 application — registers via v2.18 DAPP_REGISTER, communicates via v2.19 DAPP_CALL
- Each DApp ships its own browser-side client per `DAPP_SDK_GUIDANCE.md` (FIPS profile uses SJCL; MODERN profile uses native DLT client + web interface)
- DApp pricing follows §9.2 / §7.2 patterns: per-principal subscription, per-action with cap, delegation-credential issuance fees
- zk-VM-using DApps (§5.2: D.4, D.6, D.7, D.8) gate on Bundle B substrate completion; chain-primitive-only DApps (§5.1: D.1, D.2, D.3, D.5, D.9) gate only on Bundle A (DSSO) completion

### 5.4 Mission-alignment priority

Two DApps directly serve the founding-mandate threat model from `MOTIVATION.md`:
- **D.5 (government random-selection)** — Bulgarian random-judge-selection compromised by closed-source implementation. Highest demonstration value for the originating use case.
- **D.9 (Merritt-witness voting)** — extends mission alignment to elections themselves: government voting under hybrid-warfare conditions where some voting infrastructure may be compromised. Reuses Beaconless-v2 §Q5 Merritt infrastructure already paid for at consensus layer. Highest strategic value for the broader "public-interest cryptographic substrate" framing.

**Ship-priority recommendation:** D.5 first (validates mission alignment with the most-cited use case from MOTIVATION.md), D.9 second (extends mission alignment to the deeper elections use case), then D.4 + D.6 (commercially and strategically broadest). D.1, D.2, D.3 validate commercial use cases; D.7, D.8 validate zk-VM substrate.

## 7. Sequencing within v1.1 (updated 2026-06-06)

```
v1.0 mainnet
    │
    ├──> Bundle A (DSSO-DApp) ──────────────┐
    │                                        │
    │                                        ├──> Bundle D §5.1 chain-primitive DApps (D.1, D.2, D.3, D.5, D.9)
    │                                        │   (depend on DSSO only)
    │                                        │
    │                                        ▼
    ├──> Bundle E (post-v1.0 Additive)  Bundle B (zk-VM-DApp substrate; depends on DSSO for identity)
    │                                        │
    │                                        ▼
    ├──> Bundle C (sketch-v2.x)         Bundle D §5.2 zk-VM-augmented DApps (D.4, D.6, D.7, D.8)
    │                                        (depend on DSSO + zk-VM substrate)
    │
    └──> (Bundle C parallelizes throughout)
```

**Critical path:** Bundle A → DSSO usable → Bundle D §5.1 chain-primitive DApps can start; Bundle B (zk-VM) starts; once Bundle B ships → Bundle D §5.2 zk-VM-augmented DApps can start. Bundle C parallelizes throughout (sketch items don't depend on DSSO or zk-VM).

**Wall-clock estimate (with 4-32 fungible threads parallelizing):**
- Bundle A: ~6-8 weeks (starts immediately post-mainnet)
- Bundles C, E start in parallel with A
- Bundle D §5.1 (5 chain-primitive DApps: D.1, D.2, D.3, D.5, D.9) starts when DSSO ships (~2 months post-mainnet); parallelizes across threads (~3-4 weeks wall-clock per DApp; bounded by longest)
- Bundle B (zk-VM substrate) starts in parallel with Bundle A; ~3-5 months
- Bundle D §5.2 (4 zk-VM-augmented DApps: D.4, D.6, D.7, D.8) starts when both Bundle B ships AND Bundle A ships (~5-6 months post-mainnet); parallelizes; ~4-6 weeks wall-clock per DApp (D.4 longest due to zk-VM integration)
- **Net v1.1 horizon: ~5-8 months post-mainnet** (slight increase from prior ~4-7 months due to D.9 addition + zk-VM-augmented DApp dependency on Bundle B)

Total project horizon (v1.0 launch + v1.1 ship): **~11-15 months from current state.**

## 9. Composition with v1.0 closure items

When v1.0 mainnet launches, several things become available for v1.1:
- Stable substrate (consensus, fee market, subsidy, DApp registry/call/streaming)
- §7.5 discriminator fields in v1.0 schema (preserve Additive paths)

**v1.1 takes these primitives and shows what they enable.** The substrate-is-rich-enough thesis (per `MOTIVATION.md` and V2-DESIGN.md §God-protocol "everything else is a DApp") gets validated by Bundle D's 9 reference DApps — 5 chain-primitive-only DApps (§5.1) + 4 zk-VM-augmented DApps (§5.2). The mix demonstrates both that chain primitives alone are sufficient for the originating use cases (D.5, D.9 government/elections; D.1, D.2, D.3 commercial) AND that zk-VM composition unlocks the verifiable-compute use cases (D.4 AI-agent, D.6 private rollup, D.7 verifiable AI inference, D.8 anonymous credentials).

## 10. What this plan is NOT

- **Not v3.** No protocol-breaking changes. All v1.1 work is Additive under `dlt-no-migrations-constraint`.

## 11. Decision needed

The biggest blocking decision is **zk-VM stack choice** for Bundle B. RISC Zero and SP1 are both production-ready RISC-V-based zk-VMs; pick one and start integration spec ~now (during v1.0 implementation; doesn't block v1.0). StarkNet has PQ-friendly STARK proofs but proprietary Cairo VM. Custom is too risky for v1.1 horizon. **This decision now gates 4 killer DApps (D.4, D.6, D.7, D.8) — not just Bundle B substrate.**

Killer-DApp prioritization is secondary but should align with go-to-market focus:
- **Mission-aligned first** (D.5 government random-selection per MOTIVATION.md originating use case; D.9 Merritt-voting as deeper extension to elections themselves)
- **Commercially-broadest first** (D.2 B2B settlement or D.4 AI-agent infra)
- **Demonstration-richest first** (D.1 gambling — uses every privacy + audit primitive)
