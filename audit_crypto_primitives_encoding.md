### Assumption consistency findings

1. **S009 label drift (MINOR).** S009DelayHashRemoval cites Preliminaries "A2" for SHA-256 preimage resistance — F0 assigns A2 = collision resistance, A3 = preimage (Preliminaries.md:60-61). It also cites "A3" for the CSPRNG — F0 assigns A4 (Preliminaries.md:62). Both section-level cites (§2.1, §2.3) are correct; only the labels are off by one. Substance unaffected.
2. **S009 adversary-label namespace violation (MINOR).** S009 §3 names its adversary classes "A1..A3" (bare digits). Preliminaries.md:70 explicitly reserves bare A-digit labels for primitive assumptions and requires adversaries to be subscripted (A_outside, A_flood, ...). Collision-prone in a corpus this size.
3. **A4 instantiation naming (NOTE, self-flagged).** Preliminaries.md:62 instantiates A4 as RAND_bytes; the shipped name is determ_rng_bytes (src/crypto/rng/rng.c, verified — getrandom/BCrypt OS-entropy shim). CryptoBackendMigrationSoundness already flags this; the fix belongs in Preliminaries.
4. **ML-DSA EUF-CMA is an unregistered assumption (MINOR).** PqHashAddressSoundness and the PQ envelope/accept-rule stack reduce to ML-DSA-44/65/87 EUF-CMA, but the F0 registry table (Preliminaries.md:57-63) lists only A1-A4 plus the A6 note. The PQ consensus path's signature assumption has no F0 label. Registry should be extended (A5 or A7) now that PQ_TRANSFER is a live accept-rule.
5. **F0 usage elsewhere is consistent.** BatchSigningSoundness (A1/A2/A4), the parity/codec/digest documents (A2/A3), and the migration document all use the F0 labels correctly. No doc assumes anything stronger than F0 plus the unregistered ML-DSA item.
6. **Quantum honesty is consistent.** PHA-L-2 (Ed25519 cross-path Shor-broken) and the FROST_NOTICE §9 absorbed claim (only unique-signature schemes give unbiasable-by-construction beacons) are both stated within the classical F0 adversary model and labelled as limits, not hidden.

### Composition gaps

**G1 — MAJOR: the signing-bytes injectivity premise is false as stated.** CanonicalSigningBytesParity.md line 100 asserts the two NUL separators are unambiguous "because the chain rejects embedded NUL in those fields at the validity layer". Code reading (spot check 3) shows this holds only for anon addresses (0x + 64 hex). Domain names (REGISTER `from`, genesis `domain`) and the `to` field accept arbitrary bytes including NUL, and the binary wire transports them length-prefixed. Counterexample to injectivity: (from="a\0b", to="c") and (from="a", to="b\0c") with identical type/amount/fee/nonce/payload produce byte-identical signing_bytes, hence identical tx_hash. **Exploitability is bounded:** a mauled parse changes `from`, so the second parse only verifies if both sender accounts map to the same key — i.e., an attacker registering two same-key domains can create self-ambiguous transactions (one signature, two debits of their own accounts), duplicate tx_hashes in tx_root, and receipt-hash aliasing (validator.cpp:1392 matches receipts by tx_hash). No cross-victim forgery path was identified. Rating reflects that a stated premise of record is factually wrong and the injectivity conclusion survives only on a restricted input space the document never states; the parity theorems T-1..4 are unaffected. Fix options: charset-gate domains and `to` at the validity layer (making the doc's claim true), or restate the theorem over the NUL-free subspace.

**G2 — MAJOR (honestly disclosed): constant-time assurance has no machine-checked or in-CI leg.** Across CRYPTO-C99-SPEC §8, ConstantTimeInventory, TimingProbeDesign, and TimingProbeCTPQCoverage, the entire CT claim rests on (a) source-level branchlessness review and (b) a reporting-only probe with smoke-size runs that is deliberately out of run_all.sh, with dudect/ctgrind vendoring unauthorized. Every document says this plainly; the gap is between the corpus's CT reality (source review + empirical smoke) and any reading of "provable security" that includes a timing leg. The docs are honest; the cluster's CT assurance level should be quoted exactly as the docs state it, never upgraded in summary material (CRYPTO-C99-SPEC §8's "every primitive verified constant-time" comes closest to overstatement).

**G3 — MINOR: ML-DSA shipped into a consensus accept-rule while its secret-path timing audit remains owner-gated and undone.** PQ_TRANSFER verification (pq_tx_auth.cpp, verified live) makes ML-DSA consensus-relevant, but MLDSAConformance NP-1 records that no full secret-path timing audit has been performed. Consensus-side impact is nil (verify's non-CT compare is over public data — confirmed at sign.c:281); the exposure is client-side PQ signing. The documents acknowledge this; the composition is flagged so the accept-rule's existence is not read as NP-1 closure.

**G4 — MINOR: PQSignatureEnvelopeSoundness internal contradiction.** The §4 coverage-map row still says "no accept-rule code exists / NOT shipped" while §3.2 and PQE-NC-1 record Option-A as SHIPPED — and the accept-rule is indeed live in code (spot check 5). Stale row in an otherwise careful document.

**G5 — MINOR: dangling references to deleted proof documents.** C99CryptoStackAudit §8b cites FrostThresholdSoundness.md as a proof anchor; that document was deleted in the 2026-07-09 consolidation (FROST_NOTICE §9). No live theorem depended on it (FROST was never in the consensus path — verified absent from tree), but audit-trail readers will hit a dead link. FROST_NOTICE §9 says the "surviving C99CryptoStackAudit.md" is the retained record; the surviving doc should be patched to point at the NOTICE instead.

**G6 — MINOR: CrossBinaryCanonicalFormat stale rows.** Wallet backend row claims libsodium/OpenSSL (false post-migration — verified the wallet is C99-only) and §5.1 claims wallet mirrors omit payload (false — verified payload is appended at wallet/main.cpp:8144). Both are post-dated by later documents in the same cluster (CSP, CBMS), so the corpus self-corrects, but the stale statements stand unretracted.

**G7 — NOTE: S009 internal table staleness.** §4.7's summary row still shows the pre-migration EVP_MD_CTX count (8) while §4.5 was updated to 0. The audit conclusion is unaffected (reproduced at 0), but the table misquotes the audit.

**G8 — NOTE: pervasive citation line drift.** Every line-pinned citation checked had moved (typical drift 1-100 lines); content matched everywhere except G1. The corpus already acknowledges commit-binding in places; making every code citation commit-pinned (path@commit:line) would convert this recurring drift into a non-issue.

**G9 — NOTE: shim-name anachronisms.** BatchSigningSoundness cites crypto_sign_detached/sodium_memzero (pre-migration API names now shimmed over determ::c99). Cosmetic, but it will mislead future code readers the same way G5/G6 do.

### Cluster bottom line

The cluster's consensus-critical core — signing-bytes parity, binary codec round-trip, block digest parity, PQ envelope parsing, PQ hash-address binding, signature-form fail-closure, and the backend-migration byte-identity argument — is **sound**: the theorems are structurally valid, honestly scoped, and every code citation I spot-checked (17 sites) matched the tree in content. Two documents are empirical by design and say so (ConstantTimeInventory, TimingProbeDesign); one is a governance record, not a proof (FROST_DEVIATION_NOTICE).

The cluster's real weaknesses are not broken theorems but: **(1)** one formally false premise of record (G1, NUL-free from/to) with bounded — self-ambiguity, not forgery — exploitability; **(2)** a constant-time assurance level that is source-review plus empirical smoke testing with no machine-checked or CI leg (G2), which the documents disclose but headline prose occasionally overstates; and **(3)** systematic documentation drift — stale rows, an unregistered ML-DSA assumption, off-by-one assumption labels, dangling references to deleted documents, and pervasive line-number drift (G4-G9) — the predictable cost of an append-only proof culture on a fast-moving tree. Nothing found invalidates a consensus-critical theorem. The two items worth owner action before any external "provably secure" claim is made: fix or re-scope the G1 injectivity premise, and quote the CT assurance level exactly as G2 bounds it.
