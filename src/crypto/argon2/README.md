# `src/crypto/argon2/` — Argon2id (RFC 9106) password hash

Per-module provenance + audit README required by `docs/proofs/CRYPTO-C99-SPEC.md`
§3.16. Status per spec §2 Q3 table + §3.6: **SHIPPED**, public domain, ~180 LOC,
one self-contained C99 file.

## 1. What this module implements

Argon2id v1.3 (version `0x13`, type id 2), the memory-hard password hash —
the primitive CRYPTO-C99-SPEC §3.6 designates as the libsodium-free passphrase
keyfile KDF. (Call-site reality today, per source: the shipped v2.17/S-004
keyfile envelope derives its key via OpenSSL PBKDF2-HMAC-SHA-256
(`wallet/envelope.cpp::derive_key`) + AES-256-GCM, and the tree's only
libsodium `crypto_pwhash` caller is the wallet OPAQUE stub's export-key
stretch, `wallet/opaque_primitives.cpp::argon2id` — see §5.)

One public entry point (`include/determ/crypto/argon2/argon2id.h`):

```c
int determ_argon2id(uint8_t *out, size_t outlen,
                    const uint8_t *pwd, size_t pwdlen,
                    const uint8_t *salt, size_t saltlen,
                    uint32_t t_cost, uint32_t m_cost, uint32_t parallelism);
```

Explicit-parameter (raw) interface matching the P-H-C `argon2id_hash_raw`:
`outlen >= 4`, `t_cost >= 1` passes, `m_cost` in KiB (values below
`8 * parallelism` are raised to that floor), `parallelism >= 1`. Returns 0 on
success, -1 on a bad parameter or allocation failure. The RFC 9106 optional
secret (K) and
associated-data (X) inputs are fixed at length 0 (matching libsodium's
`crypto_pwhash_argon2id`).

Everything is in `argon2id.c`: the H0 init hash (BLAKE2b-512 over the LE32
parameter block ‖ pwd ‖ salt ‖ zero-length secret/ad fields), the H'
variable-length hash (`blake2b_long`, RFC 9106 §3.3), the fBlaMka + `GB` + `P`
1024-byte block compression (`fill_block`: row pass + strided column pass over
R = prev ⊕ ref, output R ⊕ P(R), XOR-accumulating on passes > 0), the Argon2id
hybrid addressing, `index_alpha` reference-block selection (RFC 9106 §3.4), and
the final cross-lane XOR of each lane's last block → H'(outlen) tag.

Lanes are filled sequentially (no threading); for `p > 1` the output is
identical to the threaded reference — sync points are slice barriers — but
throughput does not scale with `p`.

## 2. Provenance + construction

- **Construction:** RFC 9106 / Password-Hashing-Competition (P-H-C) reference
  lineage — same H0/H'/compression/indexing structure as the P-H-C
  `argon2id_hash_raw`, re-derived as one C99 file rather than vendored.
- **Underlying hash:** the in-tree C99 BLAKE2b (`src/crypto/blake2/blake2b.c`,
  canonical RFC 7693, keyed + variable-length) — the only hash Argon2 uses.
  Includes only `argon2id.h`, `blake2b.h`, `secure_zero.h`, and libc.
- **License posture:** public domain per the CRYPTO-C99-SPEC §2 Q3 module
  table (a fresh single-file implementation of the public-domain P-H-C
  construction, not a vendored copy of the multi-file reference tree).
- **Version pin:** Argon2 v1.3 (`ARGON2_VERSION 0x13`), Argon2id
  (`ARGON2_TYPE_ID 2`), `SYNC_POINTS 4`, 1024-byte blocks. Ship commit
  `00e3efb` (BLAKE2b prerequisite `695d4f4`) per spec §3.6.

## 3. Validation evidence

The only independent Argon2id oracle in the build tree is libsodium (the
vendored OpenSSL is pinned at 1.1.1w, which has no Argon2 KDF), and the determ
daemon deliberately does not link libsodium (CMakeLists: only the `oprf` lib
and `determ-wallet` link `sodium`) — so validation is split into a pinned-KAT
in-process test plus a live byte-equality harness that links `libsodium.a`
standalone:

1. **`determ test-argon2id-c99`** (dispatch block in `src/main.cpp`; wrapper
   `tools/test_argon2id_c99.sh`) — pins 4 reference vectors generated by
   libsodium `crypto_pwhash_argon2id` (which maps `(opslimit, memlimit)` →
   `argon2id_hash_raw(t_cost=opslimit, m_cost=memlimit/1024, lanes=1, v0x13)`
   internally): (t=1, m=8 KiB, 32 B), (t=3, m=32 KiB, 32 B),
   (t=2, m=256 KiB, 64 B), (t=4, m=1 MiB, 32 B). Plus determinism (same inputs
   → same tag), salt-sensitivity (tag changes when one salt byte flips), and
   parameter-error rejection (`outlen < 4`, `t_cost = 0`).
2. **`tools/test_c99_libsodium_xval.sh`** — the standing live byte-equality
   harness (`tools/c99_libsodium_xval.c`, compiled against the build tree's
   `libsodium.a`). Its Argon2id leg asserts `determ_argon2id` byte-equal to
   `crypto_pwhash_argon2id(..., crypto_pwhash_argon2id_ALG_ARGON2ID13)` over
   the grid **t ∈ {1,2,3} × m ∈ {8,16,32,64,256} KiB** (p=1, outlen=32). The
   harness is spec §2 Q9 step 3 — the behaviour-preserving evidence for the
   Q9 step-2 call-site migration (see §5 for what the call sites actually
   are today).
3. **Ship-time grid:** the original standalone harness matched 12/12 over
   t ∈ {1,2,3} × m ∈ {8,16,32,256} KiB (p=1, outlen=32) — recorded in
   `docs/proofs/C99CryptoStackAudit.md` §8f, with the reproduction gcc command
   in the wrapper header. Any error in H0, H', the fBlaMka/P/G compression, the
   hybrid addressing, `index_alpha`, or the final XOR would diverge from
   libsodium on this grid.

The BLAKE2b substrate is independently validated by `determ test-blake2b-c99`
(`tools/test_blake2b_c99.sh`): byte-equal vs OpenSSL `EVP_blake2b512` plus
Python `hashlib.blake2b` reference vectors for the keyed/variable-length modes.

**Audit:** C99CryptoStackAudit §8f — four-dimension adversarial workflow, **6
confirmed findings, all robustness/hardening, zero correctness defects**
(`outlen`/`pwdlen`/`saltlen` > 2³²−1 silently LE32-truncating in H0;
`p*SYNC_POINTS` / `8*p` uint32 overflow UB for huge `p`). All remediated by the
consolidated entry guard at the top of `determ_argon2id` (`p <= UINT32_MAX/8`
bounds both products; lengths past 2³²−1 rejected) — finding IDs cited in the
source comment (ARG2ID-001/-002/-003 + ARGON2ID-003/-005). The correctness
dimension, including the p>1 cross-lane path not exercised by the 12/12 grid,
came back clean.

## 4. Constant-time / hygiene posture

**NOT constant-time in the data-dependent passes — by design.** That is the
Argon2d component's GPU resistance. The Argon2id hybrid keeps pass 0, slices
0..`SYNC_POINTS/2`−1 data-INdependent: their reference-block addresses are
generated by `fill_block` over PUBLIC parameters only (a zero block + the
{pass, lane, slice, mem, t_cost, type} parameter block with an incrementing
counter word — never the password). All remaining slices/passes take `pr`
from the previous block's first word, so
their memory-access pattern is a function of the password; an attacker with
fine-grained cache/power side-channel measurement could in principle learn
password information during those phases. Threat model (header comment): the
host running the KDF is trusted against fine-grained microarchitectural side
channels.

`determ_secure_zero` (`include/determ/crypto/secure_zero.h`, the
memory-hygiene half of spec §3.10) scrubs every secret-bearing intermediate:
the `R`/`Z` temporaries in `fill_block`, the chaining buffer in
`blake2b_long`, the initial-block staging buffer, the final XOR block + its
serialized form, the entire block array `B` before `free`, and `H0` on both
exits that follow its computation (success and allocation failure via the
`done` label; the parameter-reject return fires before `H0` is written).

The module performs **no secret-equality comparisons**, so nothing here routes
through `determ_ct_memcmp` (`include/determ/crypto/ct.h`, spec §3.10) — there
is no tag-verify step in a raw KDF. Consumers own that: in the C99 AEAD
modules an envelope would pair this KDF with, the tag compare routes through
`determ_ct_memcmp`. (Today's shipped keyfile envelope still pairs OpenSSL
PBKDF2 with OpenSSL AES-GCM, whose tag check is internal to OpenSSL — see §5.)

## 5. Known limitations / future work

- **Call-site migration not done** — gated on an explicit go-ahead because it
  touches on-disk keyfile format ownership (spec §2 Q9 step 2). Two distinct
  call sites, per source: (a) the v2.17/S-004 keyfile envelope
  (`wallet/envelope.cpp`) derives its key via OpenSSL PBKDF2-HMAC-SHA-256 +
  AES-256-GCM — NOT `crypto_pwhash` — so moving it onto `determ_argon2id`
  changes the derived key and therefore the on-disk envelope (a
  format-versioning change, not a byte-equal swap); (b) the tree's only
  libsodium `crypto_pwhash` caller, the wallet OPAQUE stub
  (`wallet/opaque_primitives.cpp::argon2id`, `ALG_ARGON2ID13`, called with
  ops=3 / mem=64 MiB by `wallet/opaque_adapter.cpp`) — for that one
  `determ_argon2id` is the byte-equal drop-in per §3 (its t=3 row is on the
  validated grid, though its 64 MiB m_cost sits above the 256 KiB live-grid
  ceiling and the 1 MiB pinned-KAT ceiling).
- **FIPS profile does not use this module.** `cluster` + `tactical` profiles
  use PBKDF2-HMAC-SHA-256 (SP 800-132) for keyfile encryption instead —
  substantially weaker as a password hash, but FIPS-validated (spec §2 Q10
  feature matrix + §3.8b).
- **Sequential lanes.** `p > 1` is output-correct but single-threaded; a
  threaded fill is a throughput optimization, not a security gate (the
  designated keyfile use inherits libsodium's `p = 1` mapping, one-shot).
- **Secret/ad inputs fixed at zero length.** Matching libsodium's API surface;
  expose them only if a consumer needs RFC 9106 K/X.
