# `src/crypto/sha3/` — SHA-3 / SHAKE (NIST FIPS 202)

Per-module provenance + audit README required by `docs/proofs/CRYPTO-C99-SPEC.md`
§3.16. Status: **increment 1 of the on-chain post-quantum signature track**
(ML-DSA/Dilithium prerequisite XOF), public domain, one self-contained C99
file. Header at `include/determ/crypto/sha3/sha3.h`, translation unit
`sha3.c` — the Keccak-f[1600] permutation plus the four FIPS 202 functions and
the incremental SHAKE XOF context.

## 1. What this module implements

C99-native SHA-3 (fixed-output hashes) and SHAKE (extendable-output functions,
XOFs) from NIST FIPS 202 — all built on the same Keccak-f[1600] sponge. This
is the libsodium-free, OpenSSL-free XOF the PQ signature track is built on
(§5): SHAKE128 for matrix expansion and SHAKE256 for rejection sampling.

Public entry points (`include/determ/crypto/sha3/sha3.h`):

| Function | Purpose |
|---|---|
| `determ_sha3_256(out, in, inlen)` | One-shot fixed 32-byte SHA-3-256 digest. |
| `determ_sha3_512(out, in, inlen)` | One-shot fixed 64-byte SHA-3-512 digest. |
| `determ_shake128(out, outlen, in, inlen)` | One-shot SHAKE128 XOF — arbitrary `outlen`. |
| `determ_shake256(out, outlen, in, inlen)` | One-shot SHAKE256 XOF — arbitrary `outlen`. |
| `determ_shake128_init(ctx)` then `determ_keccak_absorb(ctx, in, inlen)` / `determ_keccak_finalize(ctx)` / `determ_keccak_squeeze(ctx, out, outlen)` | Incremental SHAKE128 XOF — absorb in chunks, then squeeze output in chunks. |
| `determ_shake256_init(ctx)` then the same `determ_keccak_absorb` / `_finalize` / `_squeeze` | Incremental SHAKE256 XOF. |

Constants: rate (bytes absorbed/squeezed per permutation) is derived from the
capacity `c = 2n` for each function — SHA-3-256 rate 136 (capacity 512),
SHA-3-512 rate 72 (capacity 1024), SHAKE128 rate 168 (capacity 256), SHAKE256
rate 136 (capacity 512); the state is a fixed 200-byte (1600-bit) array.

The incremental XOF is the surface ML-DSA needs: `absorb` accepts the message
in any chunking, `finalize` applies the pad, and repeated `squeeze` calls emit
an unbounded, position-continuous output stream (the rejection-sampling loop
squeezes more bytes on demand without re-permuting from scratch). One-shot
`determ_shake128`/`_256` are thin wrappers over init/absorb/finalize/squeeze.

Scope: the four FIPS 202 named functions only. No SHA-3-224 / SHA-3-384, no
RawSHAKE, no cSHAKE / KMAC / TupleHash / ParallelHash (FIPS 202 does not define
those; SP 800-185 does), no Keccak variants at other widths.

## 2. Provenance + construction

- **Construction:** the canonical Keccak sponge from FIPS 202, written for
  Determ. The permutation is **Keccak-f[1600]**: `24` rounds, each the five
  step mappings **θ ρ π χ ι** over the 5×5×64-bit state, with the ρ rotation
  offsets, the π lane permutation, and the ι round constants taken directly
  from the standard. Absorbing XORs each rate-sized message block into the
  state and permutes; the pad is **pad10\*1 with domain separation** — the
  first appended byte is **`0x06`** for SHA-3 (the `01` FIPS 202 suffix +
  the `1` pad start) and **`0x1F`** for SHAKE (the `1111` suffix + pad
  start), and the final rate byte is OR'd with `0x80`. Squeezing reads the
  rate portion of the state, permuting between blocks. No code vendored from
  libsodium, OpenSSL, or XKCP.
- **License posture:** public domain (Determ-original implementation of the
  public-domain Keccak construction), per the CRYPTO-C99-SPEC §2 Q3 module
  table.
- **Version pin:** FIPS 202 (August 2015). Keccak-f[1600], 24 rounds, 200-byte
  state; the four rate/capacity pairs above.

## 3. Validation evidence

python's `hashlib` ships `sha3_256`, `sha3_512`, `shake_128`, and `shake_256`
(the FIPS 202 functions) — an independent, non-Determ oracle for every entry
point. Validation is a pinned FIPS 202 known-answer corpus with a reproducibility
gate:

1. **`tools/vectors/sha3_shake.json`** — the KAT corpus, cross-generated by
   python `hashlib` (`sha3_256`, `sha3_512`, `shake_128`, `shake_256`). It spans
   message lengths straddling every rate boundary (0, `"abc"`, rate−1, rate,
   rate+1, and a multi-block input for each of the four rates 72/136/168) and
   SHAKE output lengths from a short squeeze through `== rate`, `> rate`
   (squeeze-permute), and 512 bytes (multi squeeze-permute). 32 vectors: 6
   SHA-3-256, 6 SHA-3-512, 10 SHAKE128, 10 SHAKE256.
2. **`tools/verify_sha3_vectors.py`** — the self-check + reproducibility gate.
   Every stored digest is REGENERATED from the message with `hashlib` and
   asserted byte-equal, so a drifted / hand-edited / fabricated digest turns it
   RED; it also enforces the corpus's structural rules (strict hex, XOF
   `outlen` present and `len(digest_hex) == 2·outlen`, fixed-length algs carry
   no `outlen`). Prints `sha3 vectors: N/N OK`; needs no determ binary, stdlib
   `hashlib` only. Any error in θρπχι, the ρ/π offsets, the ι round constants,
   the rate selection, or the `0x06`/`0x1F` pad byte would diverge from hashlib
   on this grid.

3. **`determ test-sha3-c99`** — the in-process gate (wrapper
   `tools/test_sha3_c99.sh`, FAST-eligible). Recomputes SHA3-256/512 and
   SHAKE128/256 through `sha3.c` and asserts byte-equality against the OpenSSL
   §Q9 oracle (`EVP_sha3_256/512`, `EVP_shake128/256` via `DigestFinalXOF`) over
   a fuzzed length grid crossing the sponge rate boundaries and XOF outputs
   exceeding the rate; plus the FIPS 202 empty/`"abc"` KATs, incremental
   absorb/squeeze `==` one-shot, and a rate-boundary byte-by-byte check.
4. **`determ test-c99-vectors`** (offline sibling `tools/test_c99_vector_files.sh`)
   — `sha3_shake.json` is in the file-list of both, so every vector is
   recomputed through `sha3.c` (binary side) and through `hashlib` (offline
   side). A bug in this module — not just a corrupted vector — turns the
   corpus RED, closing the gap that `verify_sha3_vectors.py` alone (which only
   pins the vectors against `hashlib`) could not.

## 4. Constant-time / hygiene posture

- **Naturally constant-time — no secret-dependent branches or table lookups.**
  Keccak is data-independent by construction: the permutation is pure bitwise
  logic (XOR, AND, NOT, rotate-by-compile-time-constant) over a
  public-position state; χ is the only nonlinearity and is a fixed 5-lane
  formula, not a substitution table. Every branch depends solely on public
  lengths (`inlen`, `outlen`, the rate, buffer fill), never on message
  content, and there are no data-dependent memory indices. This is why FIPS 202
  functions are the standard XOF choice for constant-time lattice signatures.
- **Key hygiene (`determ_secure_zero`, CRYPTO-C99-SPEC §3.10):** the SHA-3
  fixed-output paths scrub the 200-byte sponge state after emitting the digest.
  The XOF context holds live sponge state across `squeeze` calls, so the caller
  owns its lifetime; callers deriving secrets through SHAKE (ML-DSA key/nonce
  expansion) should `determ_secure_zero` the ctx once the stream is exhausted.
- **Comparisons:** this module exposes no verify entry point and performs no
  secret comparisons — nothing here routes through `determ_ct_memcmp`
  (`include/determ/crypto/ct.h`, spec §3.10). A hash or XOF has no tag-compare
  step; consumers that compare a SHAKE-derived value own that call.

## 5. Known limitations / future work

- **Intended consumer: the on-chain PQ signature track (owner-authorized).**
  This module is increment 1 — the prerequisite XOF for **ML-DSA / Dilithium**
  (FIPS 204): **SHAKE128** expands the public matrix **Â** from the seed ρ, and
  **SHAKE256** drives coefficient/nonce sampling and the rejection loop (the
  incremental `squeeze` surface of §1). The same primitive also underlies
  **SLH-DSA** (FIPS 205, the SHAKE parameter set). ML-DSA itself is a later
  increment; today this module is additive with no in-tree signature consumer.
- **In-binary vector gate: done.** `sha3_shake.json` is wired into both
  `determ test-c99-vectors` (`src/main.cpp` file-list, recomputed through
  `sha3.c`) and the offline `tools/test_c99_vector_files.sh` (recomputed
  through `hashlib`), and `determ test-sha3-c99` (wrapper `test_sha3_c99.sh`)
  is the in-process OpenSSL-oracle / KAT / incremental==one-shot check —
  matching the sibling `test-*-c99` modules. `tools/verify_sha3_vectors.py`
  remains as the module-local corpus generator/reproducibility gate.
- **No SP 800-185 derivations.** cSHAKE / KMAC / TupleHash / ParallelHash are
  out of scope; add them only if a consumer needs a domain-separated XOF beyond
  raw SHAKE.
- **Scalar permutation.** Keccak-f[1600] is a straight 64-bit-lane
  implementation — no SIMD/AVX2 or bit-interleaved 32-bit path. Correctness and
  constant-time posture are the gates; throughput tuning is a later
  optimization, not a security concern.
