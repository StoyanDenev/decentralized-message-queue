# src/crypto/blake2 — BLAKE2b (RFC 7693)

Per-module provenance + audit README required by CRYPTO-C99-SPEC.md §3.16.

## 1. What this module implements

C99-native BLAKE2b (RFC 7693): a cryptographic hash with keyed (MAC) and
variable-length-output modes. One translation unit, `blake2b.c` (~130 lines);
header at `include/determ/crypto/blake2/blake2b.h`.

Public entry points:

| Function | Purpose |
|---|---|
| `determ_blake2b_init(ctx, outlen, key, keylen)` | Init for a 1..64-byte digest with optional 0..64-byte key. Returns 0 / -1 on parameter error (`outlen` 0 or >64, `keylen` >64). |
| `determ_blake2b_update(ctx, in, inlen)` | Incremental absorb. |
| `determ_blake2b_final(ctx, out)` | Emit the digest fixed at init; zeroizes the context. |
| `determ_blake2b(out, outlen, key, keylen, in, inlen)` | One-shot wrapper over init/update/final. |

Constants: `DETERM_BLAKE2B_BLOCKBYTES` 128, `DETERM_BLAKE2B_OUTBYTES` 64,
`DETERM_BLAKE2B_KEYBYTES` 64.

Scope: RFC 7693 sequential mode only. The parameter block sets digest length,
key length, fanout=1, depth=1 (XORed into `h[0]` in `determ_blake2b_init`);
no salt / personalization fields, no tree mode, no BLAKE2bp/BLAKE2s.

Role in the stack: this is the hash Argon2id is built on (CRYPTO-C99-SPEC
§3.6) — `src/crypto/argon2/argon2id.c` includes this header and uses it for
the H0 init hash and the H′ variable-length hash (`blake2b_long`). It is also
the libsodium-free equivalent of `crypto_generichash` (which IS BLAKE2b) for
any standalone caller.

## 2. Provenance + construction

- **Construction:** canonical RFC 7693 reference construction, written for
  Determ — the IV (SHA-512 IVs), the 12-round schedule over the SIGMA
  permutation table, the G quarter-round (rotations 32/24/16/63), the 128-bit
  byte counter `t[2]`, the last-block flag `f[0]`, and the keyed mode's
  key-as-own-padded-first-block prepend all follow the RFC's reference
  algorithm directly. No code vendored from libsodium or OpenSSL.
- **License posture:** Public domain (Determ-original implementation of a
  public-domain construction), per the CRYPTO-C99-SPEC §2 Q3 module table.
- **Version pin:** shipped at commit `695d4f4` (CRYPTO-C99-SPEC §3.6;
  C99CryptoStackAudit.md §8d). Adversarial audit pass at commit `c1caba5`.

## 3. Validation evidence

Subcommand: `determ test-blake2b-c99` (dispatch block in `src/main.cpp`).
Wrapper: `tools/test_blake2b_c99.sh` (gates on the
"PASS: blake2b-c99 all cross-validation + KATs matched" summary line).

What it cross-validates, per the dispatch block:

1. **OpenSSL oracle (the §Q9 cross-validation gate).** Unkeyed 64-byte digest
   byte-equal vs OpenSSL `EVP_blake2b512` (`EVP_Digest`) over a fuzzed
   message-length grid {0, 1, 63, 64, 65, 127, 128, 129, 200, 255, 256, 1000}
   — spans the empty input, sub-block, exact-block, block+1, and multi-block /
   counter-increment paths, i.e. the full compression / G / SIGMA / IV / finalize
   machinery.
2. **Independent oracle for the modes OpenSSL can't reach.** OpenSSL's EVP
   exposes only the unkeyed 64-byte digest, so keyed and variable-length
   outputs are pinned against reference vectors generated by Python's
   `hashlib.blake2b` (a separate BLAKE2 implementation): BLAKE2b-256(""),
   BLAKE2b-256("abc"), and keyed BLAKE2b-512 + BLAKE2b-256 KATs — exercising
   the param-block key-length/digest-length bytes and the key-block prepend.
3. **Self-consistency.** Incremental `update` == one-shot across the
   unkeyed/keyed × 64/32-byte-output matrix with an uneven chunk schedule
   (1, 63, 64, 1, 121 over 250 bytes).
4. **Mode wiring + parameter hardening.** Keyed ≠ unkeyed; output length is in
   the param block (BLAKE2b-256 is not a truncated BLAKE2b-512); `outlen`
   0 / >64 and `keylen` >64 rejected with -1.

Downstream evidence: `determ test-argon2id-c99` (byte-exact vs libsodium
`crypto_pwhash_argon2id` vectors, CRYPTO-C99-SPEC §3.6) transitively exercises
this BLAKE2b through Argon2id's H0 / H′ paths.

Audit: C99CryptoStackAudit.md §8d — four-dimension adversarial workflow,
**0 confirmed findings**; both the correctness and memory/API dimensions
clean.

## 4. Constant-time / hygiene posture

- **No secret-dependent control flow or memory indexing.** The core is pure
  ARX (64-bit add, XOR, rotate by compile-time constants in `rotr64`). SIGMA
  is indexed only by round number and position — public values; no
  data-dependent table lookups anywhere. Branches depend only on lengths
  (`inlen`, `buflen`, `outlen`, `keylen`), never on key or message bytes.
- **Key hygiene (`determ_secure_zero`, CRYPTO-C99-SPEC §3.10):**
  `determ_blake2b_init` zeroizes the padded 128-byte key block after absorbing
  it; `determ_blake2b_final` zeroizes the full-width 64-byte output buffer and
  the entire context (which holds buffered input and the chaining state derived
  from the key).
- **Comparisons:** this module performs no secret comparisons — BLAKE2b
  exposes no verify entry point. Callers using keyed BLAKE2b as a MAC must
  compare tags through `determ_ct_memcmp`
  (`include/determ/crypto/ct.h`, CRYPTO-C99-SPEC §3.10), not `memcmp`.
